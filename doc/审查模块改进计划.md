# 审查模块改进计划

> 创建日期：2026-07-20
> 修订日期：2026-07-21
> 状态：方案已确认，2026-07-21 开始实施 P0/P1
> 前置门禁：完成本文“四、统一章节标识门禁”后，才能改造审查存储和 UI
> 共同前置方案：`doc/卷章节目录与大纲关联公共方案.md`
> 数据基线：当前项目数据均为可清空的测试数据；不实现旧报告、旧章节引用或旧键格式的迁移与兼容读取

## 一、背景

当前审查 Tab 左侧以扁平列表展示已有审查报告。该结构在单卷、少量章节时可以使用，但不适合多卷长篇小说。

当项目增长到 5 卷、3000 章时，现有方案会同时遇到以下问题：

- 左侧缺少“卷 → 章节”层级，章节定位成本过高。
- 左侧只展示已有报告的章节，无法看出哪些章节尚未审查。
- 进入审查 Tab 时读取全部报告正文，历史报告增长后加载成本过高。
- 审查状态、跳转参数、报告文件名和报告分组只使用 `chapterId`。
- 不同卷可以同时存在 `ch001`，报告可能覆盖、混合或跳转到错误章节。
- 一致性检查结果目前只保存在组件状态中，离开页面后丢失。
- 章节修改后，旧报告仍可能显示为“通过”，缺少待复查状态。

本计划不只是给扁平列表增加卷标题，而是按以下顺序完成改造：

1. 统一章节标识。
2. 建立可恢复的报告存储 v2。
3. 定义可计算的审查状态。
4. 拆分摘要加载与详情加载。
5. 建立适合 3000 章的卷章节树。

## 二、已确认的当前问题

### 2.1 章节身份不完整

章节正文实际由 `volume + chapterId` 共同定位，但当前审查链路只传递 `chapterId`：

- `ReviewPanel` 使用 `expandedChapter: string | null` 保存选择。
- `ProjectView` 跳转审查页时只保存 `reviewChapterId`。
- `resolveChapterHtml()` 使用 `chapters.find(c => c.id === chapterId)`。
- `ChapterReviewData` 和 `ReviewReportMeta` 不包含 `volume`。

### 2.2 轻量报告会跨卷覆盖

当前轻量报告路径为：

```text
tracks/review-reports/light/ch001.json
```

后审查的同 ID 章节会覆盖先前报告。

### 2.3 深度报告会跨卷混合

当前深度报告文件名为：

```text
ch001_<timestamp>.json
```

加载时只按 `chapterId` 分组，多卷同 ID 章节的历史会混入同一组。

### 2.4 未审查章节不可见

当前 `loadChapterReviews()` 从报告文件构造章节列表。没有报告的章节不会进入结果，因此审查 Tab 无法展示完整章节目录。

### 2.5 问题数量会累计全部历史

当前 `countIssues()` 会累加轻量报告和所有深度历史报告的问题。同一个问题在多次深度审查中重复出现时会被重复计数，不能代表章节当前问题数。

### 2.6 一致性检查没有持久化

`runConsistencyChecks()` 的结果目前只写入 `ReviewPanel` 的 `consistencyResult` 状态。重新进入审查 Tab 后无法恢复，也不能参与左侧摘要。

### 2.7 当前文件 API 的约束

- `listProjectFiles()` 只枚举指定目录的直接子项，不递归。
- `writeProjectFile()` 使用普通 `fs::write`，不适合直接承担可恢复索引写入。
- 项目已有 Rust `atomic_write_json()`，支持临时文件、备份、替换失败恢复，应复用该实现，不新增另一套原子写入逻辑。

## 三、本次确定的设计决策

以下为实施时必须遵守的主方案，不在实施过程中临时更换。

### 3.1 当前阶段使用位置标识，不引入 UUID

统一章节引用：

```ts
export interface ChapterRef {
  volume: string
  chapterId: string
}
```

当前阶段不新增 `chapterUid`。`ChapterRef` 被定义为章节的当前位置标识，唯一性范围是同一项目当前章节树。

因此必须满足以下生命周期约束：

- 删除章节前归档该 `ChapterRef` 的审查数据。
- 删除卷前逐章归档审查数据。
- 新建同路径、同编号章节时不得继承旧报告。
- 当前系统没有跨卷移动章节功能；未来新增移动功能时，必须同步更新所有按 `ChapterRef` 保存的数据。

### 3.2 报告是事实源，摘要索引是可重建缓存

- v2 报告文件保存完整引用、时间、内容哈希和结果。
- `index.json` 只用于快速生成左侧目录。
- 索引缺失、损坏或处于 dirty 状态时，从 v2 报告重建。
- 索引中的“未审查”不能仅由缺少条目推断；加载器必须先确认索引有效。

### 3.3 左侧只统计各检查的最新结果

- 轻量检查：当前报告一份。
- 一致性检查：当前报告一份。
- 深度审查：以最新一份报告参与当前问题统计，其余只作为历史。
- 历史深度报告数量单独展示，不累加到当前问题数。

### 3.4 “无问题”不等于三类检查全部执行

不使用含义模糊的单一“已通过”状态。左侧同时表达：

- 是否从未检查。
- 最新结果是否有问题。
- 结果是否已过期。
- 哪些检查已经执行。

深度审查评分只展示数值，不设置人为通过分数。

### 3.5 树形导航不分页，历史报告使用通用分页

- 卷章节树使用折叠、搜索、筛选和虚拟列表。
- 深度历史报告使用项目通用 `<Pagination>`，默认每页 20 条。

审查树消费写作端的完整章节目录，不以大纲节点或“已有报告列表”为目录来源；共享树行为、章节段与搜索语义以公共方案为准。

### 3.6 虚拟列表采用固定主方案

新增依赖：

```text
@tanstack/react-virtual
```

使用扁平化 `TreeRow[]` 和固定行高实现虚拟滚动，不手写另一套虚拟列表算法。

## 四、统一章节标识门禁

本节是审查模块实施前的强制前置。未全部完成时，不得开始报告目录改造。

### 4.1 类型唯一归属

在 `src/types/chapter.ts` 定义唯一基础类型：

```ts
export interface ChapterRef {
  volume: string
  chapterId: string
}

export type ChapterKey = string & { readonly __brand: 'ChapterKey' }
```

现有 `CurrentChapterRef` 改为扩展该类型：

```ts
export interface CurrentChapterRef extends ChapterRef {
  projectId: string
  chapterTitle: string
}
```

禁止在审查、素材、灵感或其他模块重复声明形状相同的章节引用类型。

### 4.2 建立唯一编解码模块

新建 `src/lib/chapter-ref.ts`：

```ts
export function chapterRefKey(ref: ChapterRef): ChapterKey
export function parseChapterRefKey(key: string): ChapterRef | null
export function chapterMetaToRef(chapter: ChapterMeta): ChapterRef
export function sameChapter(left: ChapterRef, right: ChapterRef): boolean
```

规范格式：

```text
卷2:ch001
```

约束：

- `volume` 必须来自 `listChapters()` 返回的内部卷目录名。
- `chapterId` 必须非空并符合当前章节 ID 规则。
- 编码函数缺字段时抛出错误，不能生成空 key。
- `parseChapterRefKey()` 对非法 key 返回 `null`。
- 现有 `src/services/chapterDisplay.ts` 中的 `chapterRefKey()` 改为从新模块重导出，逐步替换调用方。

### 4.3 新数据只使用唯一键格式

清空测试数据后，所有模块的新写入统一使用 `ChapterKey` 的冒号格式。审查模块不保留旧斜杠格式或其他历史键格式的读取分支。

### 4.4 修复写作页选择与审查导航

门禁必须完成：

1. `ChapterManager` 当前选择改为 `ChapterKey` 或 `ChapterRef`。
2. 当前章节查找同时匹配 `volume` 与 `chapterId`。
3. React active 判断使用 `ChapterKey`。
4. `Editor.onNavigateToReview` 参数改为 `ChapterRef`。
5. `ProjectView` 保存完整 `reviewTarget: ChapterRef | null`。
6. 第一卷 `ch001` 和第二卷 `ch001` 的选择、编辑器加载和审查跳转测试通过。

### 4.5 门禁完成标准

- `ChapterRef` 只有一个定义来源。
- 新代码中不存在只传 `chapterId` 的审查入口。
- 写作页可以准确选择多卷同 ID 章节。
- `chapterRefKey()`、解析、非法输入和重复 ID 均有单元测试。

## 五、报告 v2 数据模型

### 5.1 报告封装

所有 v2 报告使用统一 envelope：

```ts
export type ReviewKind = 'light' | 'deep' | 'consistency'

export interface ReviewReportEnvelope<T> {
  schemaVersion: 2
  reportId: string
  kind: ReviewKind
  ref: ChapterRef
  createdAt: string
  contentHash: string
  result: T
}
```

规则：

- 新生成报告的 `reportId` 使用 `crypto.randomUUID()`。
- `createdAt` 使用完整 ISO 时间。
- 文件名使用 `Date.now()` 和 `reportId`，不得把含冒号的 ISO 时间直接作为 Windows 文件名。
- 新生成报告的 `contentHash` 为审查输入 HTML 的 SHA-256，不能为空。
- 轻量、深度和一致性报告都必须包含 envelope，不能只从路径反推归属。

### 5.2 单项检查摘要

```ts
export type ReviewFreshness = 'not_run' | 'current' | 'stale'

export interface ReviewCheckSummary {
  freshness: ReviewFreshness
  issueCount: number
  checkedAt: string | null
  reportId: string | null
  contentHash: string | null
}

export interface DeepReviewSummary extends ReviewCheckSummary {
  score: number | null
  historyCount: number
}
```

### 5.3 章节摘要

```ts
export interface ReviewChapterSummary {
  ref: ChapterRef
  light: ReviewCheckSummary
  deep: DeepReviewSummary
  consistency: ReviewCheckSummary
}

export interface ReviewIndex {
  schemaVersion: 2
  updatedAt: string
  chapters: Record<ChapterKey, ReviewChapterSummary>
}
```

卷名和章节标题不写入索引，进入页面时由 `listChapters()` 与 `loadChapterDisplayMetadata()` 提供，避免重命名后索引过期。

### 5.4 派生状态算法

以下状态不持久化，由摘要计算：

```ts
hasRun = 任一检查 freshness !== 'not_run'
hasIssues = 任一最新报告 issueCount > 0
isStale = 任一已运行检查 freshness === 'stale'
currentIssueCount = light.issueCount + deep.issueCount + consistency.issueCount
```

筛选规则：

- `未审查`：`hasRun === false`。
- `有问题`：`hasIssues === true`，包括已经过期但最新报告仍记录问题的章节。
- `待复查`：`isStale === true`。
- `无问题`：`hasRun === true && hasIssues === false && isStale === false`。
- `全部`：不限制状态。

章节可以同时命中“有问题”和“待复查”，UI 显示例如 `8 个问题 · 待复查`。

## 七、索引写入、恢复与并发规则

### 7.1 原子 JSON 写入

新增 Tauri 命令和前端包装：

```ts
writeProjectJsonAtomic(projectId, subdir, filename, value): Promise<void>
```

Rust 实现复用现有 `src-tauri/src/commands/material.rs` 中 `atomic_write_json()` 的临时文件、`sync_all`、备份和失败恢复逻辑。不得继续用普通 `writeProjectFile()` 写报告或索引 JSON。

### 7.2 项目级写入队列

在 `reviewIndexStorage.ts` 中建立按 `projectId` 分组的 Promise 队列：

```ts
withReviewIndexLock<T>(projectId: string, task: () => Promise<T>): Promise<T>
```

同一项目的以下操作必须串行：

- 保存报告并更新索引。
- 标记报告过期。
- 重建索引。
- 归档章节报告。

不同项目可以并行。

### 7.3 保存事务顺序

每次保存报告严格执行：

1. 获取项目级锁。
2. 原子写入 `index.dirty.json`，记录本次操作 ID。
3. 原子写入报告 envelope。
4. 读取或恢复有效索引。
5. 更新对应 `ChapterKey` 摘要。
6. 原子写入 `index.json`。
7. 重新读取并校验 `schemaVersion`、目标条目和 `reportId`。
8. 删除 `index.dirty.json`。
9. 释放锁。

任一步失败时保留 dirty 文件。下一次进入审查 Tab 或写入报告时必须先重建索引。

### 7.4 索引恢复

满足任一条件时执行 `rebuildReviewIndex(projectId)`：

- `index.json` 不存在。
- JSON 无法解析或 schema 不正确。
- `index.dirty.json` 存在。
- 详情加载或报告更新时发现索引引用的报告不存在、损坏或 envelope 引用不匹配。

重建规则：

- 扫描 v2 报告目录。
- 验证 envelope。
- 每类检查只取最新报告进入当前摘要。
- 深度历史数量单独统计。
- 损坏报告记录日志并跳过，不让整个审查 Tab 崩溃。
- 重建成功并校验后删除 dirty 文件。

正常进入页面不得逐条验证索引中的报告文件，也不得扫描报告正文；全量扫描只发生在索引恢复路径。

## 八、报告过期规则

### 8.1 正文变化

新建 `hashChapterContent(html)`，统一计算 SHA-256。

章节保存成功后调用：

```ts
markChapterReviewStale(projectId, ref, currentContentHash)
```

规则：

- 如果检查摘要的 `contentHash` 与当前哈希不同，将该检查改为 `stale`。
- 如果哈希相同，不改变状态。
- 未运行检查保持 `not_run`。
- 标记过期只更新索引，不修改历史报告。

必须接入：

- Editor 自动保存成功路径。
- Editor 手动保存 / `runSavePipeline()` 成功路径。
- AI 生成后实际落盘的保存路径。
- 章节恢复历史版本后的保存路径。

### 8.2 一致性数据变化

一致性检查还依赖伏笔、角色认知和时间线。以下数据保存后调用：

```ts
markProjectConsistencyStale(projectId)
```

该函数只把所有已运行的一致性摘要改为 `stale`，不影响轻量和深度报告。

接入点至少包括：

- `saveForeshadows()`。
- `memorySync.ts` 中写入 `character-states.json` 的路径。
- `memorySync.ts` 中写入 `timeline.json` 的路径。

实施前用 CodeGraph 再确认上述文件的全部写入调用方。

## 九、一致性检查的卷感知改造

### 9.1 全局章节顺序

新增：

```ts
interface ChapterSequenceEntry {
  ref: ChapterRef
  globalOrder: number
}

buildChapterSequence(chapters: ChapterMeta[]): ChapterSequenceEntry[]
```

算法：

1. 使用现有 `compareChapters()` 自然排序卷名，再按卷内 `order` 排序。
2. 排序后赋予从 1 开始的 `globalOrder`。
3. 所有“相隔 N 章”计算使用 `globalOrder`，不使用卷内 `order`。
4. Rust `list_chapters` 返回顺序不作为业务顺序事实源，前端必须显式调用统一排序函数。

### 9.2 接口与问题定位

`runConsistencyChecks()` 改为接收完整引用：

```ts
runConsistencyChecks(
  projectId: string,
  currentChapter: ChapterRef,
  presentCharacterNames: string[],
  thresholds?: Partial<ConsistencyThresholds>,
): Promise<ConsistencyCheckResult>
```

`ConsistencyIssue` 改为：

```ts
interface ConsistencyIssue {
  id: string
  type: ConsistencyCheckType
  severity: ConsistencySeverity
  chapterRef: ChapterRef
  globalOrder: number
  description: string
  suggestion?: string
  detail?: string
  foreshadowId?: string
}
```

### 9.3 依赖数据的章节引用

当前伏笔字段 `plantedChapterId`、`targetChapterId`、`resolvedChapterId` 和 `ForeshadowClue.chapterId` 缺少卷信息。实施一致性改造前统一替换为：

```ts
interface ForeshadowEntry {
  plantedChapter: ChapterRef
  targetChapter?: ChapterRef
  resolvedChapter?: ChapterRef
  // 其余现有字段保持不变
}

interface ForeshadowClue {
  chapter: ChapterRef
  description: string
  timestamp: string
}
```

测试数据初始化后，伏笔条目和线索只按以上完整 `ChapterRef` 结构写入；不实现旧 `chapterId` 字段的转换或冲突处理。

章节快照、时间线和角色认知也必须升级为卷感知结构：

```ts
interface ChapterSnapshot {
  chapterRef: ChapterRef
  globalOrder: number
  // 其余现有字段保持不变
}

interface TimelineEntry {
  chapterRef: ChapterRef
  globalOrder: number
  storyDate?: string
  events: string[]
}

interface CognitionState {
  characters: CharacterCognition[]
  readerKnows: string[]
  lastUpdatedChapterRef: ChapterRef
  lastUpdatedGlobalOrder: number
}
```

新快照文件按 `volume + chapterId` 保存，不能继续只用数字章号命名。`memorySync.ts` 查找、替换和排序时间线条目时使用 `ChapterKey` 和 `globalOrder`。

测试数据初始化后，章节快照、时间线和角色认知只保存 `ChapterRef + globalOrder`；不保留旧数字字段或兼容读取器。

## 十、报告存储初始化

实施前删除当前测试项目下的旧审查报告目录。首次进入审查 Tab 或首次写入报告时：

1. 创建空的 v2 报告目录和 `index.json`。
2. 只读取 v2 envelope 和索引。
3. 索引缺失、损坏或存在 dirty 文件时，从当前 v2 报告重建索引。

不实现 v1 读取、报告迁移、迁移状态、冲突清单或迁移确认 UI。

```ts
ensureReviewStorageV2(projectId): Promise<void>
```

该函数只负责创建空目录或恢复损坏的 v2 索引；它不扫描历史目录，也不转换任何旧报告。

## 十一、章节删除与报告归档

### 11.1 归档而非直接删除

删除章节前调用：

```ts
archiveChapterReviews(projectId, ref, reason): Promise<void>
```

归档流程：

1. 获取项目级锁并写 dirty 标记。
2. 将该章轻量、一致性和深度报告复制到 `archived/<archiveId>/`。
3. 写入包含原 `ChapterRef`、归档时间和原因的 `manifest.json`。
4. 校验归档文件。
5. 删除活跃报告文件。
6. 删除索引条目并原子写入索引。
7. 删除 dirty 标记。
8. 归档成功后才允许继续删除章节正文。

卷删除时按章节顺序逐章归档。任一章节归档失败时中止卷删除并提示错误，不能留下正文已删但报告仍活跃的半完成状态。

### 11.2 重建同编号章节

创建章节时不读取 archived 数据。只有活跃目录和索引中的报告可以关联到新章节，从而避免新 `卷1:ch001` 继承旧章节历史。

## 十三、左侧卷章节树

### 13.1 布局

```text
审查报告                                  [规则]
[搜索章节……]
[全部] [有问题] [待复查] [未审查] [无问题]

    第 3 章 夜访                           未审查
▸ 第三卷 破局        201/600 已检查 · 41 个问题
```

### 13.2 数据来源

进入页面并行加载：

1. `listChapters(projectId)`。
2. `loadChapterDisplayMetadata(projectId)`。
3. 经 `ensureReviewStorageV2()` 验证后的 `index.json`。

将三者在内存合并。未出现在有效索引中的现有章节显示为未审查。

### 13.3 卷节点

卷节点展示：

- 卷显示名称。
- 已执行任一检查的章节数 / 总章节数。
- 最新报告问题总数。
- 待复查章节数。
- 展开 / 折叠状态。

默认折叠全部卷；存在目标章节时只自动展开目标所在卷。用户可手动展开多个卷。

### 13.4 章节节点

章节节点展示：

- 卷内章号与自定义标题。
- 最新问题数。
- 待复查状态。
- 已执行检查类型的紧凑标记。
- 当前选中状态。

点击章节表示选择，不再使用 `expandedChapter` 表达选中。

### 13.5 搜索和筛选

搜索范围：

- 卷显示名称。
- 卷内章号。
- 自定义章节标题。

行为：

- 搜索和筛选都在 3000 条摘要内存数据上执行，不读取报告详情。
- 搜索命中时临时展开包含结果的卷。
- 清空搜索后恢复用户原有折叠状态。
- 筛选条件变化时保留仍可见的当前选择，否则清空右侧详情。

### 13.6 虚拟列表模型

构造：

```ts
type TreeRow =
  | { kind: 'volume'; volume: string }
  | { kind: 'segment'; volume: string; startOrder: number; endOrder: number }
  | { kind: 'chapter'; key: ChapterKey; ref: ChapterRef }
```

规则：

- 折叠卷只产生一个 volume row。
- 展开且不超过公共方案章节段阈值的卷，产生 volume row 和过滤后的 chapter rows。
- 超过阈值的卷先产生章节段 rows；仅展开当前章节、外部跳转或搜索命中所在的章节段，才产生对应 chapter rows。
- 搜索命中时同时临时展开目标卷与目标章节段；清空搜索后恢复用户原有展开状态。
- 卷行固定 36px，章节行固定 32px。
- `@tanstack/react-virtual` overscan 设为 8。
- React key 使用 `volume:<volume>`、`segment:<volume>:<startOrder>` 或 `chapter:<ChapterKey>`。

## 十四、右侧详情和异步规则

### 14.1 右侧结构

```text
第二卷 · 第 2 章《暗流》
[轻量检查] [AI 深度审查] [一致性检查]

```

检查按钮移动到右侧顶部，因为操作目标是当前章节。

### 14.2 详情按需加载

选择章节后只读取：

- 该章轻量报告。
- 该章一致性报告。
- 该章最新深度报告。
- 当前历史分页需要的深度报告。

深度历史默认每页 20 条，使用 `<Pagination>`。

### 14.3 竞态保护

`ReviewPanel` 使用递增请求序号：

```ts
const detailRequestId = useRef(0)
```

每次选择章节时递增。异步结果写入状态前必须同时满足：

- 请求序号仍为最新。
- 返回的 `ChapterKey` 等于当前选择。

运行检查时捕获发起操作的 `ChapterRef`。即使用户切换章节，报告仍保存到原目标；完成后只在当前选择仍相同时刷新右侧。

### 14.4 选择优先级

进入审查 Tab 时按以下优先级决定选择：

1. 写作页显式跳转的 `reviewTarget`。
2. `ProjectView` 保存的上一次审查选择。
3. 当前写作章节。
4. 无选择。

显式跳转目标消费一次后清空，避免以后重新进入审查 Tab 时反复覆盖用户选择。
