# 世界观模块改进计划

> 创建日期：2026-07-23<br>
> 状态：方案已确认；第一阶段于 2026-07-23 完成，第二、三阶段待实施<br>
> 当前范围：让世界观从“手工资料夹”升级为“以人工设定为核心、可选择接入 AI 辅助与一致性检查的创作中枢”

---

## 一、审查结论

世界观模块已经具备品类预设、自定义栏目与子字段、示例、按栏目 AI 生成、局部 AI 改写，以及在灵感模块中作为上下文来源的能力。它解决了“能记录、能编辑”的问题，但核心价值尚未完全进入写作闭环：用户手工维护的设定并不会稳定进入章节正文生成上下文，也缺少防丢稿、批量建档和一致性检查能力。

本次通过 CodeGraph 核对了以下实现：

- `src/components/WorldviewPanel.tsx`
- `src/components/worldview-panel/WorldviewEditor.tsx`
- `src/components/worldview-panel/worldviewMarkdown.ts`
- `src/services/worldviewConfig.ts`
- `src/services/aiContext.ts`
- `src/contextEngine/index.ts`
- `src/contextEngine/sources.ts`
- `src/services/brainstormContext.ts`

建议先补齐“写作实际使用世界观”和“编辑不丢稿”两个基础能力，再增加可选的 AI 建档与设定审查。世界观始终以用户手工创建、编辑和确认的内容为准；AI 不是必经步骤，不配置 AI 或从不点击 AI 入口时，全部手工工作流仍须完整可用。所有 AI 结果均应先以草稿或差异预览交给用户确认，不能直接覆盖现有设定。

## 二、现状与已确认边界

### 2.1 当前数据与编辑模型

世界观栏目配置保存在 `worldview/_worldview_sections.json`，每个栏目可定义名称、文件名、提示与子字段。具体内容按栏目保存为 Markdown；有子字段时，以 `# 栏目`、`## 子字段` 的层级生成内容。

用户可以：

- 使用随小说品类初始化的默认栏目；
- 新增、重命名、删除栏目与子字段；
- 在自由文本和结构化子字段之间编辑；
- 查看示例；
- 对当前栏目执行 AI 生成，或对选中文本执行 AI 改写、扩写、润色；
- 在项目品类变更后选择重置为新预设，已有内容不随配置重置删除。

### 2.2 已有的下游使用

灵感模块会按世界观栏目配置读取 Markdown 内容，并在“世界扩展”和“场景灵感”等模式中作为可选上下文。它会保留内容来源清单、优先级和截断标记，因此可以作为后续世界观引用追踪与按需取用的基础。

### 2.3 关键断点：正文生成未使用世界观

章节正文的上下文引擎当前只注册了最近章节摘要、角色认知、伏笔、素材、备注和文风；未注册世界观数据源。因此，世界观内容可能被灵感功能使用，却不会稳定进入续写提示词。这是本模块最优先需要修复的产品断点。

### 2.4 设计原则：人工优先，AI 可选

世界观是作者的创作资产，不是 AI 配置页。后续所有设计须遵循以下边界：

1. 用户可以只使用预设、手工新增栏目/字段、手工填写内容和手工维护规则，完成全部世界观工作。
2. AI 入口必须是显式的操作，例如“生成草案”“扩展本栏”“检查一致性”；不得在保存、切换栏目、打开项目或写作时自动调用 AI。
3. AI 只能提出草案、问题或修订建议；最终内容、规则强度、例外和忽略项均由用户决定。
4. 未配置 AI Provider、AI 请求失败或用户关闭 AI 功能时，不得阻断编辑、预设、检索、规则卡片和确定性检查。
5. AI 生成或审查使用哪些资料、哪些内容因预算被截断，都应向用户可见且可控制。

## 三、改进项

### 3.1 P0：将世界观接入章节正文生成上下文

**问题**

用户花时间填写的设定不能保证影响正文生成，容易出现规则被忽略、称谓不一致、能力体系失效等问题。

**改进方向**

1. 新增 `worldview` 上下文数据源，沿用上下文引擎现有的优先级和总 Token 预算机制。
2. 默认读取已配置的 Markdown 栏目，跳过 `_worldview_sections.json` 等配置文件。
3. 首期按栏目配置顺序取用，并对单一来源设置预算；后续再根据本章大纲、人物、地点和关键词检索最相关的栏目或片段。
4. 在章节 AI 生成前展示“本次引用的世界观”，支持用户临时勾选、排除或调整优先级。
5. 章节生成完成后，将实际引用栏目记录在生成记录中，为日后追踪和审查提供依据。

本项只影响用户主动使用“章节 AI 生成/续写”时的提示词组装，不改变纯手工写作流程，也不要求用户使用世界观 AI 辅助。

**验收标准**

- 世界观填写后，章节生成的上下文来源中能看到“世界观”。
- 配置 JSON 不会进入模型上下文。
- 上下文超出预算时有明确截断策略，不影响大纲、上一章结尾等高优先级内容。
- 用户可知道并控制每次生成使用了哪些设定。

### 3.2 P0：增加未保存保护与草稿恢复

**问题**

当前切换栏目会退出编辑状态并加载新栏目内容；未保存的 `content` 或 `subValues` 没有保存确认，也没有草稿恢复机制，存在误切换后丢失输入的风险。

**改进方向**

1. 在切换世界观栏目、切换项目 Tab、关闭项目或离开页面时，若存在未保存修改，弹出“保存 / 放弃 / 留在此处”。
2. 增加项目级本地草稿：按“栏目 + 子字段”保存编辑中的快照，并在下次打开时让用户恢复或丢弃。
3. 保存成功后清理对应草稿；保存失败时保留草稿并给出明确错误信息。
4. 删除栏目时保留可恢复副本，或至少先展示该栏目是否含内容及删除影响，避免把“删配置”误认为“仅隐藏”。

**验收标准**

- 有未保存修改时不能无提示切换到其他栏目。
- 刷新或意外关闭后可恢复未保存草稿。
- 删除含内容栏目时，用户能明确知道内容会受影响并拥有恢复路径。

### 3.3 P1：提供可选的“从项目资料生成首版世界观”向导

**问题**

现有 AI 辅助只能对当前栏目单次生成。其上下文主要是项目资料和世界观目录的前几个文件，未直接带入大纲、主要角色和未回收伏笔；希望借助 AI 起步的用户仍需逐栏面对空白编辑器。选择手工创建的用户不应被引导流程打断。

**改进方向**

1. 保留“使用预设”“新建栏目”“手工填写”作为默认主入口；另提供显式、可跳过的“生成世界观草案”入口。
2. 生成的基础输入是小说名、品类和简介；简介为空时，允许用户临时填写本次创作方向。大纲、角色、伏笔和已有世界观只要存在便作为可勾选的补强来源，不作为前置条件。
3. 按栏目或子字段展示差异预览，支持“全部采纳、采纳本栏、重新生成、保留现有内容”。
4. 生成过程分阶段进行：先给出核心规则和冲突，再补势力、地理、历史等扩展项，避免一次性输出大量互相矛盾的设定。
5. 允许用户保存并复用自己的“世界观起步模板”，满足预设可用、细节可改的原则。

**输入策略**

| 模式 | 必选输入 | 可选补强来源 | 适用场景 |
|---|---|---|---|
| 从零建立 | 小说名、类型、简介；简介为空时填写临时创作方向 | 已存在的大纲、角色、伏笔、世界观 | 先定世界观、后写大纲的新项目 |
| 基于现状补全 | 当前世界观 | 新增或修改过的大纲、角色、伏笔 | 创作推进后补充、修订世界观 |

可选来源只要存在即默认勾选，并显示缺失来源（例如“未找到角色资料，因此未参考”）；用户可在请求前取消勾选。AI 不可用时，模板、栏目编辑、示例和手工填写仍正常工作。

**验收标准**

- 新项目能在少量输入后得到可编辑的完整首版，而非一组空字段。
- 已有内容默认不被覆盖。
- 每一段 AI 内容都有明确来源、预览和单独采纳入口。
- 用户不使用或未配置 AI 时，仍能用预设和手工编辑完成同样的栏目配置与内容维护。

#### 3.3.1 AI 草案输出与采纳协议

AI 不直接返回一整份 Markdown 并写入项目文件，而是返回“提案包”。客户端先提供允许修改的栏目和字段，模型只能引用这些目标，不能自行创建写入路径。

```ts
interface WorldviewProposalResponse {
  schemaVersion: 1
  mode: 'bootstrap' | 'fill_empty' | 'expand' | 'review_update'
  summary: string
  usedSources: Array<{ type: string; label: string }>
  proposals: Array<{
    target: { sectionKey: string; fieldKey?: string }
    action: 'fill_empty' | 'suggest_append' | 'suggest_replace'
    content: string
    rationale: string
    dependsOn: string[]
    conflicts: Array<{ existingExcerpt: string; explanation: string }>
  }>
  questions: Array<{
    question: string
    suggestedTarget?: { sectionKey: string; fieldKey?: string }
    whyNeeded: string
  }>
}
```

| 提案动作 | 默认选择 | 用户可做的事 |
|---|---|---|
| `fill_empty` | 默认勾选 | 采纳、编辑后采纳、跳过、重生成 |
| `suggest_append` | 默认勾选 | 追加到现有字段或跳过 |
| `suggest_replace` | 默认不勾选 | 查看旧/新对比后替换、合并或跳过 |
| `questions` | 不写入 | 用户回答后，仅重生成相关字段 |

局部重生成只传入目标字段、现有内容、用户反馈和已采纳提案，并返回同一协议中的单字段提案。首期采用“提示词要求纯 JSON + 容错 JSON 解析 + 客户端校验目标 ID”的方式，避免依赖不同 AI Provider 未必支持的专有结构化输出参数。

### 3.4 P1：增加可选的世界观一致性审查

**问题**

纯 Markdown 记录便于自由创作，但难以主动发现人物能力超出体系、同一地点出现多个称呼、年代/势力关系矛盾、章节剧情突破硬规则等问题。

**改进方向**

1. 提供用户主动触发的“检查设定一致性”操作；不在编辑、保存或写作时自动检查。检查可读取世界观、角色、大纲、近期章节和伏笔，缺失资料只提示“未参与检查”，不视为错误。
2. 输出结构化审查结果：问题级别、涉及设定、证据片段、风险说明、建议修订。
3. 支持规则类别：术语/别名不一致、力量或科技规则冲突、人物状态冲突、时间线矛盾、未定义名词、关键栏目缺失。
4. 对每项建议提供“定位原文、忽略本次、标记为已知设定、生成修订草案”；不直接改写任何内容。
5. 审查结果可保存为项目记录，并在后续审查中对已忽略项目去重。

**验收标准**

- 每个问题都能追溯到具体栏目、角色或章节片段。
- 用户能逐条决定采纳、忽略或继续编辑。
- 审查不会自动覆盖用户的世界观或正文。

### 3.5 P1：修正 AI 世界观上下文的取用方式

**问题**

当前 `buildAIContext()` 直接从世界观目录取前 5 个文件，未过滤文件类型，也没有按照栏目配置读取。配置文件可能占用模型上下文；文件名也会替代栏目显示名，导致用户重命名栏目后 AI 仍看到旧文件名。

**改进方向**

1. 优先读取 `worldview/_worldview_sections.json` 中定义的栏目顺序和显示名。
2. 仅读取 `.md` 内容文件，显式排除配置、草稿和恢复文件。
3. 根据任务区分上下文：生成某栏目时优先引用其依赖栏目；润色某段时只引用必要规则和相关实体。
4. 以 Token 预算而不是“固定前 5 个文件”截断，并向用户显示截断情况。
5. 统一 AI 辅助与灵感模块的世界观读取逻辑，避免两条链路对同一项目取到不同内容。

### 3.6 P2：将关键设定升级为可控规则

**问题**

完全自由文本无法表达“必须遵守”的优先级和适用范围，模型也难以区分气氛描述与不可突破的约束。

**改进方向**

在保留 Markdown 自由编辑的基础上，为关键条目提供可选结构化标记。规则卡片应可完全手工创建、编辑、删除和标记例外；AI 最多提供“从现有内容建议提取规则”的可选按钮。

- 规则强度：硬规则、惯例、待确认设定；
- 实体信息：正式名、别名、类型、所属势力；
- 适用范围：时代、地域、角色、阵营；
- 状态：有效、已废弃、剧情保密；
- 溯源：首次出现章节、修改日期、相关条目。

这些字段应是可选增强，而非强制填表；未使用结构化字段时，原有 Markdown 工作流必须完整可用。

### 3.7 P2：提升组织、检索与追踪效率

建议补充以下编辑效率能力：

1. 栏目与子字段拖拽排序；
2. 复制栏目、复制预设、导入/导出项目级模板；
3. 全局搜索世界观术语，并定位到栏目、字段和章节引用；
4. 从某条设定反向查看哪些角色、伏笔、灵感建议和章节生成使用过它；
5. 对高频实体提供别名管理，避免“同一地点/势力多种写法”造成检索和生成偏差。

## 四、建议实施顺序

### 第一阶段：让已有世界观真正生效

1. 新增章节生成世界观数据源。
2. 修正 AI 辅助读取目录内容的过滤、排序和 Token 预算。
3. 增加未保存确认与本地草稿恢复。

这一阶段优先保护用户现有投入，并使填写内容直接影响续写质量。

### 第二阶段：提供可选辅助，降低建档成本

1. 实现“从项目资料生成首版世界观”向导。
2. 实现按栏目差异预览、选择采纳和局部重生成。
3. 支持项目级自定义世界观模板。

### 第三阶段：提升长期一致性与可追溯性

第三阶段应按以下顺序拆分，避免一开始把世界观强制改造成复杂数据库：

#### 3A：关键规则卡片（手工优先）

在 Markdown 之外新增可选规则卡片，字段包括规则名称、陈述、强度（硬规则/惯例/待确认）、适用范围、别名、状态（有效/废弃/保密）和来源栏目。用户可以完全手工维护；未创建任何规则卡片时，原有 Markdown 工作流不受影响。

#### 3B：确定性检查（无需 AI）

基于规则卡片和别名，提供稳定、低误报的本地检查：别名重复或冲突、同名硬规则多版本、已废弃规则仍被引用、保密规则被加入普通 AI 上下文、规则来源已删除等。该能力不调用 AI，适用于未配置 Provider 的项目。

#### 3C：带证据的 AI 语义审查（可选）

在用户主动运行后，AI 才检查术语冲突、硬规则冲突、时间线风险、角色状态风险和缺失定义。每一条发现必须包含来源类型、来源标识和原文摘录；无法引用证据的结果最高只能是“信息提示”，不能作为事实性错误。审查只生成问题与修订草案，不修改世界观或正文。

建议将审查严重度约定为：

- `blocker`：有硬规则原文和冲突内容两类证据；
- `warning`：存在可能矛盾或待确认的解释空间；
- `info`：术语、定义或资料可能缺失。

默认范围为“自上次审查后的改动 + 最近三章”，而非每次全书扫描；用户仍可选择扩大范围。

#### 3D：问题生命周期与反向引用

审查发现保存为 `open → accepted / fixed / ignored / known_exception` 等状态，并记录其证据来源和内容指纹。只有相关来源变更后，已忽略或已修复问题才重新激活，避免反复提示作者已确认的例外。最后再提供规则、角色、伏笔、章节和 AI 生成记录之间的反向引用，编辑某条规则时可预览受影响内容。

## 五、实施约束

- 所有涉及 AI 批量生成、审查修订的功能必须先展示草案或差异，用户确认后才写入项目文件。
- AI 只能是显式、可跳过的辅助能力；不得将 AI 配置、AI 请求成功或 AI 建议采纳作为创建、编辑、保存、检查规则或使用世界观的前置条件。
- 新增弹窗必须复用 `src/components/Modal.tsx`；若审查结果或引用列表分页，必须复用 `src/components/Pagination.tsx`。
- 世界观页仍须遵守项目的左右分栏高度传递链；新增顶部提示或生成进度条时，应使用外层 column wrapper 与内层 `flex: 1; min-height: 0` 的 row 容器。
- 改动任何 `src/**/*.ts` 或 `src/**/*.tsx` 时，交付前必须运行 `npm run lint:risk`；涉及上下文构建行为还应运行 `npm run check`。
- 目前 CodeGraph 未发现世界观核心函数的覆盖测试；实施时应补充世界观配置解析、上下文过滤/预算、草稿恢复和审查结果解析的测试。

## 六、已确认的存储与 AI 校验设计

### 6.1 草稿存储与保留策略

世界观草稿属于用户尚未正式保存的创作内容，应随项目移动、备份和同步，而不是只存在当前设备的应用本地存储。草稿文件放在：

```text
worldview/.drafts/<栏目文件名>.json
```

草稿使用以下最小模型：

```ts
interface WorldviewDraft {
  schemaVersion: 1
  sectionFile: string
  savedAt: string
  baseContentHash: string
  content: string
  subValues: Record<string, string>
}
```

- 编辑停止约 1.5 秒后写入草稿；
- 正式保存成功后删除对应草稿；
- 再次打开栏目发现草稿时，提供“恢复草稿 / 查看差异 / 丢弃”；
- 草稿不自动按时间清理，只能由保存、明确丢弃或用户手动清理删除；
- `.drafts` 必须从世界观栏目列表、AI 上下文和普通世界观搜索中排除。

### 6.2 规则、审查记录与备份文件

第三阶段的数据按职责拆分，目录约定如下：

```text
worldview/
  _worldview_sections.json
  _worldview_rules.json
  _worldview_issue_state.json
  audits/<audit-id>.json
  .drafts/<section-file>.json
  backups/<timestamp>-<filename>.json
```

| 文件 | 职责 | 编辑方式 |
|---|---|---|
| `_worldview_rules.json` | 当前规则卡片、别名、状态 | 通过规则卡片 UI 手工维护 |
| `audits/<audit-id>.json` | 单次审查的范围、资料清单与发现项快照 | 历史只读展示 |
| `_worldview_issue_state.json` | 问题状态与内容指纹 | 通过审查 UI 更新 |
| `backups/` | 迁移、重置前的可恢复副本 | 不建议直接编辑 |

迁移遵循以下安全规则：

1. 文件缺失时返回空数据，首次保存时才创建；
2. 旧版本先在内存迁移和校验，成功后备份旧文件再写入新版本；
3. JSON 损坏或遇到未知的更高版本时，只读报错，绝不静默重置或覆盖；
4. 审查历史按单文件保存为不可变快照，读取时兼容历史版本，不进行批量改写。

### 6.3 AI 提案与审查结果校验

新增两个独立解析入口：

```ts
parseWorldviewProposalResponse(raw, allowedTargets)
parseWorldviewAuditResponse(raw, allowedSources)
```

处理顺序为：提取 JSON 候选内容、解析或修复 JSON、校验顶层结构和版本、校验字段类型与枚举值、校验目标栏目/字段或来源引用，再生成可展示内容。解析器不直接写入任何项目文件。

- 顶层结构无效或没有任何有效提案时，整个请求失败且不写入；
- 少数单项无效时，保留合法项并提示已忽略的数量和原因；
- `suggest_replace` 默认不选，用户确认前再次比较原内容哈希，内容已变化时要求重新生成或手工合并；
- 审查项缺少原文证据时不能标为 `blocker`，最多作为 `info`；
- 未配置 AI、请求失败或格式错误时，提示用户可以继续手工编辑，不影响任何手工功能。

建议的用户提示包括：

- `AI 辅助不可用：未配置分析模型。你仍可继续手工编辑。`
- `AI 未返回可用内容，未修改任何设定。你可重试或继续手工填写。`
- `AI 草案格式无法识别，未写入任何内容。请重试。`
- `AI 草案包含无法定位的栏目，已阻止写入以保护现有设定。`
- `已忽略缺少原文证据的审查建议。`
- `生成期间该字段已被修改；为避免覆盖，请重新生成或手工合并。`
