## 一、审查结论

大纲模块当前可以完成「总纲 → 分卷纲 → 章节细纲」的基本文件编辑，但它是一个独立的 Markdown 文件树，并未与写作章节建立可靠的双向关联。最紧急的问题不是编辑器外观，而是章节身份、编号生成和删除操作：在多卷或删后重建的项目中，可能发生元数据串卷，甚至覆盖仍存在的大纲文件。

本次通过 CodeGraph 核对了以下实现：

- `src/components/OutlinePanel.tsx`
- `src/components/outline-panel/OutlineSidebar.tsx`
- `src/components/outline-panel/OutlineEditor.tsx`
- `src/services/settings.ts`
- `src/services/aiContext.ts`
- `src/components/ChapterManager.tsx`
- `src/services/chapterDisplay.ts`
- `src-tauri/src/lib.rs`

结论按“已确认问题”和“待产品确认的设计选择”分列；后者不能直接当作缺陷或擅自实施。

## 二、现有模型与已确认边界

### 2.1 当前文件模型

| 对象 | 实际存储 | 当前身份来源 |
|---|---|---|
| 总纲 | `outline/outline.md` | 固定文件名 |
| 分卷纲 | `outline/volumes/卷N.md` | 内部卷位置 |
| 章节细纲 | `outline/chapters/卷N/chNNN.md` | 完整 `ChapterRef` |
| 正文 | `chapters/<volume>/chNNN.md` | `{ volume, id }` |

大纲和正文不共享内容文件，但共享明确的卷章节位置。细纲路径由 `ChapterRef` 直接生成，不从展示名称或文件名正则反推身份。大纲只显示“卷 N / 第 M 章”，实际卷名和章节名只由写作维护。

### 2.2 系统已有但尚未被大纲统一采用的章节键

写作相关代码已有 `chapterRefKey({ volume, id })`，返回 `${volume}:${id}`。正文 `ChapterMeta` 也同时携带 `volume` 与 `id`。

因此应复用并收敛为同一份公共 `ChapterRef` / `chapterRefKey`，而不是在大纲模块另起一套只含章号的标识规则。

### 2.3 已核验的非问题

当前大纲根容器复用 `.panel-layout`、`.panel-sidebar`、`.panel-editor`。CSS 已满足项目的 flex 高度传递和双栏滚动约定；本轮没有发现需要用 `height: calc(100vh - Npx)` 修补的布局问题。

后续改造仍须保持这一链路，不得因增加工具栏、提示条或树控件破坏侧栏/编辑器的独立滚动。

## 三、已确认问题

### 3.1 P0：跨卷同章号会共用预计字数

**现状**

`outlineFileToChapterId()` 将：

```text
卷1_第1章.md -> ch001
卷2_第1章.md -> ch001
```

预计字数服务在 `outline/_chapter_meta.json` 中又仅以该 `chapterId` 为键：

```json
{
  "ch001": { "expectedWords": 4000 }
}
```

两卷第一章会读取、修改同一条记录。正文编辑器的 `resolveChapterWordCount()` 同样只接收 `chapterId`，因此即使只修大纲侧的解析，正文侧仍会把跨卷同章号当作同一个覆盖项。

**影响**

- 多卷项目无法为同章号章节分别设置预计字数。
- 编辑一卷的细纲可能改变另一卷正文编辑器的字数目标。
- 现有手动字数覆盖也使用单独的“仅章号”键，改造不能只处理 `_chapter_meta.json`。

**改进方向**

统一使用：

```ts
interface ChapterRef {
  volume: string
  chapterId: string
}

chapterRefKey({ volume, chapterId })
// => `${volume}:${chapterId}`
```

所有“按章节存储”的预计字数读、写、删、优先级决议都接收 `ChapterRef`，并统一调用公共键函数。

### 3.2 P0：删后新建按“数量 + 1”编号，存在静默覆盖文件的风险

**现状**

- 新增分卷使用 `volumes.length + 1` 生成 `卷N.md`。
- 新增细纲使用当前卷细纲数量 `existing.length + 1` 生成 `第N章.md`。
- 写作模块新增正文采用当前最大序号 + 1；两套模块也没有共享创建入口。

例如只剩 `卷2.md` 时，分卷数量为 1，再新增会生成 `卷2.md`；若某卷只剩“第2章”细纲，新增也会再次生成“第2章”。底层写文件接口会直接写入目标路径，没有冲突检查或二次确认。

**影响**

这是可导致已有大纲被覆盖的数据丢失风险，优先级高于交互优化。

**改进方向**

1. 创建前必须检查目标正文或细纲文件是否已存在；冲突时禁止覆盖。
2. 写作新建卷一次创建 `卷N/ch001` 正文，并要求填写实际卷名和章节名；不存在空的实际写作卷。
3. 写作新建章节只追加到当前卷最后一章之后；大纲新建细纲也只创建该卷下一个位置，不能依赖文件数量补号。
4. 从细纲“开始写作”只允许落地该卷下一个应写章节；无正文的卷必须从 `ch001` 开始。

### 3.3 P0：大纲与正文没有同步契约，删除会产生孤儿数据

**现状**

大纲的“删除分卷”会删除该分卷纲文件及同标签的所有细纲；“删除章节细纲”只删除一个细纲文件。两者均不检查或处理 `chapters/<volume>/chNNN.md` 中的正文、版本、备注、伏笔、审查结果及章节元数据。反方向亦然：正文删除不会清理对应细纲。

**影响**

- 用户无法从操作文案判断“删除分卷”到底删除大纲，还是删除小说内容。
- 大纲和正文可长期一边存在、一边缺失，后续 AI 或统计无法可靠选择上下文。
- 删除后遗留的字数覆盖、标题/卷名映射会继续影响以后复用同编号的章节。

**已确认的产品语义**

采用“**写作维护实际名称和正文，大纲按卷章节位置维护规划内容**”的模型，详细契约见 `doc/卷章节目录与大纲关联公共方案.md`：

- 大纲细纲直接以 `ChapterRef` 为目标位置，既不维护独立章节名称，也不需要显式绑定关系。
- 细纲可先于正文创建；“开始写作”直接在同一位置创建正文。正文存在后，大纲只显示“卷 N / 第 M 章”，名称继续由写作页维护。
- 正文已经存在时，用户可从任意位置按需创建细纲；不自动生成大量空细纲。
- 删除正文时保留同位置细纲，删除细纲时保留正文；审查报告则随正文归档。

### 3.4 P1：文件名正则是树结构和章节身份的唯一来源，异常文件会消失

**现状**

细纲仅识别 `卷<数字>_<任意文字>.md`。不符合规则的 Markdown 文件会得到空分卷标签，而侧栏只遍历已存在的分卷纲文件来渲染章节；这些文件不会显示、打开或删除。分卷纲则把除 `outline.md` 外的任意根目录 Markdown 都视为分卷。

**影响**

- 文件名同时承担展示名、排序、父子关系和 ID，导致跨卷位置和预计字数无法可靠定位。
- 用户改写 Markdown 文件名会让章节位置丢失，且侧栏可能不再显示该文件。

**改进方向**

- 使用公共方案确定的 `outline/volumes/<volume>.md` 与 `outline/chapters/<volume>/<chapterId>.md` 路径。
- 总纲、分卷纲和细纲的选择对象只保存文件类型与内部位置；文件路径由唯一辅助函数生成。
- 大纲不维护实际卷名或章节名，避免与写作端形成第二份名称数据。

### 3.5 P1：排序不自然，缺少折叠、搜索和大项目的树策略

**现状**

- 分卷文件按字符串排序，`卷10` 会排在 `卷2` 前。
- 章节虽按首个数字排序，但同一渲染轮次会对每个分卷重复过滤全部章节。
- 侧栏会一次性渲染全部卷和细纲，没有折叠、搜索、筛选、分页或虚拟列表。

**影响**

几十卷、数百至数千章时，定位困难，DOM 数量和重复筛选都会扩大；当前“新增”入口也要求用户先在长树中找到目标卷。

**改进方向**

1. 使用稳定的数值排序字段或自然排序，不能依赖展示字符串。
2. 默认折叠非活动分卷，记住项目内的展开状态。
3. 提供卷章节位置搜索、仅看“只有细纲”或“只有正文”的筛选，以及“跳转到卷/章”；不在大纲侧搜索或编辑实际名称。
4. 在确定性能门槛后，对超长树采用虚拟列表或按卷懒渲染；不要把现有分页组件硬套到层级树上。

### 3.6 P1：章节身份问题不止预计字数；写作模块已有同类键冲突

**现状**

`ChapterManager` 的活动章节、章节标题映射、删除状态过滤等多处只使用 `chapter.id`。多卷同为 `ch001` 时，选择、标题展示和删除状态都可能命中另一卷同 ID 章节。

**影响**

这不是大纲页面单独改一个解析函数能解决的问题。若大纲改为完整 `ChapterRef`、但写作模块继续只传 `chapterId`，两页仍无法建立可靠关联。

**改进方向**

将“章节身份统一”作为跨模块前置改造：活动选择、标题/卷名元数据、预计字数覆盖、删除清理、备注/伏笔/审查跳转等按章节存储或跳转的入口，均逐一审计是否丢失 `volume`。

### 3.7 P1：未保存草稿可因切换、删除或异步读取而丢失

**现状**

编辑状态只保存在当前组件内存中。切换总纲、分卷或细纲时直接加载新文件并退出编辑，没有未保存确认。删除当前或其他节点也没有处理当前草稿。多次快速切换时，文件读取没有取消或请求序号保护，较早请求可能在较晚请求之后写回编辑器状态。

**影响**

- 用户可在无提示情况下丢失输入。
- 异步竞态可能让标题与正文内容不对应；若随后保存，会写错文件。

**改进方向**

- 将“切换/删除/离开项目”统一接入脏数据守卫：保存、放弃、取消三种明确选择。
- 加入请求版本号或 `AbortController`，只接受当前选择对应的读取结果。
- 保存失败要保留草稿、显示可理解的错误和重试入口，不能只输出到控制台。

### 3.8 P1：预计字数覆盖不能重置，保存失败被静默吞掉

**现状**

预计字数输入框允许清空，但清空后不会删除已有的大纲覆盖项；重新进入页面仍会读到旧值。写入失败在 `persistExpectedWords()` 中被忽略，用户界面没有状态反馈。防抖定时器也没有在组件卸载时清理。

**影响**

用户无法回到“继承系统默认值”，且可能误以为修改已保存。

**改进方向**

- 提供“使用项目默认值/清除本章覆盖”操作，调用明确的删除接口。
- 显示当前值的来源（手动覆盖 / 大纲设置 / 项目默认），与正文编辑器的三级优先级保持一致。
- 保存采用可见的成功、失败、重试状态，并在卸载时清理防抖任务。

### 3.9 P1：AI 生成不了解当前大纲上下文，且会整体替换编辑区

**现状**

大纲 AI 只读取项目名称、类型、简介和最多五份世界观片段；不会带入当前总纲/分卷纲、同卷已写细纲、前后章节、正文进度、未保存编辑内容或关联章节标题。生成结果直接替换编辑器内容，只标为“未保存”，没有差异预览、插入位置或追加方式。

**影响**

- 分卷纲和细纲容易互相矛盾、重复或遗漏承接关系。
- 用户点击 AI 辅助即可覆盖正在编辑的完整内容，撤销能力不足。
- 当前提示词中的“根据项目信息”与实际可用上下文不匹配。

**改进方向**

1. 为总纲、分卷纲、细纲定义不同的最小上下文契约，并在 UI 中显示本次会发送哪些资料。
2. 章节细纲至少应包含关联卷纲、相邻细纲摘要、章节标题/预计字数和用户已选的角色、伏笔或设定；超长内容需可控截断与优先级。
3. 生成结果进入预览：替换、插入光标处、追加、丢弃四选一；复用现有改写预览交互，不直接覆盖草稿。
4. 让用户可以编辑提示词，但把项目级默认提示词和当前节点专属补充说明分开，避免所有章节共用一份难以追溯的提示词。

### 3.10 P2：编辑体验和可访问性仍偏“文件编辑器”

**现状**

大纲预览以保留换行的纯文本显示，AI 输出的 Markdown 标题和列表不会形成结构化阅读体验。侧栏可点击节点使用普通 `div`，没有语义化树控件、键盘导航或屏幕阅读器状态；删除按钮主要依赖鼠标悬停才显现。

**改进方向**

- 明确产品选择：继续纯文本、支持 Markdown 预览，或改为结构化情节点表单；三者的数据模型与 AI 输出格式不同，不宜混做。
- 如保留树，采用可聚焦的 `button` / treeitem 语义，支持键盘上下移动、展开收起和焦点可见。
- 关键操作不要只依赖 hover；窄屏和键盘用户也应能发现新建、重命名、删除和更多操作。

## 四、统一章节标识前置约定

后续所有实现以以下语义为准：

```ts
interface ChapterRef {
  volume: string
  chapterId: string
}

function chapterRefKey(ref: ChapterRef): string {
  return `${ref.volume}:${ref.chapterId}`
}
```

大纲的选择对象不能被强行简化为 `ChapterRef`，因为总纲和分卷纲并非章节；章节细纲则直接以完整章节位置为目标。实际文件路径由位置辅助函数生成，不能由用户可编辑名称拼接：

```ts
type OutlineSelection =
  | { type: 'outline'; file: 'outline.md' }
  | { type: 'volume'; volume: string }
  | { type: 'chapter'; ref: ChapterRef }
```

`volume`、`ChapterRef`、实际名称和派生文件路径是不同概念。大纲显示的“卷 N / 第 M 章”由位置生成，写作端的实际名称不参与大纲身份或文件定位。

## 五、测试数据初始化与新文件规则

实施前清空当前测试项目的旧大纲、正文和章节元数据；不实现迁移、旧路径兼容、冲突处理或迁移 UI。

新数据只允许以下路径：

```text
outline/outline.md
outline/volumes/<volume>.md
outline/chapters/<volume>/<chapterId>.md
chapters/<volume>/<chapterId>.md
```

所有预计字数、正文手动字数覆盖和章节相关元数据统一以 `ChapterKey` 保存。大纲细纲可在正文不存在时创建；其预计字数同样按目标 `ChapterRef` 保存。

## 六、建议的实施顺序

### 阶段 A：数据安全与身份统一

1. 引入公共 `ChapterRef`、键函数和跨卷测试夹具。
2. 修复所有按章节 ID 存储或选择的入口，不限于大纲预计字数。
3. 改为冲突安全的创建编号，补齐删除后的元数据清理策略。
4. 清空测试数据后，用新路径与完整 `ChapterKey` 初始化测试夹具。

### 阶段 B：落实大纲与正文的关系

1. 按公共方案以 `ChapterRef` 直接定位细纲；不建立独立名称或绑定表。
2. 落实“只有细纲 / 只有正文 / 两者都有”三种状态及创建、删除文案。
3. 实现“从细纲开始写作”与“从正文创建细纲”的双向入口。

### 阶段 C：树与编辑可靠性

1. 接入确定路径辅助函数，摆脱文件名正则承载业务含义。
2. 完成自然排序、折叠、位置搜索、长树性能方案及“只有细纲 / 只有正文”筛选。
3. 加入未保存守卫、异步竞态防护、可见的保存失败与重试。
4. 补齐字数覆盖的重置和来源展示。

#### 2026-07-21 实施审计

P0：单卷超过 50 章时按章节段懒展开，避免展开卷后一次渲染全部章节；搜索命中临时展开目标卷和章节段。

P1：增加“全部 / 只有细纲 / 只有正文”筛选、卷章直达和目标滚入可视区；清空搜索后恢复用户原有展开状态。

从细纲开始写作时，卷名和章节名均为可选自定义名称。留空时写作页使用 `卷N`、`第M章` 位置名称，不显示“未命名”。

### 阶段 D：AI 与结构化写作体验

1. 设计分层上下文和用户可见的发送范围。
2. 采用结果预览与合并操作，禁止直接覆盖草稿。
3. 决定纯文本、Markdown 或结构化情节点的编辑模型，再相应调整 AI 输出。

每个阶段完成前都应先跑多卷、删后重建和确定路径的回归场景；不得以 UI 演示替代数据读写验证。

## 七、验收场景

后续方案和实现至少覆盖：

- 第一卷、第二卷均有 `ch001`，两章预计字数、手动覆盖、标题和选择状态彼此独立。
- 删除卷1或卷1的第1章后，新建不会覆盖卷2或第2章的任何文件。
- 细纲存在而正文缺失、正文存在而细纲缺失时，用户能看见状态并按已确认规则处理。
- 切换节点、删除节点、切换项目或关闭页面时，有未保存草稿会先出现明确选择。
- 快速连续切换节点后，编辑器标题、内容和保存目标始终一致。
- `卷2` 位于 `卷10` 前；数百/数千章节可搜索并保持可用，不强制一次渲染整棵树。
- 只有细纲的位置可从大纲开始创建正文；只有正文的位置可按需创建细纲。
- 清除本章预计字数后，正文编辑器正确回退至下一优先级来源。
- AI 生成前能确认上下文，生成后能预览、替换、插入、追加或丢弃，不会覆盖未保存草稿。
- 大纲始终显示“卷 N / 第 M 章”，实际卷名与章节名仅在写作、审查中显示。
