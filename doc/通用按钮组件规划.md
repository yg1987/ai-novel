## 一、背景

当前项目中按钮散落在 33 个组件中，使用了 18+ 种互不兼容的 CSS 类，同级别操作按钮外观各异。更严重的是弹窗中"确定/取消"按钮的左右位置也不统一——有的确定在左取消在右，有的反过来。

### 1.1 典型问题：Editor toolbar 的 6 个同级操作按钮

同一行 toolbar 中，6 个对当前章节的操作按钮（都是"操作章节内容"这一层级），却呈现 3 种完全不同的视觉风格：

| 按钮 | CSS 类 | 视觉效果 |
|------|--------|----------|
| ✏️ 改写 / 📝 扩写 / ✨ 润色 | `.toolbar-btn` + override | 无背景、灰字、`padding: 0 8px`、`font-size: 0.78rem` |
| 🔍 审查 | `.toolbar-btn.review-btn` | 无背景、灰字、`padding: 0 10px`、`font-size: 0.8rem` |
| ✨ 生成 | `.toolbar-btn.generate-btn` | **实心 accent 背景 + 白字**、`padding: 0 12px`、`font-size: 0.8rem`、圆角、weight 500 |
| 保存 | `.toolbar-btn.save-btn` | **outline accent 边框 + accent 字色**、透明背景、`padding: 0 12px`、`font-size: 0.8rem`、weight 500 |

3 套 padding、2 套 font-size、3 种色彩方案（透明 / 实心 / outline），视觉上完全不像是同一组操作。

### 1.2 全局混乱：同类按钮不同类

| 场景 | A 面板 | B 面板 | 差异 |
|------|--------|--------|------|
| 主操作按钮 | Editor: `.toolbar-btn.generate-btn` | 伏笔: `.btn-primary` | 完全不同的 CSS 体系 |
| "新增"操作 | 伏笔: `.btn-add`（几乎无样式定义） | 角色: `.btn-small`（24px 圆形） | 大小、形状完全不同 |
| "保存"操作 | Editor: `.toolbar-btn.save-btn`（outline） | 角色/世界观/大纲: `.btn-primary`（实心） | outline vs filled |
| 辅助操作 | Editor: `.toolbar-btn.review-btn` | 角色: `.btn-secondary` | 透明 toolbar-btn vs 带边框 |

### 1.3 弹窗按钮布局不一致

#### 1.3.1 左右位置不一致

同样是"确定操作 + 取消操作"两个按钮，6 个地方遵循「取消左、确定右」，另外 6 个地方反过来：

**✅ 取消左、确定右（标准）**：

| 文件 | 弹窗 | 布局 |
|------|------|------|
| `ConfirmDialog.tsx` | 删除/确定 | `[取消]` `[确定]` |
| `SettingsModal.tsx` | 保存 | `[取消]` `[保存]` |
| `CreateProjectDialog.tsx` | 创建 | `[取消]` `[创建]` |
| `EditProjectDialog.tsx` | 保存 | `[取消]` `[保存]` |
| `ProviderConfig.tsx` | 保存 | `[取消]` `[保存]` |
| `ReviewRulesEditor.tsx` | 保存 | `[取消]` `[保存]` |

**❌ 确定左、取消右（需修正）**：

| 文件 | 弹窗 | 布局 |
|------|------|------|
| `ForeshadowPanel.tsx` | 删除确认 | `[确定删除]` `[取消]` |
| `ForeshadowPanel.tsx` | 推进确认 | `[确认推进]` `[取消]` |
| `ForeshadowPanel.tsx` | 新增/编辑 | `[保存]` `[取消]` |
| `ForeshadowPanel.tsx` | 灵感分析 | `[开始分析]` `[取消]` |
| `VersionHistoryPanel.tsx` | 恢复确认 | `[确认恢复]` `[取消]` |
| `FocusModeOverlay.tsx` | 开始专注 | `[开始专注]` `[取消]` |

#### 1.3.2 底部操作栏样式不一致

所有弹窗/表单底部的按钮容器用了 7 种不同的类名和样式，同一个视觉概念却有 7 套实现：

| 容器类 | justify | gap | margin/padding | 顶部分割线 | 使用位置 |
|--------|---------|-----|---------------|:---:|------|
| `.dialog-actions` | `flex-end` | `8px` | `margin-top: 20px` | ❌ | ConfirmDialog、Settings、Create/EditProject、ProviderConfig |
| `.note-edit-actions` | `flex-end` | `8px` | `margin-top: 6px` | ❌ | NotesPanel 行内编辑 |
| `.modal-actions` | `flex-end` | `8px` | `margin-top: 12px; padding-top: 16px` | ✅ | ForeshadowPanel 全部弹窗 |
| `.export-footer` | `flex-end` | **无** | `padding: 12px 16px` | ✅ | ExportDialog |
| `.focus-actions` | **`center`** | **`12px`** | `margin-top: 16px` | ❌ | FocusModeOverlay |
| `.version-confirm-restore` | **默认(flex-start)** | `8px` | 无 | ❌ | VersionHistoryPanel |
| ReviewRulesEditor | `space-between` + 嵌套 flex | `8px` | `padding: 12px 20px` | ✅ | 审查规则编辑器 |

**问题**：gap 两种值（8px/12px/无）、justify 三种值（flex-end/center/flex-start）、分割线有的有有的没有。

#### 1.3.3 统一规范

**规则一**：取消/次要操作在左，确定/主要操作在右。

**规则二**：弹窗底部操作栏统一用 `.dialog-footer` 类，一个定义覆盖所有场景：

```css
.dialog-footer {
  display: flex;
  justify-content: flex-end;   /* 按钮靠右 */
  align-items: center;
  gap: 8px;                     /* 统一 8px 间距 */
  padding-top: 12px;
  margin-top: 12px;
  border-top: 1px solid var(--border);
  flex-shrink: 0;
}
```

| 决策 | 理由 |
|------|------|
| 始终带分割线 | Modal 和非 Modal 对话框视觉一致，分割线是廉价的信息层次信号 |
| `flex-end` 始终靠右 | 所有弹窗底部操作区都靠右，无一例外（ReviewRulesEditor 的左按钮"恢复默认"是特例，用 `margin-right: auto` 推左即可） |
| `gap: 8px` | 原大部分用 8px，`.focus-actions` 的 12px 是异常值，统一到 8px |
| `padding-top + margin-top` | 保留原 `.modal-actions` 的间距，既让分割线不紧贴内容、又让按钮区与内容有呼吸感 |

**规则三**：非弹窗场景（行内编辑、面板内操作）如果不需要分割线，直接在具体组件内用 `style={{ borderTop: 'none', paddingTop: 0, marginTop: 0 }}` 覆写。这种场景极少（当前仅 NotesPanel 行内编辑 1 处），不值得为此拆两个类。

### 1.5 问题清单

| # | 问题 | 严重度 | 影响 |
|---|------|:---:|------|
| 1 | 同级别按钮视觉不一致 | 🔴 S1 | 用户困惑：改写和生成看起来不像同一组操作 |
| 2 | 弹窗按钮左右布局不一致 | 🔴 S1 | 用户肌肉记忆错乱：同一 App 内有时确定在左有时在右 |
| 3 | `.btn-add` 几乎无样式定义 | 🟡 S2 | 新增伏笔按钮样式靠上下文继承，脆弱 |
| 4 | `.btn-small` 被覆写成两种完全不同的形态 | 🟡 S2 | 语义混乱，圆形 24px vs flex 拉满 |

| 6 | `.btn-sm` 无 CSS 定义 | 🟢 S3 | 用在 ForeshadowPanel 但无实际效果 |
| 7 | 新增按钮需手写 CSS 类 | 🟢 S3 | 每次加按钮都要查有哪些类、会不会冲突 |

## 二、设计方案

### 2.1 文件结构

```
src/components/Button.tsx     ← 统一按钮组件
src/style.css                  ← 替换 15+ 类为统一 .btn 体系
```

不需要 Hook — 按钮不涉及状态管理逻辑。与 `<Pagination>`（Hook + Component）和 `<Modal>`（纯 Component）一致：只在有状态管理需求时才拆 Hook。

### 2.2 `<Button>` 组件设计

```typescript
// src/components/Button.tsx

import type { ReactNode, ButtonHTMLAttributes } from 'react'

interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  /** 视觉变体 */
  variant?: 'primary' | 'secondary' | 'danger' | 'text' | 'ghost'
  /** 尺寸 */
  size?: 'xs' | 'sm' | 'md' | 'lg'
  /** 加载中状态（显示 spinner 并禁用） */
  loading?: boolean
  /** 前置图标 */
  icon?: ReactNode
  children?: ReactNode
}

export default function Button({
  variant = 'primary',
  size = 'md',
  loading = false,
  icon,
  children,
  className,
  disabled,
  ...rest
}: ButtonProps) {
  return (
    <button
      className={`btn btn-${variant} btn-${size}${className ? ` ${className}` : ''}`}
      disabled={disabled || loading}
      {...rest}
    >
      {loading ? <span className="btn-spinner" /> : icon}
      {children}
    </button>
  )
}
```

### 2.3 变体（Variant）定义

| variant | 背景 | 边框 | 字色 | 适用场景 |
|---------|------|------|------|----------|
| `primary` | `var(--accent)` | 无 | `#fff` | **主操作**：保存、生成、创建、确认、开始分析 |
| `secondary` | 透明 | `1px solid var(--border)` | `var(--text)` | **次操作**：编辑、取消、关闭 |
| `danger` | `var(--danger)` | 无 | `#fff` | **破坏性操作**：删除、清空 |
| `text` | 透明 | 无 | `var(--accent)` | **轻量操作**：状态流转、小链接、hover 下划线 |
| `ghost` | 透明 | 无 | `var(--text-secondary)` | **图标/工具栏**：格式按钮、关闭按钮、hover 显浅背景 |

**设计决策：**

| 决策 | 理由 |
|------|------|
| `text` hover 仅下划线，无背景色 | 对齐 `<a>` 链接的交互预期，轻量操作不抢注意力 |
| `ghost` hover 显浅背景（`rgba(0,0,0,0.06)`），无下划线 | 对齐工具栏按钮预期：悬停反馈但不破坏工具条整洁感 |
| `ghost` 不叫 `icon` | 虽然常用于纯图标，但也能放文字（如关闭的 ✕），命名覆盖性更好 |
| 不提供 `outline` 变体 | 原 Editor 的 `.save-btn` 是 outline accent，但「保存」是最主要的操作，应为 `primary`；若确有次要操作需要 outline，用 `secondary` 即可 |

### 2.4 尺寸（Size）定义

| size | 高度 | padding | 字号 | 适用场景 |
|------|------|---------|------|----------|
| `xs` | 22px | `0 6px` | `0.7rem` | 章节列表内联操作、列表项微型按钮 |
| `sm` | 28px | `0 12px` | `0.8rem` | Toolbar 操作按钮（改写/审查/生成/保存）、面板内紧凑操作 |
| `md` | auto | `8px 16px` | `0.9rem` | **默认**，面板主操作（编辑、保存、开始分析） |
| `lg` | auto | `10px 24px` | `1rem` | Hero CTA（创建项目）、大弹窗确认按钮 |

**设计决策：**

| 决策 | 理由 |
|------|------|
| `sm` 用固定高度 28px | Editor toolbar 中所有操作按钮需要行高一致，固定高度确保对齐 |
| `md` / `lg` 用 auto 高度 + padding | 面板按钮不需要固定高度，padding 控制更自然 |
| `xs` 用固定高度 22px | 对齐原 `.btn-tiny` 的 22px，确保章节列表操作按钮不撑大行高 |
| `min-width`：xs=22px, sm=28px | 纯图标按钮（无 children）也能有合理的点击区域 |

### 2.5 旧类 → 新组件映射

| 旧 CSS 类 | 新用法 |
|-----------|--------|
| `.btn-primary` | `<Button variant="primary" size="md">` |
| `.btn-secondary` | `<Button variant="secondary" size="md">` |
| `.btn-danger` | `<Button variant="danger" size="md">` |
| `.btn-text` | `<Button variant="text" size="md">` |
| `.btn-text.btn-sm` | `<Button variant="text" size="sm">` |
| `.btn-icon` | `<Button variant="ghost" size="md">` |
| `.btn-small` | `<Button variant="secondary" size="xs">` |
| `.btn-small.btn-ai` | `<Button variant="primary" size="xs">` |
| `.btn-tiny` | `<Button variant="ghost" size="xs">` |
| `.btn-tiny-danger` | `<Button variant="danger" size="xs">` |
| `.btn-add` | `<Button variant="primary" size="sm">` |
| `.toolbar-btn` (格式：B/I/U) | `<Button variant="ghost" size="sm">` |
| `.toolbar-btn.rewrite-btn` | `<Button variant="text" size="sm">` |
| `.toolbar-btn.review-btn` | `<Button variant="text" size="sm">` |
| `.toolbar-btn.generate-btn` | `<Button variant="primary" size="sm">` |
| `.toolbar-btn.save-btn` | `<Button variant="primary" size="sm">` |
| `.toolbar-btn.stop-btn` | `<Button variant="danger" size="sm">` |
| `.pagination-btn` | **不整合**（Pagination 组件内部使用，自管理样式） |
| `.chapter-add-btn` | 特殊场景，保留专用样式或使用 `<Button variant="ghost">` |
| `.character-dropdown-btn` | 非按钮，保留专用样式 |
| `.fetch-models-btn` | `<Button variant="secondary" size="sm">` |

### 2.6 使用示例

#### Editor toolbar 统一后

```tsx
// 改前 — 6 个同级按钮用 4 套不同 CSS
<button className="toolbar-btn">B</button>                            // ghost, transparent
<button className="toolbar-btn rewrite-btn">✏️ 改写</button>          // text, auto width
<button className="toolbar-btn review-btn">🔍 审查</button>          // text, different padding
<button className="toolbar-btn generate-btn">✨ 生成</button>         // primary, filled!
<button className="toolbar-btn save-btn">保存</button>               // secondary, outlined!

// 改后 — 统一组件，variant 表达层级
<Button variant="ghost" size="sm">B</Button>
<Button variant="text" size="sm" icon="✏️">改写</Button>
<Button variant="text" size="sm" icon="📝">扩写</Button>
<Button variant="text" size="sm" icon="✨">润色</Button>
<Button variant="text" size="sm" icon="🔍">审查</Button>
<Button variant="primary" size="sm" icon="✨">生成</Button>
<Button variant="primary" size="sm">保存</Button>
```

> **这里生成和保存仍是 `primary`（突出），但所有按钮的 `size` 统一为 `sm`（28px 高、0.8rem 字、12px padding），视觉上不再乱七八糟。** Variant 表达操作层级，size 保证物理一致性。

#### 面板操作统一后

```tsx
// 角色面板
<Button variant="primary" size="xs">+</Button>         // 新建角色（原 .btn-small）
<Button variant="primary" size="xs" loading>AI</Button> // AI 生成（原 .btn-small.btn-ai）
<Button variant="primary" size="md">保存</Button>       // 保存（原 .btn-primary）
<Button variant="secondary" size="md">编辑</Button>    // 编辑（原 .btn-secondary）
<Button variant="text" size="md">取消</Button>          // 取消（原 .btn-text）

// 世界观面板 — 完全相同的用法
<Button variant="primary" size="md">保存</Button>
<Button variant="secondary" size="md">编辑</Button>
<Button variant="text" size="md">取消</Button>

// 大纲面板 — 完全相同的用法
<Button variant="primary" size="md">保存</Button>
<Button variant="secondary" size="md">编辑</Button>

// 伏笔面板
<Button variant="primary" size="sm">+ 新增伏笔</Button>         // 原 .btn-add
<Button variant="primary" size="md">开始分析</Button>           // 原 .btn-primary
<Button variant="text" size="sm">推进</Button>                  // 原 .btn-text
<Button variant="danger" size="sm">删除</Button>                // 原 .btn-text + inline style
```

### 2.7 RewriteButtons 组件处理

`RewriteButtons` 是已有的复用组件（改写/扩写/润色三连按钮组），保留其作为**组合组件**，内部三个按钮改用 `<Button>`：

```tsx
// src/components/RewriteButtons.tsx — 改后
export default function RewriteButtons({ enabled, loading, onRewrite, onExpand, onPolish }: Props) {
  return (
    <span className="rewrite-btn-group">
      <Button variant="text" size="sm" icon="✏️" disabled={!enabled || loading}
        onClick={onRewrite} title={enabled ? '改写选中文字' : '请先选中文字'}>
        改写
      </Button>
      <Button variant="text" size="sm" icon="📝" disabled={!enabled || loading}
        onClick={onExpand} title={enabled ? '扩写选中文字' : '请先选中文字'}>
        扩写
      </Button>
      <Button variant="text" size="sm" icon="✨" disabled={!enabled || loading}
        onClick={onPolish} title={enabled ? '润色选中文字' : '请先选中文字'}>
        润色
      </Button>
    </span>
  )
}
```

## 三、弹窗按钮布局规范

> 解决两个问题：① 按钮左右顺序不一致、② 底部操作栏容器样式不统一。

### 3.1 规则一：按钮顺序 — 取消左，确定右

任何包含"确定类操作 + 取消类操作"的弹窗/表单底部，统一布局为：取消/次要操作在左，确定/主要操作在右。

```
┌──────────────────────────────────────┐
│  弹窗内容                             │
│                                      │
│          [取消]  [确定/保存/删除]      │  ← 取消左，确定右
└──────────────────────────────────────┘
```

这是所有主流平台的惯例：

| 平台 | 规则 |
|------|------|
| macOS Human Interface Guidelines | 取消在左，默认操作在右 |
| Windows UX Guidelines | 取消在左，确定在右 |
| Material Design | 次要操作在左，主要操作在右 |
| Ant Design | `cancelButtonProps` + `okButtonProps`，默认取消左确定右 |

### 3.2 适用场景（按钮顺序）

| 场景 | 左按钮 | 右按钮 |
|------|--------|--------|
| 删除确认 | `取消`（`secondary`） | `确定删除`（`danger`） |

## 六、样式设计

```css
/* ─── Button — unified component ─────────── */

/* ▸ Base */
.btn {
  display: inline-flex;
  align-items: center;
  justify-content: center;
  gap: 6px;
  border: none;
  border-radius: var(--radius-sm);
  font-family: inherit;
  cursor: pointer;
  transition: all 0.15s;
  white-space: nowrap;
  user-select: none;
  line-height: 1;
}

.btn:disabled {
  opacity: 0.5;
  cursor: not-allowed;
}

/* ▸ Sizes */
.btn-xs { height: 22px; min-width: 22px; padding: 0 6px; font-size: 0.7rem; }
.btn-sm { height: 28px; min-width: 28px; padding: 0 12px; font-size: 0.8rem; }
.btn-md { padding: 8px 16px; font-size: 0.9rem; }
.btn-lg { padding: 10px 24px; font-size: 1rem; }

/* ▸ Primary — filled accent */
.btn-primary { background: var(--accent); color: #fff; }
.btn-primary:hover:not(:disabled) { background: var(--accent-hover); }

/* ▸ Secondary — outlined */
.btn-secondary {
  background: var(--bg);
  color: var(--text);
  border: 1px solid var(--border);
}
.btn-secondary:hover:not(:disabled) { border-color: var(--accent); color: var(--accent); }

/* ▸ Danger — filled red */
.btn-danger { background: var(--danger); color: #fff; }

.btn-text {
  background: none;
  color: var(--accent);
  padding-left: 4px;
  padding-right: 4px;
  min-width: unset;
}
.btn-text:hover:not(:disabled) { text-decoration: underline; }

/* ▸ Ghost — toolbar / icon style */
.btn-ghost {
  background: transparent;
  color: var(--text-secondary);
}
.btn-ghost:hover:not(:disabled) {

}

/* ▸ Loading spinner */
.btn-spinner {

  height: 14px;

```

**样式设计决策：**

| 决策 | 理由 |
|------|------|
| `gap: 6px` 在 base | icon + text 间距统一，不再需要手动 `margin-right` |
| `min-width` 仅在 xs/sm（固定高度尺寸） | 纯图标按钮无文字时也能保持可点击的正方形区域 |
| xs/sm 用 `height`，md/lg 用 `padding` | 小按钮需要精确高度对齐（toolbar 行高敏感），大按钮用 padding 更灵活 |
| `text` 变体覆写 `min-width: unset` | link 风格按钮不应有最小宽度约束 |
| Loading spinner 用 CSS border 动画 | 零依赖，比引入 SVG 或 icon library 更轻 |

---

## 七、边界情况

1. **纯图标按钮（无 children）**：`min-width` 保证 xs(22px) / sm(28px) 的点击区域。`md`/`lg` 无此约束，调用方应确保有足够内容
2. **loading + icon 互斥**：loading 时显示 spinner 替代 icon，避免两个图标并排
3. **disabled + loading 同时**：`disabled || loading` 确保 loading 时必然 disabled
4. **className 透传**：允许调用方附加自定义类（如 `foreshadow-form`），与旧 CSS 过渡兼容
5. **原生 button 属性透传**：`...rest` 展开到 `<button>`，支持 `onClick`、`title`、`type`、`id` 等
6. **text 变体的 padding**：`padding-left/right: 4px` 覆写 size 的 `0 12px`，因为 link 风格按钮不应有太大内边距。但 `padding-top/bottom` 保持 size 定义（影响高度）
7. **btn-sm 纯图标 + 无文字**：28px 高 + 28px 宽，正方形，对齐 toolbar 格式按钮
8. **换行**：`white-space: nowrap` 防止按钮内文字换行

## 八、迁移兼容策略

### 8.1 新旧 CSS 共存期

Step 1 时新增 `.btn` 体系 CSS，**不删除旧类**。改造期间新旧并存：

```css
/* 新增（Step 1） */
.btn { /* ... */ }
.btn-primary { /* ... */ }
/* ... */

/* 保留（Step 5 删除） */
/* .btn-primary { ... } */  ← 旧定义与新的同名，先注释旧的，新增用 .btn-primary 前缀
```

> **`.btn-primary` 命名冲突**：旧 CSS 已有 `.btn-primary`。解决方案：
> 1. 新 CSS 用 `.btn.btn-primary`（双类选择器），优先级高于单类 `.btn-primary`
> 2. Step 5 统一清理时删除旧的 `.btn-primary`、`.btn-secondary`、`.btn-danger`、`.btn-text`、`.btn-icon` 定义
> 3. 迁移期间旧的 `.btn-primary` 直接改名为 `.btn-old-primary` 作为 fallback（无实际使用后删除）

### 8.2 逐文件迁移

每个文件改动独立，改完即可用。不改动数据层和业务逻辑，只替换 JSX 中的 `<button className="...">` 为 `<Button variant="..." size="...">`。
