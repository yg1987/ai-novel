> 基于以下四个开源项目的分析，结合个人使用场景设计：
> - [character-arc](https://github.com/uu201/character-arc)
> - [QMAI](https://github.com/Mochocyang/QMAI)
> - [oh-story-claudecode](https://github.com/worldwonderer/oh-story-claudecode)
> - [MuMuAINovel](https://github.com/xiamuceer-j/MuMuAINovel)
> 技术路线：Tauri 桌面应用 | 前端 AI 编排层 | 可配置 AI Provider（OpenAI 兼容协议）

## 一、核心工作流（用户视角）

### 第一条线：开书准备（一次性的）

#### 第 1 步：立项

| 用户操作 | 系统行为 | AI 参与 |
|---|---|---|
| 输入书名、类型（玄幻/都市/言情等）、一句话简介 | 创建项目目录结构 | AI 根据类型和简介，生成世界观框架、角色种子、大纲骨架 |
| 确认或微调 AI 的生成结果 | 保存到项目文件 | — |

**产出**：`project.json`（元数据）+ `世界观/`（初版）+ `角色/`（初版）+ `大纲/`（骨架）

#### 第 2 步：精修设定

| 用户操作 | 系统行为 | AI 参与 |
|---|---|---|
| 逐个点开角色，补充细节 | 保存角色文件 | AI 辅助补全角色卡（外貌/性格/动机/说话风格） |
| 完善世界观条目 | 保存到对应目录 | AI 辅助扩展世界观细节 |
| 调整角色关系 | 更新关系图谱数据 | AI 建议关系连线 |

**核心数据模型——角色卡**：
```
name: 林尘
role: 主角
appearance: 白衣黑发，眉眼锋利
personality: 隐忍坚毅，重情义，多疑
background: 前世被挚友背叛，重生回少年时代
motivation: 这一世要守护该守护的人，让背叛者付出代价
speaking_style: 话语简短，很少长篇大论，讽刺时常带冷笑
tags: [重生, 剑修, 成长型]
```

#### 第 3 步：规划大纲

| 用户操作 | 系统行为 | AI 参与 |
|---|---|---|
| 创建分卷 | 卷纲结构化存储 | AI 生成卷的情绪曲线、爽点规划 |
| 在卷下创建章节细纲 | 细纲结构化存储 | AI 生成每章情节点 + **字数预算合同** |
| 拖拽调整顺序 | 更新排序字段 | — |

**章节细纲结构**：
```
chapter_title: 第1章 重生
outline_items:
  - content: 主角在剑道大会上被围攻，含恨而死
    word_budget: 300
    type: 铺垫
  - content: 主角惊醒，发现自己回到了少年时代
    word_budget: 200
    type: 爽点
  - content: 面对曾经的选择岔路口，决定改修剑道
    word_budget: 400
    type: 推进
  - content: 展现前世记忆带来的视野差距
    word_budget: 350
    type: 高压
total_budget: 1250   # 必须在 [目标字数, 目标字数×1.1] 内
target_words: 1200
```

### 第二条线：写作循环（每章重复）

#### 第 4 步：写前准备

**用户操作**：选中一个章节细纲，点击"开始写"

**系统自动做**（无用户操作）：

**上下文引擎装载**以下内容到 AI prompt：
1. **当前章节目标** — 来自细纲
2. **项目文风设定** — 全局风格约束
3. **最近剧情** — 上一章结尾 + 前 3 章摘要 + 每章末尾钩子
4. **角色状态** — 本章出场角色当前知道什么 / 不知道什么
5. **角色认知清单**：
   ```
   林尘知道：对手的剑法破绽、秘境入口位置
   林尘不知道：幕后黑手的真实身份
   读者知道但林尘不知道：反派已在秘境设伏
   ```
6. **未解伏笔** — 按紧迫度排序
7. **相关知识检索** — 向量搜索 + 关键词搜索相关设定

**产出**：一份完整的生成 prompt（用户不需要看，但如果打开"显示上下文"面板可以看到）

#### 第 5 步：写正文

| 用户操作 | 系统行为 |
|---|---|
| 点击"生成本章" | AI 流式输出正文到编辑器（streaming） |
| 随时点击"暂停" | 中断生成流，保留已生成内容 |
| 手动修改某句话 | 编辑器实时更新 |
| 选中文字 → 右键"AI 改写此段" | 局部重写（只改选中范围） |
| 选中文字 → 右键"AI 扩写" | 扩写选中段落 |
| 点击"继续生成" | 从光标位置继续流式生成 |

**可选**：生成完后自动跑 "去 AI 味" 检查：
- 翻转句式检测（"不是 A，而是 B"）
- 万能状语检测（"带着……"）
- 章末总结体检测
- 抽象升华词检测（命运/宿命）
- 万能比喻检测（潮水般/闪电般）
- 禁用标点检查（破折号 `——`、省略号 `……` 用句号/动作打断代替）

#### 第 6 步：写后处理

**用户不需要做任何操作**，系统后台自动完成：

##### ① 记忆提取（Chapter Ingest）

AI 分析本章内容，提取结构化信息：
```
summary: "林尘在秘境中遇到前世仇人，一番交手后发现对方也带着前世记忆"
character_appearances: [林尘, 苏云, 无名老者]
new_characters: [无名老者]                # 首次出场
character_state_changes:
  林尘: 震惊 → 警惕 → 杀意               # 心理状态变化
events:
  - type: 冲突, desc: 与苏云交手
  - type: 发现, desc: 苏云也保留了前世记忆
foreshadowing_planted:
  - desc: 无名老者似乎知道秘境的真正秘密
    target_resolve: 第15章前后
    importance: 高
foreshadowing_resolved:
  - desc: 秘境入口的机关
    resolution: 用前世记忆中的方法破解了
timeline: 秘境第二天
```

##### ② 状态更新

- 更新角色心理状态到角色文件
- 更新角色认知（谁知道了什么新信息）
- 更新伏笔状态（新埋 / 已解）
- 更新全局时间线

##### ③ 一致性审查

AI 自动跑一轮检查，产出审查报告：
```
审查维度：
✓ 时间线冲突        → 无问题
✓ 角色认知越界      → 无问题（林尘不知道的事没有写他知道）
⚠ 伏笔遗忘          → 第8章埋的"刘长老的身份"已过15章仍未提及
✗ 设定自洽          → 第3段"林尘用御剑术飞行"，但世界观设定中御剑术需要金丹期
                    而林尘当前才筑基期 → 建议修改
```

#### 第 7 步：审查 & 修改

| 用户操作 | 系统行为 |
|---|---|
| 查看审查报告 | — |
| 点击"没问题，继续" | 进入下一章写作循环 |
| 点击"AI 修复此问题" | AI 自动修，修完自动复审确认（5.5 复审闭环） |
| 手动修改后点"重新审查" | 重跑审查 |
| 重大问题："重新生成本章" | 回到第 4 步，调整细纲后重走 |

### 2.6 AI 上下文引擎（核心模块）

#### 架构：DataSourceRegistry（可插拔数据源）

每个上下文来源注册为一个 `DataSource` 对象，由注册中心按优先级并发加载：

```
DataSourceRegistry
├── outlineDS (priority: 1)     ← 章节细纲
├── styleDS (priority: 2)       ← 文风设定
├── chapterGoalDS (priority: 3) ← 当前章节目标
├── recentSummaryDS (p: 4)      ← 最近章节摘要
├── prevEndingDS (priority: 5)  ← 上一章结尾
├── characterStateDS (p: 6)     ← 角色当前状态
├── cognitionDS (priority: 7)   ← 角色认知（知道/不知道）
├── foreshadowDS (priority: 8)  ← 未解伏笔 + 紧迫度
├── timelineDS (priority: 9)    ← 故事时间线
├── searchDS (priority: 10)     ← 向量+关键词检索结果
└── writingStyleDS (p: 11)      ← 写作风格约束
```

特性：
- 每个数据源独立加载、独立错误处理（一个失败不影响其他）
- 按 SECTION_PRIORITY 表排序拼入 prompt
- CJK 感知 token 预算控制，超限时自动压缩低优先级数据源
- 稳定前缀缓存：同一生成会话中，轮廓+上下文 block 保持字节不变以触发缓存命中

| 功能 | 说明 |
|---|---|
| 上下文装配 | DataSourceRegistry 并发加载各数据源，按优先级组装 |
| 角色认知注入 | 防止角色信息泄露 |
| 伏笔提醒注入 | 未解伏笔按紧迫度排序追加到 prompt |
| 知识检索注入 | 向量 + 关键词混合搜索相关设定 |
| 上下文预算控制 | CJK 感知 token 估算，超出时自动压缩低优先级 |
| 流式生成 | SSE 协议流式输出 |
| 中断/恢复 | 支持生成过程中断 |

### 2.8 搜索系统

#### 混合搜索架构：RRF 融合

```
用户输入查询
    ↓
关键词分支（TS 端，2.5s 超时）
  ├── CJK 感知分词（单字/双字/停用词过滤）
  ├── 文件名精确匹配 bonus
  └── 内容词频评分
    ↓
向量分支（Rust + LanceDB ANN，2.5s 超时）
  ├── Markdown 递归分块（headingPath 面包屑）
  ├── 调用 embedding API
  └── LanceDB 嵌入式向量库 ANN 搜索（L2 distance → score）
    ↓
RRF 融合（Reciprocal Rank Fusion, K=60）
    ↓
可选 LLM 重排（top-N 以内）
    ↓
结构化搜索结果（带来源标注：keyword/vector/graph）
```

| 功能 | 说明 |
|---|---|
| 关键词搜索 | CJK 分词 + 文件名精确匹配 + 内容词频评分 |
| 向量搜索 | Markdown 感知分块 → embedding → LanceDB ANN 搜索（_distance → 1/(1+d)） |
| RRF 融合 | 基于排名的无偏融合，K=60 标准常数 |
| 多源超时控制 | 每源 2.5s 超时，慢 API 不阻塞整体 |
| 搜索范围 | characters / worldview / chapters / notes / memory / outline |
| 增量索引 | 保存后自动分块 + upsert，无需全量重建 |
| LLM 重排 | 可选：对 top-K 结果用 AI 进行相关性重排 |

#### Markdown 文本分块策略

- 按 heading 层级递归分割（`#` → `##` → `###` → 段落 → 行）
- 每 chunk 携带 headingPath 面包屑（如 `## 力量体系 > ### 修仙境界`）
- 不切割代码块和表格
- chunk 大小 500-1000 tokens，相邻 chunk 50 token 重叠
- 纯确定性算法，相同输入产生相同分块

#### 向量存储：LanceDB（嵌入式 ANN）

- 存储位置：`{project_dir}/.lancedb/` — LanceDB 原生列式存储
- 每次章节保存时 delete-then-insert（按 page_id 删除旧 chunks → 批量写入新数据）
- 使用 Arrow RecordBatch + FixedSizeList<Float32> 向量列
- 搜索使用 LanceDB 原生 ANN 索引：`table.vector_search(embedding).limit(k).execute()`
- 距离转分数：`score = 1.0 / (1.0 + _distance)`（L2 距离 → [0,1] 相似度）
- Rust Tauri command 层：`vector_upsert_chunks` / `vector_search_chunks`

#### 写作统计看板

数据来源：`stats/YYYY-MM.jsonl` — 追加式 JSONL 日志，每个事件一行。

| 追踪事件 | 触发时机 | 数据字段 |
|---|---|---|
| `chapter_saved` | 保存章节 | chapter, char_count, word_count, timestamp |
| `ai_generated` | AI 生成完成 | prompt_tokens, output_tokens, duration_ms |
| `session_start/end` | 进入/离开编辑器 | timestamp |

展示指标（前端计算，不存储）：
- 日更字数柱状图（7 天/30 天）
- 写作速度趋势（字符/分钟）
- 连载进度（当前章节 / 总计划）
- AI 生成 vs 手动写作比例
- 写作连续天数（Streak）
- 伏笔债评分（来自 ForeshadowDebt 分析）

统计面板 UI 遵循 `panel-layout` 约定（AGENTS.md），纯 CSS 可视化（无需额外图表库）。

| AI 辅助分类 | 粘贴素材后 AI 自动建议分类和标签 |
| AI 辅助补全 | 灵感条目可让 AI 扩展成完整段落 |
| 写作时素材面板 | 编辑器右侧可打开素材面板，拖拽/插入到正文 |
| 素材关联项目 | 每条素材可标记关联到某个项目，写前上下文引擎可选装入 |
| 外部导入 | 支持粘贴网页内容、图片 OCR 文字提取、TXT/Markdown 导入 |

### 4.2 模型分工

| 场景 | 推荐模型层级 | 原因 |
|---|---|---|
| 正文写作（小说正文） | 强模型（GPT-4o/Claude Sonnet） | 质量第一 |
| 大纲/设定生成 | 中等模型 | 创意够用即可 |
| 章节分析/记忆提取 | 小模型或本地模型 | 批量跑、量大 |
| 一致性审查 | 小模型或本地模型 | 逻辑判断为主 |
| 向量嵌入 | 专用 embedding 模型 | 可离线（如 bge-m3） |
| 去 AI 味检测 | 规则引擎 + 小模型 | 确定性检查为主 |

### 4.3 AI 调用架构

```
Context Engine (prompt 组装)
    ↓
Streaming Request (SSE)
    ↓
AI Provider (用户配置的目标)
    ↓
Streaming Response (逐 token 推送)
    ↓
Editor (实时渲染)  +  Buffer (完整文本收集)
    ↓
Post-process (去AI味 → 记忆提取 → 审查)
```

## 六、设计原则

1. **本地优先** — 所有数据在本地，不依赖云服务。AI API 可选云端也可选本地模型
2. **文件即存储** — Markdown + JSON，可直接用任何编辑器打开。项目目录 = 完整备份
3. **渐进式** — 先实现核心写作循环（第 4→5→6→7 步），再逐步加辅助功能
4. **记忆为王** — 100 万字长篇的核心问题是 AI 记不住前面写了什么，60% 工程投入在记忆系统
5. **多看少改** — AI 写完后必须有一轮审查，用户确认后才算完成。防止 AI 乱改已写好的内容
6. **可配置** — AI 模型可自由切换，不绑定任何厂商
