# 关系图模块改造计划

> 创建日期：2026-07-16
> 状态：方案设计中，待确认
> 参考：QMAI（https://github.com/Mochocyang/QMAI）图谱系统

---

## 一、背景

关系图 Tab 是 v0.5 引入的功能，当前基于 d3-force 力导向图。问题：

1. 原生 DOM 操作与 React 脱节，维护成本高
2. 数据层和渲染层耦合脆弱（`node not found` 白屏 bug）
3. 只有一种节点类型（角色），快照中已有的地点/物品数据全浪费
4. 无模式切换、无洞察分析、无文档/脑图视图

决定参考 QMAI 的完整图谱设计，改造“关系图”模块。

> 范围说明：本轮只改造 `RelationshipGraph`。现有 `ChapterGraph` 仍依赖 `d3-force`，因此本轮**不移除** `d3-force` / `@types/d3-force`。章节图迁移另开任务。

---

## 二、现状 vs 目标

| 维度 | 当前（d3-force） | 目标（Sigma.js + graphology） |
|---|---|---|
| 渲染库 | d3-force `createElementNS` | Sigma.js v3 + `@react-sigma/core` |
| 图数据 | 手动 nodes/links 数组 | plain graph data + graphology 派生实例 |
| 布局 | d3-force 力模拟 | ForceAtlas2 + 黄金角冷启动 |
| 节点类型 | 1 种（角色） | 7 种（角色/地点/物品/组织/事件/章节/伏笔） |
| 关系类型 | 8 种平铺 | 3 级 tier + 5 信号加权边权重 |
| 显示模式 | 1 种（力导向图） | 3 种（力导向图/文档视图/脑图） |
| 分析模式 | 无 | 5 种（总览/人物/章节/故事线/伏笔） |
| 洞察 | 无 | 惊喜连接 + 知识缺口 + 社区发现（Louvain） |
| 图例 | 静态颜色列表 | 双击切换类型显隐 |
| 过滤 | 无 | 三源合并管线 + empty-graph 兜底 |
| 刷新 | 仅挂载时 | 模式切换 + 手动重新生成按钮 |

## 四、核心数据约定

### 4.1 节点 ID 命名空间

多类型节点必须使用带类型前缀的稳定 ID，避免角色、地点、组织、物品同名时串节点。

```ts
export type GraphNodeId =
  | `character:${string}`
  | `location:${string}`
  | `item:${string}`
  | `organization:${string}`
  | `event:${string}`
  | `chapter:${string}`
  | `foreshadowing:${string}`
```

规则：

- `id`：永远是带命名空间的图节点 ID。
- `label`：展示用原始名称。
- `RelationshipLink.source/target`：永远使用图节点 ID，不再是角色名。
- `relationshipChanges` 仍是旧格式角色名：解析后映射为 `character:${name}`。
- 所有 `addEdge` 前必须检查 source/target 节点存在。

示例：

```ts
character:林烬
location:青云宗
organization:青云宗
item:赤霄剑
chapter:ch003
event:ch003:0
foreshadowing:fs_abc123
```

### 4.2 章节节点 ID

章节节点使用章节稳定 ID，而不是 `order`：

```ts
id = `chapter:${chapter.id}`
label = `第${chapter.order}章 ${chapter.title}`
```

原因：`order` 会因插入/重排章节变化，`chapter.id` 更适合做图节点主键。

### 4.3 事件节点 ID

第一版不跨章合并事件。每条 `timelineEvents` 在所属章节下生成独立节点：

```ts
id = `event:${chapterId}:${index}`
label = timelineEventText
```

后续如果需要跨章事件合并，再引入 `hash(normalizedText)` 或事件库。

### 4.4 plain data 与 graphology 的边界

业务服务层返回 plain data：

```ts
export interface RelationshipGraph {
  nodes: GraphNode[]
  links: RelationshipLink[]
  insights: InsightItem[]
}
```

组件内部或 `GraphLoader` 再把 plain data 转为 graphology Graph 实例，用于 Sigma 渲染、ForceAtlas2、Louvain 等。

这样做的原因：

- plain data 更容易调试、测试、文档视图/脑图复用。
- React state 存 plain object 更稳定。
- graphology Graph 是渲染/分析派生对象，不作为业务持久结构。

```ts

  filters: GraphFilterState,

```

过滤顺序：

1. 手动隐藏节点
2. 类型白名单
3. 类型黑名单
4. 结构节点（`hideStructural`）
5. 最小边权重
6. 移除端点不存在的边
7. 孤立节点（`hideIsolated`）
8. empty-graph 兜底：如果过滤后清空全部节点，则回滚最后一步或提示“一键重置筛选”

注意：最小边权重要在孤立节点之前执行，否则低权重边被过滤后会残留孤立节点。

## 七、数据源映射

`ChapterSnapshot` 现有字段：`characters`、`locations`、`items`、`timelineEvents`。缺少结构化 `events` 和 `organizations`。

| 节点类型 | 数据来源 | 提取方式 |
|---|---|---|
| `character` | `characters/` 目录 | 文件名建节点，正文解析 group/tags/organization |
| `location` | `ChapterSnapshot.locations` | 去重按名称建节点，首次出场记 `firstAppearance` |
| `item` | `ChapterSnapshot.items` | 去重按名称建节点 |
| `chapter` | `listChapters(projectId)` | 每章一个节点，`id = "chapter:{chapter.id}"` |
| `event` | `ChapterSnapshot.timelineEvents` | 每条 timeline 事件建章节内事件节点 |
| `organization` | `worldview/` + 角色卡 | 从文件名/标题/阵营/势力字段提取 |
| `foreshadowing` | `loadForeshadows(projectId)` | 伏笔条目建节点，关联章节/角色 |

### 7.1 角色卡解析策略

角色卡是 Markdown 自由文本，第一版使用轻量规则解析：

```ts
interface ParsedCharacterCard {
  name: string
  group: 'protagonist' | 'supporter' | 'antagonist' | 'neutral'
  tags: string[]
  organizations: string[]
}
```

识别字段：

- `身份：主角/反派/配角/中立`
- `标签：冷静、剑修、复仇`
- `阵营：青云宗`
- `势力：青云宗`

默认值：

```ts
group = 'neutral'
tags = []
organizations = []
```

### 7.2 世界观组织提取策略

第一版只做保守提取：

1. `worldview/` 下文件名作为候选组织名。
2. Markdown 一级标题 `# xxx` 作为候选组织名。
3. 正文里的 `组织：xxx` / `势力：xxx` 作为候选组织名。
4. 去重后生成 `organization:${name}` 节点。

不做复杂 NLP，避免误判。

### 7.3 伏笔节点状态规则

伏笔类型来自 `ForeshadowStatus`：`planted` / `advanced` / `resolved` / `abandoned`。

显示规则：

- 总览模式：显示所有非 `abandoned` 伏笔。
- 伏笔模式：默认显示 `planted` / `advanced`，可切换显示 `resolved`。
- `abandoned` 默认隐藏。

边规则：

- `foreshadowing -> plantedChapter`
- `foreshadowing -> clue.chapterId`
- `foreshadowing -> resolvedChapterId`（如果存在）
- `foreshadowing -> relatedCharacters`

## 八、边生成与权重

### 8.1 5 信号边权重算法

| 信号 | 权重 | 说明 |
|---|---|---|
| 直接链接 | ×3.0 | A↔B 存在关系记录 |
| 同源共现 | ×4.0 | 同一章快照同时出现 |
| 共同邻居 | ×1.5 | Adamic-Adar 公式 |
| 类型亲和 | ×1.0 | 跨类型 > 同类型 |
| 关系强度 | ×1.5 | tier 反哺权重 |

### 8.2 共现边规模控制

多类型图不能无脑两两共现，否则边数会爆炸。规则：

- `chapter -> entity` 是结构边，`structural: true`，默认可隐藏。
- `character -> character` 共现边只有满足以下任一条件才保留：
  - 有显式 `relationshipChanges`
  - 共现次数 ≥ 2
  - 权重超过当前模式阈值
- `character -> location/item/event` 可保留，权重较低。
- `location -> item` / `item -> event` 不默认两两连接，除非后续有明确规则。
- 每种分析模式可设置 `minimumEdgeWeight`。
- 如果大图性能不足，可增加最大边数限制，按 `weight` 排序保留 Top N。

### 8.3 关系合并规则

同一对节点可能从多个来源生成边：显式关系、共现、结构关系、伏笔关联。合并策略：

- key 使用无向边：`[source, target].sort().join('::')`
- 显式关系优先保留 `type`
- `mentions` 累加
- `firstMentioned` 取最小
- `lastMentioned` 取最大
- `weight` 累加后归一化
- `description` 保留最早显式关系，或合并为数组后展示

## 十一、关键问题与解决方案

### 11.1 `node not found` 白屏 bug 的修复

原因：快照的 `relationshipChanges` 或共现推断中出现了没有角色文件的角色名，d3-force 初始化时找不到节点。

换到 Sigma.js + graphology 后，graphology 的 `addEdgeWithKey` 同样会在 `source` 或 `target` 节点不存在时报错。

方案：

- 所有 `addEdge` 之前先检查 `graph.hasNode(source) && graph.hasNode(target)`。
- plain data 阶段也要过滤端点不存在的边。
- 过滤管线第 6 步再次移除端点不存在的边。
- 加载失败显示错误，不白屏。

### 11.2 重新生成功能

用户手动触发 AI 重新分析所有章节、覆盖快照、刷新关系图。

方案：

- 新增 `src/services/snapshotRegeneration.ts`，不要引用不存在的 `regenerateAllSnapshots`。
- 在 `RelationshipGraph.tsx` 侧栏顶部加「🔄 重新生成」按钮。
- 点击后逐章显示进度：`正在分析第 3/12 章：xxx`。
- 单章失败不中断，显示失败项，继续后续章节。
- 全部完成后自动调用 `loadRelationshipGraph` 刷新图数据。
- 失败不销毁当前图。

### 11.3 右键菜单与跨 Tab 跳转

右键菜单需要依赖上层传入导航回调。`ProjectView.tsx` 需要把已有导航函数传给 `RelationshipGraph`。

如果未传回调，对应菜单项禁用或隐藏。

### 11.4 `d3-force` 依赖处理

本轮关系图迁移后，`d3-force` 仍被 `ChapterGraph` 使用。因此：

- 不删除 `d3-force`
- 不删除 `@types/d3-force`
- 不修改 `ChapterGraph`

如果后续要彻底移除 `d3-force`，需要新增独立计划：迁移 `ChapterGraph` 到 Sigma/graphology。

### 11.5 类型重复与职责边界

- `src/types/novel.ts`：领域数据类型。
- `src/lib/graph-mode.ts`：模式/视图/标签可见性类型。
- `src/lib/graph-insights.ts`：洞察算法。
- `src/services/relationshipStore.ts`：数据加载和基础派生。
- `RelationshipGraph.tsx`：UI 状态、交互、渲染。

不要在多个文件重复定义 `GraphMode` 或 `computeInsights`。
