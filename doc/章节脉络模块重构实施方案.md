# 章节脉络模块重构实施方案

**状态**：实施前置条件已明确，可按 Phase 0 开始实施

**修订日期**：2026-07-22

**模块定位**：章节脉络是一个只读的“计划与已记录执行情况校验”工具。它帮助作者发现故事是否按预期推进，并定位应当修改的正文、细纲或伏笔记录；它不是手工维护的关系图，也不直接修改任何章节内容。

---

## 1. 设计结论

章节脉络分为两个严格隔离的层级：

1. **计划校验层**：不调用 AI，自动加载。基于章节目录和伏笔的计划/记录字段，展示“计划在何处发生”与“当前记录显示何处发生”的差异。
2. **AI 内容分析层**：只由用户手动启动。基于正文生成分析建议、证据和置信度，用来辅助判断“已记录的执行是否真的符合正文”。AI 结果不修改正文，也不自动改写正式的伏笔计划或回收记录。

这两个层级不能混淆：

- “计划回收第 65 章、已记录回收第 68 章”是确定事实，属于计划校验层。
- “AI 判断第 68 章可能没有完成回收”是分析建议，属于 AI 内容分析层。
- 用户发现偏差后，应打开第 65/68 章、细纲或伏笔面板修改源数据；章节脉络不提供拖线、改关系或直接编辑剧情的入口。

---

## 2. 要解决的用户问题

作者打开这个 Tab 时，需要快速回答：

1. 某个伏笔在哪一章埋设、计划回收，以及正式记录在哪些章节推进或回收？
2. 当前项目记录显示它实际在哪一章推进或回收？
3. 它是按计划、提前、延迟，还是尚未回收？
4. 哪些计划已经超过预期章节而没有执行记录？
5. 如果 AI 已分析正文，正文证据是否支持当前记录？
6. 发现问题后，应该打开哪一章或哪条伏笔去修正？

模块不承担以下职责：

- 不替作者决定故事应如何发展。
- 不把 AI 推断当成故事事实。
- 不让作者在图上维护一份脱离正文的“平行剧情”。
- 不在用户保存章节或打开 Tab 时隐式调用 AI。

---

## 3. 当前实现的问题

现有 `ChapterGraph` 会读取全部章节正文、按顺序生成相邻边，再叠加伏笔边并使用力导向布局展示。这不适合作为章节执行校验工具。

主要问题：

- 章节节点仅用 `chapterId` 作为键，多卷都存在 `ch001` 时会覆盖或错连。
- 打开 Tab 会逐章读取正文；500 章、1000 章时会产生大量串行 I/O。
- 正文只被用于计算字数，当前关系并不是根据正文走向生成。
- 力导向布局不能稳定表达章节顺序。
- `continuity` 只有图例，没有真实数据来源。
- 图打开后没有可靠的刷新和过期状态。
- 当前 AI 快照分析将上一章快照作为下一章输入，修改前面的章节会产生从该章到末章的级联重分析风险。

因此，本次重构不在旧图上继续加功能，而是替换为两阶段章节脉络。

## 4. 数据真相与术语

### 4.1 章节引用

所有新数据统一使用完整章节引用：

```ts
interface ChapterRef {
  volume: string
  chapterId: string
}
```

用于内存 Map、索引和任务键时使用 `volume:chapterId`。不得在任何跨卷逻辑中把 `chapterId` 当作全局唯一 ID。项目已有的 `ChapterRef`、`ChapterKey` 和 `chapterRefKey` 是唯一的章节引用工具，不再新增第二套 key 规则。

`ChapterKey` 只用于内存状态和 JSON 的对象键，不能作为文件名。涉及章节详情文件时，必须使用第 5.3 节定义的 `chapter-file-key`。

### 4.1.1 全书顺序与未来计划章节

章节脉络的唯一排序入口为 `buildChapterSequence(chapters)`。它必须返回：

```ts
interface ChapterSequence {
  chapters: ChapterMeta[]
  positionByKey: Map<ChapterKey, number> // 1-based 全书位置
  lastWrittenPosition: number
}
```

排序规则固定为：先按内部卷标识使用 `Intl.Collator('zh-CN', { numeric: true, sensitivity: 'base' })` 排序，再按 `chapter.order`，最后按 `chapter.id`。不得依赖 `listChapters` 的返回顺序，也不得在章节脉络中调用现有的裸 `compareRefs`。自定义卷名和章节标题只用于显示，永不参与排序。

`plantedChapter`、`progress.chapter`、`recordedResolutionChapter` 必须引用章节目录中的真实章节。`plannedResolutionChapter` 额外允许引用“未来章节”：卷必须已经存在于章节目录，表单以正整数章序号输入并通过唯一的 `formatChapterId(order)` 生成规范 `chapterId`（例如 `65 → ch065`），且将该引用按上述规则插入后的位置严格位于 `lastWrittenPosition` 之后。未来计划章节不出现在正文时间线卡片中，但要在对应卷的末尾显示为“计划位置”。其余不存在的计划引用均为 `invalid-reference`。

这一定义使 `pending` 可达：已有正文为空仍算“已写到”；尚未创建的、合法未来计划章节不算“已写到”。作者需要计划未来卷时，必须先在写作模块创建该卷的首章，再选择该卷中的未来章节号。

### 4.2 三类数据

| 数据 | 含义 | 是否可由 AI 修改 |
|---|---|---|
| 创作源数据 | 章节正文、细纲、作者维护的伏笔计划和记录 | 否 |
| 校验结果 | 计划章节与已记录章节的差异、缺失和逾期 | 否，完全由规则计算 |
| AI 分析结果 | 基于正文的摘要、候选回收判断、证据、置信度 | 可重新生成，但不能覆盖创作源数据 |

### 4.3 “实际”的两个含义

界面必须明确区分：

- **已记录执行**：伏笔记录中的推进/回收章节。这是当前项目的正式结构化记录。
- **AI 内容判断**：AI 从正文中发现的候选执行情况。这是建议，不是正式记录。

例如：

```text
计划回收：第 65 章
已记录回收：第 68 章
校验结果：延迟 3 章
AI 内容判断：第 68 章疑似完成回收（0.86）
```

如果 AI 判断与已记录回收冲突，显示“需要核对”，并提供“打开第 68 章”和“打开伏笔”入口；不自动改写任何字段。

## 5. 数据模型

本次不考虑旧数据兼容和字段迁移，但这不等于静默删除用户数据。切换时删除旧 `ChapterGraph` 代码和旧章节图能力；已有项目的伏笔数据通过一次用户明确确认的“初始化新伏笔数据”流程重置。用户需在新伏笔模块重新建立伏笔源数据。不得自动删除章节正文、细纲、角色设定、世界观和素材等非本次重构范围的创作内容。

新伏笔存储必须有独立版本：

```ts
interface ForeshadowStore {
  schemaVersion: 1
  entries: ForeshadowEntry[]
  updatedAt: string
}
```

读取规则是严格的：缺失或空文件返回新的内存空 store；只有 `schemaVersion === 1` 且全部字段合法时才可正常读取；非空的旧 schema、未来 schema 或 JSON 损坏必须返回带错误码的结果，不能降级为“无伏笔”。`loadForeshadows` 的业务调用方应接收该错误并停止写入，只有伏笔面板中的重置流程可以解除该状态。

重置流程必须使用通用 `Modal`，显示将要处理的文件并要求二次确认。文件名中的 `<timestamp>` 固定使用 Windows 安全的 UTC 格式 `yyyyMMddTHHmmssZ`。确认后按以下顺序执行：

1. 读取并原子写入 `memory/legacy-backups/foreshadows-<timestamp>.json`；重新读取并校验内容 hash 一致后，才允许继续。
2. 若存在旧派生文件 `memory/foreshadow-inspiration.json`，先备份到同一目录并完成相同校验；`memory/foreshadow-config.json` 保留，因为它是项目设置而非伏笔源数据。
3. 原子替换 `memory/foreshadows.json` 为新的空 `ForeshadowStore`。
4. 删除已备份的旧 `memory/foreshadow-inspiration.json`。删除仅属于旧章节图的派生文件时必须逐个列名；当前实现没有已知的旧章节图磁盘缓存，严禁按目录通配或删除未知文件。

步骤 1–2 失败时必须停止并保留原 `foreshadows.json`；步骤 3 成功后不回滚已完成的源数据重置，步骤 4 失败只报告“旧派生缓存待清理”并允许重试。`memory/snapshots/` 始终不在该流程中。

`memory/snapshots/` 当前仍被关系图和写作上下文使用，不属于章节脉络专属缓存，Phase 0 不删除它。章节脉络不得读取、写入或依赖该目录；已有快照链路也不得再自动修改新伏笔的正式字段。

### 5.1 伏笔计划和执行记录

新伏笔字段统一使用 `ChapterRef`：

```ts
interface ForeshadowProgress {
  chapter: ChapterRef
  description: string
  recordedAt: string
}

interface ForeshadowEntry {
  id: string
  name: string
  description: string
  status: 'planted' | 'advanced' | 'resolved' | 'abandoned'
  category: ForeshadowCategory
  importance: number
  plantedChapter: ChapterRef
  plannedResolutionChapter?: ChapterRef
  recordedResolutionChapter?: ChapterRef
  progress: ForeshadowProgress[]
  resolutionPlan?: string
  relatedCharacters: string[]
  notes: string
  createdAt: string
  updatedAt: string
}
```

字段含义：

- `plannedResolutionChapter`：作者计划的回收位置。本版本不提供“计划推进”字段；推进只记录已经发生的正式执行。
- `recordedResolutionChapter`：当前项目正式记录的回收位置。
- `progress`：当前项目正式记录的推进位置。
- `status`：当前正式状态。

AI 分析不得直接写入以上任何字段。

### 5.2 计划校验结果

```ts
type ForeshadowExecutionState =
  | 'abandoned'
  | 'unplanned'
  | 'pending'
  | 'on-schedule'
  | 'early'
  | 'late'
  | 'overdue'
  | 'record-incomplete'
  | 'invalid-reference'

type ForeshadowAiState =
  | 'not-analyzed'
  | 'ready'
  | 'stale'
  | 'needs-review'
  | 'failed'

interface ForeshadowExecutionCheck {
  foreshadowId: string
  plantedChapter: ChapterRef
  plannedResolutionChapter?: ChapterRef
  recordedResolutionChapter?: ChapterRef
  progress: ForeshadowProgress[]
  state: ForeshadowExecutionState
  aiState: ForeshadowAiState
  chapterDelta?: number
  message: string
}
```

计算规则使用第 4.1.1 节的全书稳定位置，不使用裸章节数字。按卷筛选只影响展示，不改变全书位置和校验结论。

本版本中“已写到某章”定义为章节目录中已经存在该章节文件；打开 Tab 不读取正文来重新判断。正文是否为空只作为章节卡的辅助展示状态，不改变逾期计算。

状态按以下优先级产生：

1. `status === 'abandoned'`：`abandoned`。不参与正常、逾期、提前或延迟统计，但保留在列表并可单独筛选。
2. 任一埋设、推进或已记录回收引用不存在，或计划回收引用既不是真实章节也不是第 4.1.1 节定义的合法未来章节：`invalid-reference`。
3. 正式状态与记录不一致（`resolved` 却没有 `recordedResolutionChapter`、未 resolved 却存在记录回收、记录回收早于埋设章节、任一推进早于埋设章节，或 `advanced` 没有任何推进记录）：`record-incomplete`。
4. 无计划回收章节：`unplanned`。
5. 有计划、未记录回收、尚未写到计划章节：`pending`。
6. 有计划、未记录回收、已写到或超过计划章节：`overdue`。
7. 已记录回收位置等于计划位置：`on-schedule`。
8. 已记录回收位置早于计划位置：`early`。
9. 已记录回收位置晚于计划位置：`late`，`chapterDelta` 为全书顺序差。

同一章可以有多条推进记录；不得仅因重复章节引用判为不完整。`abandoned` 始终按优先级 1 覆盖其他问题并保留原始记录，供作者自行检查。

`aiState` 独立于 `state`：AI 与正式记录冲突时为 `needs-review`，不得覆盖计划校验状态。筛选中的“正常”对应 `pending`、`on-schedule`、`early`、`unplanned`；“待核对”对应 `invalid-reference`、`record-incomplete` 或 `aiState === 'needs-review'`。

每条伏笔的 `aiState` 由其真实章节引用（埋设、推进、已记录回收）关联的索引项聚合，优先级固定为 `needs-review`、`failed`、`stale`、`not-analyzed`、`ready`：存在未过期的冲突 finding 时为 `needs-review`；否则存在当前输入失败的分析为 `failed`；否则存在过期项或过期 finding 为 `stale`；否则全部关联章节已有当前输入的 ready 分析时为 `ready`；其余为 `not-analyzed`。合法未来计划章节不要求 AI 分析。

### 5.3 AI 分析索引

AI 结果是可丢弃、可重建的派生数据，存储在独立目录：

```text
memory/chapter-flow/
  index.json
  analyses/
    <chapter-file-key>.json
  findings/
    <finding-id>.json
```

`chapter-file-key` 使用 `base64url(UTF-8(JSON.stringify([volume, chapterId])))`。`volume:chapterId` 只用于内存 key，不能直接作为文件名，以兼容 Windows 文件系统。

`index.json` 是打开 Tab 时的唯一分析索引来源，不需要逐个读取 1000 个分析文件：

```ts
interface ChapterAnalysisIndexItem {
  ref: ChapterRef
  contentHash?: string
  analysisInputHash?: string
  analyzedInputHash?: string
  lastAttemptInputHash?: string
  lastAttemptedAt?: string
  status: 'missing' | 'stale' | 'ready' | 'failed'
  analyzedAt?: string
  summary?: string
  error?: string
}

interface ChapterFlowFindingSummary {
  id: string
  type: ChapterFlowFindingType
  foreshadowId?: string
  source: ChapterRef
  target?: ChapterRef
  summary: string
  confidence: number
  evidenceCount: number
  sourceContentHashes: Record<ChapterKey, string>
  generatedAt: string
  status: 'ready' | 'stale'
}

interface ChapterFlowIndex {
  schemaVersion: 1
  revision: number
  updatedAt: string
  chapters: ChapterAnalysisIndexItem[]
  findings: ChapterFlowFindingSummary[]
}
```

索引项状态必须由当前输入重算，不能只相信持久化的 `status`：`ready` 表示 `analysisInputHash === analyzedInputHash` 且详情可读取；`stale` 表示存在可用旧分析但两者不等；`failed` 表示不存在当前可用分析、`lastAttemptInputHash === analysisInputHash` 且本次尝试失败；其余为 `missing`。若输入在失败后变化，旧失败自动回到 `missing`，不能继续显示为当前失败。

`index.json` 是轻量读取模型：只保存时间线、筛选、摘要卡所需字段，不保存原文摘录、详细事件或完整 finding。它在同一项目内串行、原子写入，保存正文、正文扫描和 AI 任务不得互相覆盖更新。

章节目录中存在而索引没有的章节，在轻量刷新时作为内存中的 `missing` 项合并；不得为了补齐这些项读取正文。后续保存、扫描或 AI 分析会把对应项写入索引。已删除章节在章节删除钩子中主动清理，不能依赖普通刷新悄悄删除详情。

`analyses/<chapter-file-key>.json` 保存该章详细分析；`findings/<finding-id>.json` 保存跨章或单章 finding 的完整证据。只有用户选中该章节或某条 AI 发现时才读取对应详情文件：

```ts
interface ChapterAnalysisDetail {
  schemaVersion: 1
  ref: ChapterRef
  contentHash: string
  analysisInputHash: string
  analyzedAt: string
  summary: string
  keyEvents: string[]
  endingHook: string
  findingIds: string[]
}
```

### 5.4 AI 发现

```ts
type ChapterFlowFindingType =
  | 'possible-resolution'
  | 'possible-advance'
  | 'possible-continuation'
  | 'record-conflict'

interface ChapterFlowFinding {
  id: string
  type: ChapterFlowFindingType
  foreshadowId?: string
  source: ChapterRef
  target?: ChapterRef
  summary: string
  evidence: {
    chapter: ChapterRef
    quote: string
    contentHash: string
  }[]
  confidence: number
  sourceContentHashes: Record<ChapterKey, string>
  generatedAt: string
}
```

发现只作为展示和定位依据：

- 不直接修改伏笔状态。
- 不直接写入回收章节。
- 不直接修改正文或细纲。
- 当任一 `sourceContentHashes` 中由 `ChapterKey` 指向的 hash 与当前正文 hash 不一致时，自动标记为过期。
- AI 输出的 `foreshadowId` 必须是本次输入上下文中提供的唯一 ID；找不到或重名时不建立正式伏笔关联。
- 证据摘录必须能在对应章节的规范化正文中精确验证；不能验证的 finding 不写入索引或详情文件。

“当前正文 hash”指索引中最近一次保存或正文扫描得到的 hash；应用不能在不读取正文的情况下发现外部文件改动。用户执行“检查正文变更”前，外部改动只会显示为“尚未检查”，不得错误宣称 finding 仍然有效。

### 5.5 派生存储事务与损坏恢复

前端新增 `atomicWriteProjectFile(projectId, subdir, filename, content)`，对应 Rust Tauri 命令 `atomic_write_project_file`。它只允许写入指定项目目录内的相对路径，且必须满足以下契约：

1. 在目标文件同目录创建带随机后缀的临时文件，完整写入、flush 并同步后才替换目标文件。
2. 成功返回后，读取者只能看到旧完整文件或新完整文件，不能看到空文件、截断 JSON 或目标暂时缺失。
3. 替换失败时保留旧目标文件并清理临时文件；Windows 上的已存在目标替换必须使用支持 replace 的实现，不能先删除目标再 rename。
4. `index.json`、分析详情、finding 详情和旧数据备份全部使用该命令；普通创作源文件继续沿用各自的保存流程。

所有章节脉络索引写入必须经过 `updateChapterFlowIndex(projectId, mutator)`：它在单个 WebView 进程中按项目维护 Promise 队列，在队列内部重新读取最新索引、应用 mutator、将 `revision` 加一，再原子写回。正文保存、正文扫描、AI 任务、章节创建和章节删除不得绕过该入口。

若 `index.json` 不能解析，读取服务返回“索引损坏”状态并保留原始内容；下一次用户明确执行刷新、扫描或分析时，先将原始内容原子备份到 `memory/chapter-flow/recovery/index-<timestamp>.corrupt.json`，再创建空索引。这里的 `<timestamp>` 同样使用 `yyyyMMddTHHmmssZ`。伏笔源数据损坏绝不适用这一自动恢复策略，必须停在第 5 节的错误状态。

## 6. 页面与交互设计

### 6.1 默认页面：计划校验层

默认进入页面即加载计划校验层，不等待 AI，也不读取全书正文。

```text
章节脉络
├── 工具栏
│   ├── 范围：全书 / 指定卷
│   ├── 状态筛选：全部 / 正常 / 逾期 / 延迟 / 待核对 / 已废弃
│   ├── 刷新视图
│   ├── 检查正文变更
│   └── 更新 AI 分析
├── 校验摘要
│   ├── 按计划回收
│   ├── 提前 / 延迟
│   ├── 已逾期
│   └── AI 分析待更新
└── 主内容
    ├── 左侧：伏笔计划与执行对照列表
    └── 右侧：按卷排列的章节时间线
```

页面遵守项目的左右分栏高度传递规则：根容器 `flex: 1; overflow: hidden`，工具栏和摘要 `flex-shrink: 0`，左右内容区使用独立滚动容器。禁止 `height: calc(100vh - Npx)`。

### 6.2 伏笔对照列表

每条伏笔显示：

```text
主角身世
埋设：卷一 · 第 12 章
计划回收：卷二 · 第 65 章
已记录回收：卷二 · 第 68 章
状态：延迟 3 章
AI：已分析 / 待分析 / 需要核对
```

列表操作只包含：

- 选中并定位右侧时间线。
- 打开伏笔面板。
- 打开计划章节、已记录章节或 AI 证据章节；合法未来计划章节只定位到时间线计划标记，并提示作者在写作模块创建该章后再打开。

不提供“修改关系”“拖动节点”“手动回收”按钮。

列表量较大时使用现有 `Pagination` 组件；筛选条件变化后回到第一页。

### 6.3 章节时间线

- 按卷分组，卷之间明确分隔。
- 卷内按章节顺序固定排列，位置不随刷新变化。
- 章节卡展示序号、标题、正文状态、相关计划标记、已记录推进/回收标记和 AI 分析状态。
- 计划标记与已记录标记使用不同颜色和图形，不能仅用颜色表达差异。
- 点击章节卡打开详情抽屉，展示与该章相关的计划、记录、AI 发现和跳转操作。
- 超过 200 章时使用 `@tanstack/react-virtual` 虚拟滚动；1000 章不得一次渲染全部卡片和详情。

时间线可显示“计划位置”和“已记录位置”的连线或同一轨道标记，但不使用力导向图，不允许连线位置改变章节顺序。

### 6.4 AI 发现详情

AI 发现应明确展示：

- 判断类型，例如“可能回收”或“记录冲突”。
- 置信度。
- 分析生成时间。
- 证据章节和原文摘录。
- 当前是否过期。
- `打开证据章节`和`打开伏笔`按钮。

AI 发现详情不提供“确认并写入”按钮。作者需要修改正式记录时，必须进入伏笔面板；需要调整叙事时，必须进入章节或细纲。

## 7. 刷新与分析操作

### 7.1 刷新视图：轻量、自动可用

`刷新视图`只读取：

- 章节目录和显示标题。
- 伏笔计划与执行记录。
- `memory/chapter-flow/index.json`。

它不读取每章正文、不重新计算正文 hash、不调用 AI。打开 Tab 时允许自动执行一次轻量刷新；用户点击按钮时也可以随时执行。若章节标题、伏笔计划上下文、prompt 版本或模型版本发生变化，刷新时基于已保存的 `contentHash` 重算 `analysisInputHash`，不读取正文；新指纹与 `analyzedInputHash` 不一致时标记为 `stale`。

`chapterFlowHash` 是唯一的 hash 实现，保存、扫描、AI 输入和证据验证必须共用它。规范化正文依次执行：将块级结束标签和 `<br>` 转为换行、移除其他 HTML 标签、解码 HTML 实体、将 NBSP 转为空格、统一 CRLF 为 LF、Unicode NFC 规范化、移除每行尾随空白并 trim。`contentHash = SHA-256(UTF-8(normalizedContent))`。

`analysisInputHash` 使用键排序后的 canonical JSON 计算 SHA-256，输入固定为 `hashSchemaVersion`、`contentHash`、完整 `ChapterRef`、当前显示标题、实际发送给模型且已排序的伏笔上下文、prompt 内容 hash、provider 标识、model 标识和正文截断上限。任何实际输入变化都必须改变 hash；未发送给模型的数据不得纳入 hash。章节脉络的 prompt 内容与 `hashSchemaVersion` 变更都必须使旧分析过期。

即使有 1000+ 章，这个操作只处理目录元数据、一个伏笔文件和一个索引文件，成本可控。

### 7.2 保存后过期标记与检查正文变更

应用内正常保存可直接得到本次正文内容。自动保存、组件卸载保存和手动保存必须统一经过保存协调器，而不是分别直接调用 Tauri 命令。协调器以 `projectId:chapterRefKey` 为键维护串行队列；每项操作先持久化正文，成功后才调用同一个保存后钩子：

```ts
markChapterSaved(projectId, ref, html): Promise<void>
```

手动保存仍创建版本历史，自动保存和卸载保存不创建版本历史；三者共享相同的正文 hash 和索引更新逻辑。编辑器只在正文持久化成功后更新 `lastSaved`。后到的保存必须排在同一章节队列末尾，保证旧保存不会在新保存之后覆盖正文或索引状态。

该钩子必须：

1. 使用与 AI 输入一致的规范化正文计算 SHA-256 `contentHash`。
2. 基于正文 hash、章节标题、伏笔计划上下文版本、prompt 版本和模型版本计算 `analysisInputHash`。
3. 没有 `analyzedInputHash` 时保持 `missing`；两者不一致时标记为 `stale`；相同时保持 `ready`。
4. 将证据涉及该章且 hash 已不匹配的 finding 摘要标为 `stale`。
5. 不读取其他章节正文，不调用 AI。

索引更新必须通过第 5.5 节的项目队列和原子写入执行。索引更新失败不得回滚已经成功的正文保存：协调器返回 `contentSaved: true, flowIndexUpdated: false`，手动保存显示“正文已保存，章节脉络索引更新失败”并提供重试；自动保存和卸载保存记录可重试任务，下一次轻量刷新显示待修复状态。新建空章节通过 `markChapterCreated` 创建 `missing` 索引项，不标记为 `stale`；删除章节通过 `markChapterDeleted` 删除索引和详情，并移除引用该章的 finding 摘要与详情。

用户从外部修改项目文件时，应用无法可靠获知变更。为此提供`检查正文变更`：

- 以默认 4 并发在后台读取正文并比较 hash，取消后停止调度新章节并保留已完成结果。
- 显示扫描进度和可取消状态。
- 更新 `contentHash`、`analysisInputHash` 和对应的 `missing` / `stale` / `ready` 状态，不调用 AI。
- 不在每次打开 Tab 时自动执行全书扫描。

扫描任务也必须排入对应章节的保存协调器队列：它在队列内读取正文、计算 hash 并更新索引。这样扫描与同章保存交错时，最后入队的保存结果必然成为最终索引状态。

### 7.3 更新 AI 分析：重操作，必须手动触发

`更新 AI 分析`打开通用 `Modal`，用户选择：

- 仅分析选中章节。
- 分析指定范围。
- 仅分析所有待处理章节（`missing`、`stale`、`failed`）。
- 分析全书。

默认选择“所有待处理章节”，并显示章数、预估调用次数和提示语：

```text
此操作将调用 AI 分析 12 个章节，可能耗时数分钟。
分析只更新章节脉络的派生结果，不会修改正文、细纲或伏笔正式记录。
```

任务执行时必须显示：当前章节、总数、成功数、失败数、取消按钮和继续入口。AI 队列默认单并发；取消后保留已完成结果，未开始项保持原状态，继续时重新收集 `missing`、`stale`、`failed` 项。全书分析是用户明确发起的后台批任务，不能由打开 Tab、保存正文或普通刷新隐式启动。

## 8. AI 分析设计

### 8.1 独立章节分析

为避免“修改第 10 章导致后 990 章必须重跑”的级联问题，基础 AI 分析不能依赖上一章完整快照。

每章独立分析只输入：

- 当前章节正文。
- 当前章节标题和完整章节引用。
- 可选的、受限长度的项目计划上下文，例如当前未回收伏笔的唯一 ID、名称和计划回收章节。

正文按统一规范化规则截取到模型输入上限；`analysisInputHash` 覆盖规范化正文 hash、标题、完整章节引用、实际传入的计划上下文、prompt 版本和模型版本。任何一个输入变化都会使旧分析变为 `stale`，但不会引发其他章节的级联重跑。

每章输出：

- 本章摘要。
- 关键事件。
- 可能推进或回收的伏笔名称。
- 支撑判断的原文摘录。
- 章节结尾钩子或待延续事项。

基础分析不直接修改项目中的伏笔记录。

### 8.2 跨章判断

跨章关联使用已保存的章节摘要和事件作为候选索引，不重新读取全书正文：

1. 对变更章节提取关键事件和候选伏笔名称。
2. 在分析索引中检索同一伏笔或相似事件的少量候选章节。
3. 只对候选对执行关联判断。
4. 生成带两端证据的 `ChapterFlowFinding`，将轻量摘要写入 `index.json`，将完整证据写入 `findings/<finding-id>.json`。

当某章变更时，只删除或重算“证据涉及该章”的发现。其他章节的分析结果保持可用。

### 8.3 结果边界

AI 可以说：

```text
第 68 章中出现“揭示身世”的明确叙述，可能对应伏笔“主角身世”的回收。
```

AI 不可以做：

- 自动把 `recordedResolutionChapter` 改为第 68 章。
- 自动把伏笔状态改为 `resolved`。
- 自动修改第 65 章或第 68 章正文。
- 在没有证据摘录时生成“因果关系”。

若 AI 与正式记录冲突，页面只显示“AI 与记录不一致”，并给出来源跳转。

## 9. 1000+ 章性能策略

### 9.1 允许与禁止的操作

| 操作 | 允许在打开 Tab 自动执行 | 是否调用 AI |
|---|---:|---:|
| 读取章节目录、标题、伏笔文件、分析索引 | 是 | 否 |
| 计算计划与记录差异 | 是 | 否 |
| 虚拟化渲染当前可见章节 | 是 | 否 |
| 读取 1000 篇正文 | 否 | 否 |
| 全书 hash 扫描 | 否，用户手动触发 | 否 |
| 分析待处理章节（未分析 / 过期 / 失败） | 否，用户手动触发 | 是 |
| 全书 AI 分析 | 否，用户手动触发 | 是 |

### 9.2 索引策略

- Tab 首次加载只读取一个只含摘要的分析索引文件，不枚举和解析全部详情文件。
- 保存章节时只增量更新一个章节索引项及受影响 finding 摘要，不重建全书索引。
- 选中章节时才加载 `analyses/<chapter-file-key>.json`；选中 AI 发现时才加载 `findings/<finding-id>.json`。
- 索引不保存原文摘录；缺失时从空索引开始，损坏时按第 5.5 节先备份原文件再重建，并提示用户执行正文检查或 AI 分析。
- 时间线和对照列表分别虚拟化或分页，不在 DOM 中创建上千个详情区。
- 章节标题、卷名和目录排序使用内存缓存；切换项目时失效。

### 9.3 任务策略

- AI 任务默认单并发，避免 API 限流和界面卡顿。
- 任务记录每章的输入 hash、状态、错误和完成时间；取消后不回滚已完成项，可中断后继续。
- 一个章节失败不阻塞其他章节；失败项在索引中保留原因。
- 全书任务按卷和范围分批，但由同一个任务队列管理。
- “待处理章节分析”选择 `missing`、`stale`、`failed` 项；其中 `stale` 的判定为 `analysisInputHash !== analyzedInputHash`。

## 12. 测试计划

### 12.1 计划校验测试

必须覆盖：

1. 多卷都有 `ch001` 时，章节和伏笔引用不冲突。
2. 计划第 65 章、记录第 65 章，状态为按计划。
3. 计划第 65 章、记录第 62 章，状态为提前。
4. 计划第 65 章、记录第 68 章，状态为延迟 3 章。
5. 已写到第 70 章、计划第 65 章但未记录回收，状态为逾期。
6. 尚未写到计划章节但未记录回收，状态为待执行。
7. 计划或记录章节不存在，状态为无效引用。
8. 没有计划回收章节时，不错误标记为逾期。
9. 卷名和自定义章节标题正确显示。
10. 已废弃伏笔保留显示，但不计入正常、逾期、提前或延迟统计。
11. 正式状态与记录冲突时为 `record-incomplete`，且优先于普通时间差状态。
12. 按卷筛选只影响展示，不改变全书位置和校验结论。
13. 章节文件已创建但正文为空时，仍计入“已写到”边界。
14. `vol2`、`vol10`、中文卷名混用时，全书顺序与 `buildChapterSequence` 一致，不受目录返回顺序影响。
15. 合法未来计划章节未创建时为 `pending`；该章节创建后未记录回收时转为 `overdue`。
16. 计划章节位于当前全书位置之前但章节文件不存在时为 `invalid-reference`，不伪装为 `pending`。
17. `advanced` 没有推进记录、推进早于埋设、非 `resolved` 状态却有记录回收时均为 `record-incomplete`。

### 12.2 性能测试

准备 1000+ 章目录、数百条伏笔和完整分析索引：

1. 打开 Tab 不得调用 `getChapterContent`。
2. 打开 Tab 不得调用 AI。
3. 初次渲染只创建可见范围内的章节卡片。
4. 切换卷、筛选状态和定位伏笔不触发全书正文扫描。
5. 手动扫描正文变更显示进度并可取消。
6. 手动 AI 分析只处理用户选择的范围或 `missing`、`stale`、`failed` 项。
7. 打开 Tab 只读取 `index.json`，不读取 `analyses/` 或 `findings/` 详情文件。
8. `index.json` 不包含原文证据摘录，选中章节或 finding 时才按需读取详情。
9. 新发现但未写入索引的目录章节以内存 `missing` 项显示，打开 Tab 仍不读取正文。

### 12.3 AI 边界测试

必须验证：

- 未点击“更新 AI 分析”时，不发生网络请求。
- 分析结果只写入 `memory/chapter-flow/`，不改正文、细纲和伏笔正式字段。
- AI 结果缺少证据时不生成可视化发现。
- 修改证据章节后，相关发现自动变为过期。
- 单章分析失败不会阻塞其他分析任务。
- 取消任务后，已完成结果保留，未完成项保持 `stale` 或 `missing`。
- 自动保存、卸载保存和手动保存都更新相同章节索引项，且不会触发 AI。
- 伏笔计划、标题、prompt 或模型变化后，`analysisInputHash` 变化并正确标记过期。
- 正文规范化、实体解码、换行和 Unicode NFC 差异不会让同一语义正文产生不同 `contentHash`。
- 输入上下文未变化时，刷新不会无故改变 `analysisInputHash`；实际发送的伏笔上下文、prompt 内容或模型变化时必定改变。
- finding 的伏笔 ID 不属于输入上下文或证据摘录无法在原文验证时，不写入索引和详情。
- Windows 安全文件 key 不包含 `:`，多卷同名章节不会写入同一个详情文件。

### 12.4 存储与恢复测试

必须验证：

- 正文保存、正文扫描和 AI 任务同时更新时，项目级串行写入不会丢失其他索引项。
- 自动保存、卸载保存、手动保存和同章扫描交错执行时，正文与索引最终都对应最后入队的正文。
- 索引写入失败不会回滚已经成功保存的正文，并提供可重试错误状态。
- `index.json` 缺失时从空索引开始；损坏时先备份原始内容再重建，绝不静默覆盖。
- 非空旧伏笔 schema、未来 schema 与损坏伏笔 JSON 不得被读取为“空伏笔”；确认重置前，原始文件必须保持不变。
- 确认重置时，伏笔源数据与旧灵感缓存都先完成内容校验备份；备份失败时不替换任何源文件。
- Rust 原子替换在目标已存在和替换失败时，分别验证只可见完整新文件和完整旧文件。
- finding 摘要缺少详情文件时显示数据损坏状态，不阻塞其他章节和 finding。
- 删除章节或伏笔后，悬空引用变为无效或对应派生 finding 被清理。

## 13. 验收标准

### 功能正确性

- 章节脉络能显示计划位置、已记录位置和差异状态。
- 多卷项目不存在章节 ID 覆盖、错连或错误排序。
- `pending` 只用于合法未来计划章节，不会因“计划章节必须已存在”而不可达。
- 已废弃、记录不完整和无效引用都按已定义的优先级显示和统计。
- 用户发现偏差后可直接跳转到对应正文和伏笔记录。
- 页面内没有手动编辑剧情关系的入口。

### AI 使用边界

- 打开 Tab、普通刷新、保存章节都不自动调用 AI。
- AI 仅在用户从分析弹窗明确启动后调用。
- AI 结果带证据、置信度和过期状态。
- AI 不自动写回正式伏笔记录，也不修改章节内容。
- `index.json` 只保存 finding 摘要，完整证据只存在对应的 `findings` 详情文件中。

### 性能与可用性

- 1000+ 章项目的普通打开不逐章读取正文。
- 1000+ 章项目的普通打开不调用 AI。
- 普通打开只读取 `index.json`，不枚举或解析分析与 finding 详情文件。
- 普通打开不会将损坏伏笔源数据静默显示为空，也不会删除或覆盖旧伏笔数据。
- 章节时间线采用虚拟滚动，长列表稳定可用。
- 全书扫描和全书分析都有进度、取消和继续能力。
- 页面遵守现有 flex 高度传递和滚动容器约定。

### 工程门禁

- 修改 `src/**/*.ts` 或 `src/**/*.tsx` 后运行 `npm run lint:risk`。
- 涉及构建行为时运行 `npm run check`。
- 新增分页使用 `Pagination`，新增弹窗使用 `Modal`。
- 所有改动路径运行 `git diff --check -- <paths>`。

## 14. 首个可交付版本

第一个版本交付 Phase 0、Phase 1 和 Phase 2 中不含 AI 调用的能力；不得在只完成 Phase 1 时宣称具备分析过期状态：

1. 全书稳定章节时间线。
2. 伏笔的计划回收、已记录推进、已记录回收对照。
3. 按计划、提前、延迟、逾期、记录不完整、无效引用、已废弃等状态。
4. 章节和伏笔跳转入口。
5. 轻量刷新视图。
6. 三种正文保存路径统一更新 hash；已分析项标记 `stale`，未分析项保持 `missing`。
7. 1000+ 章虚拟滚动和列表分页。
8. 严格 schema 检测、确认后备份重置、原子索引写入和损坏恢复。
9. 新建/删除章节与三种正文保存路径的统一索引更新。

AI 分析、证据展示和跨章候选关联作为第二个可交付版本。这样即使用户从未配置 AI，章节脉络也能立即作为可靠的写作计划校验工具使用。
