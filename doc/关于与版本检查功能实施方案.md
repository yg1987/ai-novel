# 关于与版本检查功能实施方案

> 创建日期：2026-07-22
> 状态：已完成并验收（2026-07-22）

---

## 一、背景

当前应用缺少面向用户的项目名称、版本、开发者、版权和许可证信息，也没有方便用户查看项目主页、更新日志与新版本状态的入口。

本方案参考 CC Switch 的“关于”页交互，但结合 AI Novel Writer 当前阶段做精简：应用只负责检查 GitHub Releases 是否存在新版本，不在应用内下载或安装更新。用户看到新版本提示后，自行决定是否打开发布页并下载安装包。

## 三、目标与边界

### 3.1 本期目标

1. 在系统设置中增加“关于”页。
2. 展示项目名称、当前版本、开发者、版权和 MIT 许可证。
3. 提供 GitHub 仓库和更新日志的外部链接。
4. 提供“检查更新”按钮，通过 GitHub Releases 查询最新正式版本。
5. 检测到新版本时展示版本号、更新说明和“前往下载”按钮。
6. 由用户自行前往 GitHub Releases 下载、退出应用并安装新版。
7. 补齐仓库与安装包中的许可证、开发者和版权元数据。

### 3.2 明确不做

- 不在应用内下载更新包。
- 不静默安装或覆盖当前版本。
- 不自动重启应用。
- 不接入 Tauri updater 插件。
- 不生成或管理 Tauri updater 更新包签名密钥。
- 不购买或接入 Windows Authenticode 代码签名证书。
- 不承诺消除 Windows 的“未知发布者”或 SmartScreen 提示。
- 不在应用启动时强制检查更新；本期仅由用户主动点击检查。

```text
系统设置

关于

┌────────────────────────────────────────────────────┐
│  AI Novel Writer                         GitHub     │

│                                          检查更新   │

```

设计要求：

- “通用”继续承载当前可编辑设置，“关于”只展示不可编辑的产品信息。
- 关于页结构参考 CC Switch，但不添加当前项目不存在的“官网”等入口。
- 外部链接按钮使用图标加文字，并通过系统默认浏览器打开。
- 复用已初始化的 Tauri Shell 插件：前端增加 `@tauri-apps/plugin-shell` 依赖并调用 `open(url)`；不新增 Opener 插件或新的 Rust 命令。现有 Rust 端已注册 Shell 插件、默认 capability 已授予 `shell:allow-open`，本期只补充 Tauri 配置中的 URL 白名单。
- “检查更新”按钮在请求期间显示加载状态并禁用，避免重复请求。
- 当前 `SettingsModal` 使用了独立的 `dialog-overlay` 遮罩；本期必须改为复用现有 `src/components/Modal.tsx`（含关闭按钮、点击遮罩和 Escape 关闭），不得保留或新增独立遮罩层。
- 界面中不显示代码签名、自动更新等实现说明。

## 五、版本检查流程

### 5.1 数据来源

使用 GitHub Releases API 查询最新正式版本：

```text
GET https://api.github.com/repos/yg1987/ai-novel/releases/latest
```

使用返回数据中的以下字段：

| GitHub 字段 | 用途 |
|-------------|------|
| `tag_name` | 最新版本号，例如 `v0.2.0` |
| `name` | 发布名称 |
| `body` | Markdown 更新说明 |
| `html_url` | “前往下载”链接 |
| `published_at` | 发布时间 |

本期不直接解析或下载 `assets` 中的安装包。

### 5.2 判断规则

1. 读取应用当前版本，例如 `0.1.0`。
   前端通过 `@tauri-apps/api/app` 的版本读取 API 获取该值，不在界面组件或服务中重复硬编码。
2. 请求 GitHub 最新正式 Release。
3. 去掉 `tag_name` 可选的 `v` 前缀。
4. 使用经验证的语义化版本实现比较版本，禁止使用普通字符串比较；依赖及其 TypeScript 类型随 `package.json` 和锁文件一并维护。
5. 预发布版本默认不参与检查，GitHub `/releases/latest` 只返回最新正式 Release。

### 5.3 状态与反馈

| 状态 | 用户反馈 |
|------|----------|
| 检查中 | 按钮显示“检查中”，同时禁用 |
| 已是最新版 | 显示“当前已是最新版本” |
| 发现新版本 | 展示新版本号、发布时间、更新说明和“前往下载”按钮 |
| 尚未发布 Release | 提示“暂未找到可用版本” |
| 网络失败或 GitHub 不可访问 | 显示可理解的失败信息，并保留更新日志链接 |
| API 限流 | 提示稍后重试，不影响应用其他功能 |

请求使用 `fetch` 和 `AbortController`，超时时间固定为 10 秒。必须按 HTTP 状态和经校验的响应字段处理：`404` 映射为“暂未找到可用版本”，`403` / `429` 映射为“请求受限，请稍后重试”，网络错误、超时、非 JSON 或缺少必填字段均映射为可理解的失败信息。Release 的 `body` 以 React 纯文本方式保留换行展示；不得直接将远端 Markdown 注入 HTML。

“前往下载”只打开 `html_url`。应用不推断用户需要 MSI、NSIS 或其他安装包，也不代替用户做下载选择。

## 十、后续可选项

以下内容不属于本方案当前范围，未来根据发布规模单独立项：

1. 应用启动后的低频后台更新提示。
2. 忽略指定版本或稍后提醒。
3. GitHub API 缓存与访问频率控制。
4. Windows Authenticode 代码签名证书，用于改善“未知发布者”和 SmartScreen 体验。
5. Tauri updater 的下载、更新包完整性签名、安装与重启流程。

其中第 4 项解决 Windows 对发布者身份的信任问题；第 5 项解决应用内自动更新及更新包完整性验证。二者用途不同，也都不是“只检查版本并跳转下载页”所必需的。
