# 前端拆分与构建体积优化分步方案

> 创建日期：2026-07-16  
> 状态：已完成  
> 整合来源：已删除的旧 Vite 构建体积提示文档  
> 范围说明：本文档一次性列出本轮必须处理的完整拆分范围；Phase 仅表示实施顺序，不表示非必做项。

## 一、背景

当前 `npm run build` 能正常通过，但构建产物暴露了明显的结构性问题：

1. `src/services/reviewService.ts` 同时被动态导入和静态导入，导致动态导入无法拆成独立 chunk。
2. 主 JS chunk 超过 500 kB，当前 `dist/assets/index-*.js` 约 1.5 MB。
3. `dist/assets` 里基本只有一个主 JS 和一个 CSS 文件，说明前端仍接近单入口静态依赖图。

这不是单个 warning 的问题，而是项目还没有建立稳定的按需加载边界。随着 Tab、图谱、统计、AI 审查、素材、灵感等功能继续增加，主包会持续膨胀。本轮处理目标是一次性把需要处理的拆分范围定义完整，再按阶段实施和验收。

## 二、当前结构问题

### 2.1 入口层静态聚合

相关文件：

- `src/App.tsx`
- `src/components/ProjectView.tsx`
- `src/components/ProviderConfig.tsx`
- `src/components/SettingsModal.tsx`

问题：

- `App.tsx` 静态引入 `ProjectView`、AI 配置面板、系统设置弹窗。
- 用户停留在书架页时，项目页和弹窗代码也会进入入口依赖图。

### 2.2 项目页 Tab 全部静态引入

相关文件：

- `src/components/ProjectView.tsx`

当前静态引入的 Tab 包括：

- 写作：`ChapterManager`
- 角色：`CharacterPanel`
- 世界观：`WorldviewPanel`
- 大纲：`OutlinePanel`
- 备注：`NotesPanel`
- 伏笔：`ForeshadowPanel`
- 搜索：`SearchPanel`
- 统计：`StatisticsPanel`
- 审查：`ReviewPanel`
- 素材：`ResourcePanel`
- 灵感：`BrainstormPanel`
- 关系图：`RelationshipGraph`
- 热门：`TrendingPanel`
- 章节图：`ChapterGraph`

问题：

- 即使用户只打开默认写作 Tab，其他 Tab 的组件代码和依赖也会进入主依赖图。
- 这是主 chunk 过大的核心原因。

### 2.3 重依赖缺少按需边界

重依赖分布：

- `@tiptap/*`
  - `src/components/Editor.tsx`
  - `src/components/VersionHistoryPanel.tsx`
- `recharts`
  - `src/components/statistics/WordCountSection.tsx`
  - `src/components/statistics/AIUsageSection.tsx`
  - `src/components/statistics/ProjectScaleSection.tsx`
- `@react-sigma/core` / `sigma` / `graphology` / `graphology-layout-forceatlas2`
  - `src/components/RelationshipGraph.tsx`
  - `src/services/relationshipStore.ts`
  - `src/lib/graph-insights.ts`
- `d3-force`
  - `src/components/ChapterGraph.tsx`

问题：

- 这些依赖都是功能页专用，但当前被上层静态 import 间接拉进主包。
- `@react-sigma/core/lib/style.css` 还在 `src/main.tsx` 全局引入，关系图样式没有跟关系图模块绑定。

### 2.4 `reviewService` 职责混合导致动态导入失效

相关文件：

- `src/services/reviewService.ts`
- `src/services/savePipeline.ts`
- `src/components/ReviewPanel.tsx`
- `src/services/statsService.ts`

当前问题：

- `reviewService.ts` 同时包含轻量检查、深度 AI 审查、报告列表、报告读取、章节审查聚合。
- `savePipeline.ts` 顶部静态导入 `runAndSaveLightCheck`。
- `savePipeline.ts` 内部又动态导入同一个 `reviewService.ts` 以加载 `runDeepReview`。
- `ReviewPanel.tsx` 静态导入 `runAndSaveLightCheck`、`runDeepReview`、`loadChapterReviews`。
- `statsService.ts` 静态导入 `loadChapterReviews`。

结果：

- `reviewService.ts` 已经被静态依赖图捕获，动态导入 `runDeepReview` 不会产生独立 chunk。

### 2.5 面板内部也存在二级按需点

相关文件：

- `src/components/ChapterManager.tsx`
- `src/components/ReviewPanel.tsx`
- `src/components/statistics/StatisticsPanel.tsx`
- `src/components/RelationshipGraph.tsx`

问题：

- `ChapterManager` 静态引入 `VersionHistoryPanel`、`MaterialSidebar`，但它们只在用户打开版本历史或素材侧栏时需要。
- `ReviewPanel` 静态引入 `ReviewRulesEditor`，但规则编辑器只在点击配置时需要。
- `StatisticsPanel` 静态引入所有统计 section，必须按 section 懒加载。
- `RelationshipGraph` 同时包含 Sigma 图、文档视图、mindmap、重建弹窗等多个子视图，必须拆成关系图子模块。

### 2.6 样式仍是全局单包

相关文件：

- `src/main.tsx`
- `src/style.css`

问题：

- `src/style.css` 约 3500 行，所有功能样式全局加载。
- 当前 CSS 不是最大体积来源，但长期维护上也缺少模块边界。

## 三、优化原则

1. 先建立按需加载边界，再做文件级重构。
2. 先拆影响构建体积的依赖图，再拆纯维护性大文件。
3. 不通过调高 `chunkSizeWarningLimit` 掩盖问题。
4. 不做无证据 fallback：每一阶段必须构建验证。
5. 不新增 barrel 文件重新导出重模块，避免把异步边界重新拉平。

## 四、分步实施计划

### Phase 0：建立基线

目标：

- 固化当前构建状态，作为所有 Phase 的对比基线。

操作：

1. 执行 `npm run build`。
2. 记录 `dist/assets` 文件列表和大小。
3. 记录当前 warning。

验收：

- 有清晰的前后对比基线。
- 确认当前主 JS 约 1.5 MB。

涉及文件：

- 无代码改动。

### Phase 1：入口和 Tab 级懒加载

目标：

- 先解决主包过大的最大来源。

改动范围：

- `src/App.tsx`
- `src/components/ProjectView.tsx`

实施内容：

1. 在 `App.tsx` 中懒加载：
   - `ProjectView`
   - `ProviderConfigPanel`
   - `SettingsModal`
2. 在 `ProjectView.tsx` 中用 `React.lazy` / `Suspense` 懒加载所有 Tab 面板：
   - `ChapterManager`
   - `CharacterPanel`
   - `WorldviewPanel`
   - `OutlinePanel`
   - `NotesPanel`
   - `ForeshadowPanel`
   - `SearchPanel`
   - `StatisticsPanel`
   - `ReviewPanel`
   - `ResourcePanel`
   - `BrainstormPanel`
   - `RelationshipGraph`
   - `TrendingPanel`
   - `ChapterGraph`
3. 保留当前状态传递和导航回调：
   - `handleNavigateToReview`
   - `handleNavigateToChapter`
   - `handleNavigateToCharacter`
   - `handleNavigateToForeshadow`
   - `handleNavigateToNotes`
   - `handleSearchOpenFile`

注意事项：

- `Suspense` fallback 必须放在 `.project-tab-content` 内，避免破坏布局高度链。
- fallback 容器应使用现有空状态样式或新增轻量 loading 样式，不要引入新的大组件。
- 不要先只懒加载部分 Tab，否则容易留下隐性主包来源。

验收：

- `npm run build` 成功。
- `dist/assets` 出现多个异步 JS chunk。
- 主入口 `index-*.js` 明显小于当前 1.5 MB。
- 书架页、进入项目、切换所有 Tab 正常。

### Phase 2：拆分 `reviewService`

目标：

- 消除无效动态导入 warning。
- 让深度 AI 审查真正按需加载。

新建文件：

- `src/services/reviewLightService.ts`
- `src/services/reviewDeepService.ts`
- `src/services/reviewReportStorage.ts`

职责划分：

`reviewLightService.ts`：

- `runAndSaveLightCheck`

`reviewDeepService.ts`：

- `runDeepReview`
- 深度审查专用 prompt、JSON 提取、AI 调用逻辑

`reviewReportStorage.ts`：

- `listReviewReports`
- `getReviewReport`
- `loadChapterReviews`
- `ChapterReviewData`

引用调整：

- `src/services/savePipeline.ts`
  - 静态导入 `reviewLightService`
  - 动态导入 `reviewDeepService`
- `src/components/ReviewPanel.tsx`
  - 静态导入 `reviewLightService`
  - 静态导入 `reviewReportStorage`
  - 点击深度审查时动态导入 `reviewDeepService`
- `src/services/statsService.ts`
  - 只导入 `reviewReportStorage`

禁止事项：

- 不要保留一个 `reviewService.ts` 统一 re-export 三个新模块。
- 不要让 `statsService.ts` 间接导入深度审查模块。

验收：

- `npm run build` 成功。
- `INEFFECTIVE_DYNAMIC_IMPORT` warning 消失。
- 保存章节后的轻量检查正常。
- 审查面板读取历史报告正常。
- 手动 AI 深度审查正常。
- 统计页的审查摘要正常。

### Phase 3：Vite 手工 vendor 分包

目标：

- 让第三方重依赖形成稳定 chunk。
- 避免重依赖混入业务 chunk。

改动范围：

- `vite.config.ts`

分包内容：

- `vendor-react`
  - `react`
  - `react-dom`
- `vendor-tauri`
  - `@tauri-apps/api`
  - `@tauri-apps/plugin-dialog`
  - `@tauri-apps/plugin-fs`
- `editor-tiptap`
  - `@tiptap/core`
  - `@tiptap/react`
  - `@tiptap/starter-kit`
  - `@tiptap/extension-placeholder`
  - `@tiptap/extension-underline`
- `charts-recharts`
  - `recharts`
- `graph-sigma`
  - `@react-sigma/core`
  - `sigma`
  - `graphology`
  - `graphology-layout-forceatlas2`
  - `graphology-communities-louvain`
- `graph-d3`
  - `d3-force`

注意事项：

- 不要把所有 `node_modules` 简单打成一个 `vendor`，否则仍会形成过大的公共包。
- 不要仅通过 `chunkSizeWarningLimit` 消除 warning。

验收：

- `npm run build` 成功。
- `dist/assets` 中出现稳定命名或可识别的 vendor chunk。
- 主入口 chunk 不再包含统计、图谱、章节图等重依赖。

### Phase 4：移动关系图 CSS

目标：

- 让关系图样式跟随关系图模块按需加载。

改动范围：

- `src/main.tsx`
- `src/components/RelationshipGraph.tsx`

实施内容：

1. 从 `src/main.tsx` 移除：

```ts
import '@react-sigma/core/lib/style.css'
```

2. 在 `src/components/RelationshipGraph.tsx` 或关系图专用 wrapper 中引入：

```ts
import '@react-sigma/core/lib/style.css'
```

验收：

- `npm run build` 成功。
- 关系图 Tab 打开后 Sigma 样式正常。
- 书架页和非关系图 Tab 不再需要加载 Sigma CSS。

### Phase 5：面板内部二级懒加载

目标：

- 继续减少功能页内的默认加载成本。

改动范围：

- `src/components/ChapterManager.tsx`
- `src/components/ReviewPanel.tsx`
- `src/components/statistics/StatisticsPanel.tsx`

实施内容：

1. `ChapterManager`
   - 懒加载 `VersionHistoryPanel`
   - 懒加载 `MaterialSidebar`
2. `ReviewPanel`
   - 懒加载 `ReviewRulesEditor`
3. `StatisticsPanel`
   - 按 section 懒加载：
     - `WordCountSection`
     - `AIUsageSection`
     - `WritingHabitsSection`
     - `ProjectScaleSection`
     - `HealthSection`

验收：

- 写作 Tab 默认打开正常。
- 点击版本历史后再加载版本历史面板。
- 点击素材侧栏后再加载素材面板。
- 打开审查规则配置正常。
- 切换统计 section 正常。

### Phase 6：关系图内部拆分

目标：

- 降低关系图业务模块复杂度。
- 为关系图维护、测试和图谱功能扩展建立明确边界。

拆分文件：

- `src/components/relationship-graph/RelationshipGraph.tsx`
- `src/components/relationship-graph/ForceGraphView.tsx`
- `src/components/relationship-graph/DocumentGraphView.tsx`
- `src/components/relationship-graph/MindMapGraphView.tsx`
- `src/components/relationship-graph/SnapshotRebuildDialog.tsx`
- `src/components/relationship-graph/GraphToolbar.tsx`
- `src/components/relationship-graph/GraphLegend.tsx`
- `src/components/relationship-graph/NodeDetail.tsx`
- `src/components/relationship-graph/InsightsPanel.tsx`

验收：

- 关系图三种展示模式正常。
- 图谱筛选、聚焦、隐藏节点、重建快照正常。
- 导出/复制图谱文档正常。

### Phase 7：大面板维护性拆分

目标：

- 改善长期维护性，不把所有业务状态、表单、列表、弹窗塞进一个文件。

处理文件：

- `src/components/ForeshadowPanel.tsx`
- `src/components/WorldviewPanel.tsx`
- `src/components/OutlinePanel.tsx`
- `src/components/CharacterPanel.tsx`
- `src/components/NotesPanel.tsx`

实施内容：

- 容器组件负责数据加载和状态协调。
- sidebar/list/editor/dialog 拆成子组件。
- 可复用业务逻辑沉到 hooks 或 services。
- 弹窗继续使用通用 `Modal`。
- 分页继续使用通用 `Pagination`。

验收：

- 功能行为不变。
- 单文件复杂度下降。
- 不引入新的全局静态聚合点。

### Phase 8：样式模块化整理

目标：

- 降低 `src/style.css` 长期膨胀风险。
- 让功能样式跟组件边界一致。

实施内容：

- 保留全局变量、基础布局、通用组件样式在 `style.css`。
- 按功能拆出：
  - 写作
  - 角色
  - 世界观
  - 大纲
  - 备注
  - 伏笔
  - 统计
  - 审查
  - 素材
  - 关系图
- CSS 拆分要晚于 JS 懒加载，避免先制造大量样式入口。

验收：

- 所有 Tab 样式正常。
- 布局高度链不被破坏。
- 不出现样式丢失、覆盖顺序异常。

## 五、分步实施顺序

实施顺序：

1. Phase 0：建立构建基线。
2. Phase 1：入口和 Tab 级懒加载。
3. Phase 2：拆分 `reviewService`。
4. Phase 3：Vite 手工 vendor 分包。
5. Phase 4：移动关系图 CSS。
6. Phase 5：面板内部二级懒加载。
7. Phase 6：关系图内部拆分。
8. Phase 7：大面板维护性拆分。
9. Phase 8：样式模块化整理。

阶段边界：

- Phase 0 到 Phase 4 解决当前构建体积、无效动态导入和重依赖入主包问题。
- Phase 5 到 Phase 8 继续完成面板内部、关系图、大面板和样式拆分。
- 所有 Phase 都属于本轮方案范围，不能把 Phase 5 到 Phase 8 当作非必做项。
- 实施时按 Phase 分批提交和验收，但完整收口标准是 Phase 0 到 Phase 8 全部完成。

## 六、每阶段通用检查清单

每完成一个 Phase，至少执行：

```powershell
npm run build
Get-ChildItem dist/assets | Sort-Object Length -Descending | Select-Object Name,Length
```

功能回归：

- 书架页加载。
- 进入项目。
- 所有 Tab 至少打开一次。
- 写作页保存章节。
- 审查页读取报告、运行轻量检查、运行深度审查。
- 统计页切换各 section。
- 关系图打开、切换展示模式。
- 章节图打开。
- AI 配置、系统设置、导出、存档弹窗打开。

构建回归：

- 不出现 `INEFFECTIVE_DYNAMIC_IMPORT`。
- 主入口 chunk 明显下降。
- 重依赖形成独立 chunk。
- 不通过调高 `chunkSizeWarningLimit` 掩盖问题。

## 七、完成标准

完成 Phase 0 到 Phase 8 后必须达到：

- 首屏入口 chunk 不再包含全部 Tab。
- 统计、关系图、章节图、审查、素材、灵感等功能按需加载。
- `reviewService` 动态导入 warning 消失。
- `recharts`、`sigma/graphology`、`d3-force`、`@tiptap/*` 被拆入独立 chunk。
- `ChapterManager` 内的版本历史、素材侧栏按需加载。
- `ReviewPanel` 内的规则编辑器按需加载。
- `StatisticsPanel` 内的各 section 按需加载。
- `RelationshipGraph` 被拆成独立子模块。
- `ForeshadowPanel`、`WorldviewPanel`、`OutlinePanel`、`CharacterPanel`、`NotesPanel` 完成维护性拆分。
- `style.css` 完成模块化整理，功能样式与组件边界一致。
- 新增功能有明确的懒加载和模块边界可遵循。
