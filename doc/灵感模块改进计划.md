## 一、改造背景

当前灵感模块已经具备基础生成能力：

1. 提供情节走向、场景创意、角色发展、世界观扩展四种模式。
2. 自动读取项目名称、类型、简介。
3. 读取最近 3 章的部分正文、角色认知状态和未回收伏笔。
4. 调用当前 AI Provider 的分析模型生成建议。
5. 在右侧以卡片形式展示结果，并提供加载和错误状态。

现状适合作为 MVP，但工作流只覆盖了：

```text
选择模式 -> 生成 -> 阅读结果
```

作者真正需要的完整流程是：

```text
说明卡点 -> 选择范围与约束 -> 生成多个方向 -> 比较筛选
        -> 深挖 / 变体 / 组合 -> 收藏或落地到创作流程
```

当前主要差距：

1. **不知道用户具体卡点**：四种模式过于宽泛，用户无法说明目标、限制和不希望出现的方向。
2. **上下文没有按模式区分**：世界观扩展没有读取世界观文档，其他模式也缺少各自最关键的数据。
3. **输出不可稳定解析**：依赖纯文本编号切分，无法可靠支持后续的逐条操作。
4. **结果无法继续推演**：不能围绕某条建议深化、生成变体或组合两个方向。
5. **结果没有落地点**：不能收藏、保存为备注、加入大纲草稿或转成伏笔候选。
6. **没有历史记录**：切换 Tab 后结果丢失，无法形成项目长期积累。
7. **缺少高级控制**：不符合项目“Preset + Customize”的两层设计原则。

## 二、产品定位与边界

### 2.1 核心定位

灵感模块负责帮助作者在已有项目事实和当前写作目标之间生成候选方案，并支持比较、深化和保存。

它需要回答四个问题：

- 我现在具体卡在哪里？
- 哪些方向符合已有设定？
- 每个方向的收益、代价和风险是什么？
- 选中的方向下一步放到哪里继续处理？

### 2.2 与其他模块的边界

| 模块 | 保存的内容 | 灵感模块与其关系 |
|---|---|---|
| 写作 | 已确认的章节正文 | 灵感不直接覆盖正文；“加入写作上下文”需要先建立通用上下文选择模型 |
| 大纲 | 已确认的剧情规划 | 现有模块没有草稿实体；“大纲草稿”需要单独确认数据模型后实施 |
| 备注 | 临时想法、问题、待办 | 灵感可保存为备注，通过可选来源字段保留会话和建议 ID |
| 伏笔 | 受管理的伏笔实体 | 灵感可预填伏笔创建表单，不能静默创建伏笔 |
| 世界观 | 已确认的项目事实 | 灵感只读取，不能自动改写世界观设定 |
| 角色 | 已确认的角色资料 | 灵感只读取，不能自动覆盖角色档案 |

### 2.3 明确不做

- 不直接把 AI 结果写入章节正文。
- 不让 AI 静默修改大纲、角色、世界观或伏笔数据。
- 不做跨项目共享的灵感库；灵感历史只属于当前项目。
- 不在第一阶段引入向量检索，先使用明确、可解释的项目数据源。
- 不展示模型的隐藏推理过程，只展示可供作者判断的依据、风险和建议。
- 不把“不断增加模式数量”作为主要扩展方式，优先完善通用输入、推演和落地闭环。

## 三、设计原则

### 3.1 Preset：新手默认可用

新手进入灵感 Tab 后，不填写任何高级设置也能完成生成：

- 默认模式：情节走向。
- 默认问题：允许留空，由 AI 根据当前项目状态主动发现可发展方向。
- 默认范围：当前章节；没有当前章节时使用最近章节。
- 默认方案数：4 条。
- 默认创意尺度：平衡。
- 默认自动选择与当前模式相关的上下文。

### 3.2 Customize：熟练用户可控

高级设置允许用户调整：

- 分析范围：当前章节、当前卷、全书、指定章节。
- 相关角色：多选项目角色。
- 创意尺度：稳妥、平衡、大胆。
- 期望氛围或节奏。
- 必须保留的条件。
- 明确禁止的方向。
- 生成数量：3–6 条。
- 本次使用的上下文来源。

设置分为两类：

- **项目偏好**：模式、创意尺度、结果数量、启用的上下文来源。保存到当前项目，下次进入灵感 Tab 时恢复。
- **会话输入**：具体问题、范围、相关角色、期望氛围、必须保留和禁止方向。只属于当前会话，不自动带入下一次全新生成。

“一键恢复默认设置”只重置项目偏好，不删除当前问题、历史会话或任何项目资料。已确认只记忆非内容型项目偏好，具体范围见第十五节。

### 3.3 作者始终拥有最终决定权

- AI 输出统一标记为候选建议。
- 所有跨模块落地操作都需要用户主动点击。
- 写入大纲、备注或伏笔前展示预填内容，允许编辑和取消。
- 每次成功生成或推演都创建新会话；新会话成为当前会话，旧会话保留在项目存储中。
- 切换 Tab 或重启应用不能导致已经成功生成的会话丢失。
- 未成功完成或未通过结构校验的响应不保存为会话。

## 四、目标交互流程

### 4.1 首次进入

```text
┌──────────────────────┬──────────────────────────────────────────────┐
│ 灵感模式             │ 当前灵感                                    │
│                      │                                              │
│ ● 情节走向           │ 还没有生成内容                              │
│ ○ 场景创意           │                                              │
│ ○ 角色发展           │ [描述当前卡点，可留空]                      │
│ ○ 世界观扩展         │                                              │
│                      │ [生成灵感]                                  │
│ 本次范围             │                                              │
│ 当前章节             │                                              │
│                      │                                              │
│ [高级设置]           │                                              │
└──────────────────────┴──────────────────────────────────────────────┘
```

左侧负责模式和范围，右侧负责问题输入、生成状态和结果。保留现有左右分栏，不增加嵌套卡片式页面结构。

### 4.2 生成前

右侧顶部提供一个可选问题输入区：

```text
我现在遇到的问题
┌──────────────────────────────────────────────────────────┐
│ 主角刚结束大战，下一章不想继续战斗，但需要推进身份伏笔。 │
└──────────────────────────────────────────────────────────┘

必须保留：女主暂时不知道主角身份
避免方向：新增重要角色、再次遭遇追杀
```

问题输入为空时仍可生成，避免增加新手使用门槛。

生成按钮附近只展示必要状态，不用长篇帮助文案：

- 已选择的范围。
- 将使用的上下文来源数量。
- Provider 或分析模型缺失时的明确错误。

### 4.3 结果展示

每条结果采用统一结构：

```text
2  宴会上的无声试探                         大胆

核心方向
让身份伏笔通过礼仪和称谓错误推进，不再安排战斗。

展开方式
1. 主角参加战后宴会
2. 旧臣使用只有皇族才知道的称谓
3. 女主注意到异常，但得到一个合理的表面解释

为什么适合
承接大战后的节奏回落，同时推进未回收的身份伏笔。

风险
需要确认旧臣此前是否已经出场；解释过强会削弱悬念。

[深化] [生成变体] [收藏] [更多]
```

结果必须让作者能够比较，而不仅是阅读几段风格相近的文本。每条建议至少包含：

- 标题。
- 核心方向。
- 具体展开步骤。
- 推荐位置。
- 与现有项目的连接依据。
- 风险或潜在冲突。
- 可利用的角色、设定或伏笔。

### 4.4 继续推演

单条建议支持：

- **深化**：把方向展开成更完整的情节链或场景方案。
- **生成变体**：保留核心目标，改变人物、地点、节奏或揭示方式。
- **调整尺度**：生成更稳妥或更大胆的版本。
- **组合**：选择两条建议，生成兼容后的新方案。
- **基于反馈重做**：用户填写“不满意的原因”，只重做当前建议。

继续推演生成新的派生结果，保留来源关系，不覆盖原建议。

### 4.5 保存与落地

每条结果的“更多”菜单提供：

- 复制全文。
- 保存为备注。
- 创建伏笔候选。
- 标记为不采用。

以下两项本轮不实施，保留在 Phase 2B 作为后续候选范围：

- 添加到大纲草稿。
- 加入当前章节的 AI 写作上下文。

其中：

- “保存为备注”打开使用通用 `<Modal>` 的预填确认弹窗；保存后写入可选的灵感来源字段。
- “创建伏笔候选”打开现有伏笔创建表单并预填字段。
- “添加到大纲草稿”不得复用正式大纲 Markdown 文件冒充草稿，必须使用经过确认的草稿模型。
- “加入写作上下文”不得伪装成 `MaterialContextSelection`；必须先建立通用写作上下文联合类型。

### 4.6 历史视图

灵感 Tab 内部提供“当前 / 历史”两个视图：

- 当前：本次问题、设置和生成结果。
- 历史：按时间倒序展示历史会话，可按模式、收藏状态筛选。

历史列表使用通用 `<Pagination>`，不得手写分页按钮。每条历史记录展示：

- 生成时间。
- 模式。
- 用户问题摘要。
- 结果数量。
- 收藏数量。
- 使用的章节范围。

打开历史会话时，AI 生成的内容字段只读；收藏和“不采用”属于可更新的用户元数据。用户点击“基于本次继续”后创建新会话，避免改写原始内容。

当前会话和历史会话使用同一套会话存储。Phase 1 在生成成功后立即落盘，以保证切换 Tab 不丢失；Phase 3 只补历史浏览、筛选、收藏、删除和分页 UI，不再延后底层持久化。

## 五、模式与上下文策略

四种模式不能继续共享完全相同的上下文。应新增专用的 `buildBrainstormContext`，根据模式、范围和 token 预算装配数据。

### 5.1 通用上下文

所有模式默认包含：

- 项目名称、类型、简介。
- 用户本次填写的问题、保留条件和禁止方向。
- 当前章节或用户选择的范围。
- 与范围相关的章节标题和顺序。

### 5.2 情节走向

优先读取：

1. 当前章节结尾，而不是只读取开头 200 字。
2. 当前卷大纲和后续章节大纲。
3. 最近章节摘要或快照。
4. 未回收伏笔。
5. 当前主要角色状态。

目标是判断“接下来发生什么”，重点保持因果、悬念和节奏连续。

### 5.3 场景创意

优先读取：

1. 当前章节目标和大纲。
2. 当前出场角色及关系。
3. 可用地点和世界观规则。
4. 上一场景结尾。
5. 用户指定的氛围、节奏和禁止条件。

目标是给出可以实际写进章节的桥段，而不是泛泛的剧情方向。

### 5.4 角色发展

优先读取：

1. 用户选择的角色档案。
2. 角色当前认知状态。
3. 角色关系和近期互动。
4. 与角色相关的未回收伏笔。
5. 角色最近出场章节和当前卷大纲。

未指定角色时，自动选择当前章节的主要出场角色；无法判断时要求用户选择，不能随意虚构角色状态。

### 5.5 世界观扩展

优先读取：

1. `worldview/` 下已有设定文档。
2. 当前卷大纲和剧情需要。
3. 已有组织、地点、能力体系和关键物品。
4. 与扩展方向相关的角色和伏笔。

从现有 `buildAIContext()` 提取可复用的世界观读取基础能力，但不能直接照搬“目录前 5 个文件”的当前策略。新读取器必须：

- 只读取世界观内容文件，排除配置 JSON 和其他非内容文件。
- 优先按照世界观 section 配置顺序读取。
- 记录实际读取的文件和被截断状态。
- 在 token 预算内按相关性和配置顺序取舍，而不是依赖文件系统返回顺序。

### 5.6 上下文可见性

生成前展示简洁的来源摘要，例如：

```text
本次上下文：当前章节、后续大纲、3 名角色、4 条未回收伏笔
```

高级设置允许取消某类来源。读取失败时必须明确提示“部分上下文未加载”，不能静默降级后仍声称结果基于完整项目数据。

### 5.7 上下文预算与截断规则

上下文预算属于 Phase 1 的生成正确性要求，不能推迟到体验收尾阶段。

- 默认输入预算：估算不超过 6000 tokens；输出预算：最多 2048 tokens。
- 用户问题、必须保留和禁止方向不得被静默截断。
- 当前章节目标、结尾和当前范围大纲属于第一优先级。
- 模式主数据属于第二优先级，例如角色模式的角色档案、世界观模式的设定文档。
- 伏笔、关系、较早章节摘要等辅助数据属于第三优先级。
- 全书范围只读取章节摘要、快照和必要片段，不把全部章节正文直接拼入请求。
- 超出预算时从最低优先级来源开始截断，并在 `contextManifest` 中记录。
- 任一单文件都设置上限，避免一个超长角色或世界观文件占满全部预算。

用户输入限制：

| 字段 | 限制 |
|---|---|
| 当前问题 | 最多 1000 字符 |
| 期望氛围 | 最多 100 字符 |
| 必须保留 | 最多 10 条，每条 200 字符 |
| 禁止方向 | 最多 10 条，每条 200 字符 |
| 相关角色 | 最多 20 个 |
| 生成数量 | 3–6 条 |

## 六、目标数据模型

### 6.1 请求模型

```ts
export type BrainstormMode =
  | 'plot_twist'
  | 'scene_idea'
  | 'character_dev'
  | 'world_expand'

export type CreativityLevel = 'safe' | 'balanced' | 'bold'

export type BrainstormContextSource =
  | 'project_meta'
  | 'chapter_content'
  | 'chapter_snapshot'
  | 'outline'
  | 'characters'
  | 'relationships'
  | 'worldview'
  | 'foreshadows'
  | 'notes'

export interface BrainstormChapterRef {
  volume: string
  chapterId: string
  chapterTitle: string
}

export type BrainstormScope =
  | {
      type: 'current_chapter'
      chapter: BrainstormChapterRef
    }
  | {
      type: 'current_volume'
      volume: string
    }
  | {
      type: 'whole_project'
    }
  | {
      type: 'selected_chapters'
      chapters: BrainstormChapterRef[]
    }

export interface BrainstormProjectPreferences {
  schemaVersion: 1
  mode: BrainstormMode
  creativityLevel: CreativityLevel
  resultCount: number
  enabledContextSources: BrainstormContextSource[]
}

export interface BrainstormRequest {
  projectId: string
  mode: BrainstormMode
  problem: string
  scope: BrainstormScope
  relatedCharacters: string[]
  creativityLevel: CreativityLevel
  desiredTone: string
  mustKeep: string[]
  avoid: string[]
  resultCount: number
  enabledContextSources: BrainstormContextSource[]
}
```

范围校验规则：

- `current_chapter` 必须带完整章节引用。
- `current_volume` 必须带非空卷标识。
- `selected_chapters` 至少选择 1 章，并去除重复章节。
- 没有当前章节时，默认范围解析为最近一章；项目没有章节时禁止使用章节范围，并提示用户可改用全书背景模式。
- 当前卷无法从当前章节解析时禁用“当前卷”，不能隐式选择第一卷。
- UI 输入先转成合法的 `BrainstormScope`，服务层仍需再次校验，不能信任组件状态。

### 6.2 结构化结果

```ts
export interface BrainstormLocation {
  chapterId?: string
  chapterLabel: string
  positionNote: string
  verified: boolean
}

export interface BrainstormEntityRef {
  type: 'character' | 'worldview' | 'outline' | 'foreshadow' | 'chapter'
  entityId?: string
  label: string
  reason: string
  verified: boolean
}

export interface BrainstormIdea {
  id: string // 应用生成，不能采用 AI 返回的 ID
  title: string
  summary: string
  developmentSteps: string[]
  suggestedLocation: BrainstormLocation
  whyItFits: string
  connections: BrainstormEntityRef[]
  risks: string[]
  hooks: string[]
  creativityLevel: CreativityLevel
  favorite: boolean
  dismissed: boolean
  parentIdeaIds: string[]
}

export interface BrainstormResponse {
  summary: string
  ideas: BrainstormIdea[]
}
```

模型输出必须是 JSON，并经过运行时校验。AI 只返回标题、标签和内容，不负责生成应用内部 ID。解析器完成以下规范化：

1. 为每条建议生成稳定的应用 ID。
2. 将 AI 返回的章节、角色、伏笔等标签与本次允许引用列表匹配。
3. 匹配成功后写入真实 `entityId` 并标记 `verified: true`。
4. 无法匹配时保留显示标签，标记 `verified: false`，禁止跳转和自动预填。
5. 解析失败、结果为空或缺少必要字段时显示明确错误，不创建空白结果卡片，也不把原始响应伪装成正常建议。

运行时校验沿用项目现有的 `unknown` + type guard 风格，不为这一项单独引入 schema 库。

### 6.3 推演操作

```ts
export type BrainstormOperation =
  | 'generate'
  | 'deepen'
  | 'variant'
  | 'combine'
  | 'redo_with_feedback'

export interface BrainstormDerivation {
  operation: BrainstormOperation
  parentSessionId?: string
  parentIdeaIds: string[]
  feedback: string
}
```

规则：

- 首次生成使用 `generate`，没有父会话和父建议。
- 深化、变体和反馈重做必须有 1 个父建议。
- 组合必须有 2 个且只能有 2 个父建议。
- 每次推演创建新的子会话，不修改父会话。
- 推演默认重新读取最新项目上下文，同时把父建议原文作为高优先级输入；历史中保留父子链路。
- 单条推演使用独立加载状态和 `AbortController`，不能锁死其他已生成建议的复制、收藏等本地操作。

### 6.4 上下文清单与生成信息

```ts
export interface BrainstormContextManifestEntry {
  source: BrainstormContextSource
  entityIds: string[]
  labels: string[]
  estimatedTokens: number
  truncated: boolean
  error?: string
}

export interface BrainstormGenerationInfo {
  promptVersion: number
  providerName: string
  model: string
  durationMs: number
  inputTokens?: number
  outputTokens?: number
}
```

历史不保存完整拼接后的 prompt 和正文副本，避免重复存储大量项目内容；保存实际使用的数据来源、实体 ID、显示名称、估算 token、截断和错误信息，以便解释结果基于哪些资料生成。严禁保存 API Key。

### 6.5 历史会话

```ts
export interface BrainstormSession {
  schemaVersion: 1
  id: string // 应用生成
  projectId: string
  derivation: BrainstormDerivation
  request: BrainstormRequest
  response: BrainstormResponse
  contextManifest: BrainstormContextManifestEntry[]
  contextWarnings: string[]
  generation: BrainstormGenerationInfo
  createdAt: string
  updatedAt: string
}
```

存储位置：

```text
brainstorm/preferences.json
brainstorm/sessions/<session-id>.json
```

存储规则：

- 每个会话单独保存，避免新增、收藏或删除时重写一份无限增长的历史文件。
- Phase 1 每次成功解析结果后立即保存会话，再更新当前界面状态。
- `ProjectView` 保存当前会话 ID，切换 Tab 后重新挂载时恢复；应用冷启动时读取创建时间最新的有效会话。
- 历史列表初期可以读取会话文件元数据后在前端分页，不提前引入数据库或向量索引。
- 不得自动删除旧会话；删除由用户主动触发并通过通用 `<Modal>` 确认。
- 单个会话解析失败时跳过该会话并显示损坏警告，严禁用空数据覆盖原文件。
- 遇到高于当前支持范围的 `schemaVersion` 时只读提示“不支持的较新版本”，不得改写。
- 多个收藏、删除或保存操作按文件维度串行化；同一个会话的后写入不能静默覆盖尚未完成的前一次更新。
- 删除当前会话后清空 `ProjectView` 中的当前会话 ID，并选择下一条最新有效会话；没有其他会话时回到空状态。

### 6.6 跨模块数据契约

保存为备注属于 Phase 2A，使用向后兼容的可选来源字段：

```ts
export interface BrainstormSourceRef {
  type: 'brainstorm'
  sessionId: string
  ideaId: string
}

// 在现有 NoteEntry 上增加
source?: BrainstormSourceRef
```

`favorite` 和 `dismissed` 由应用初始化为 `false`，不要求 AI 输出。历史中只允许更新这两个用户元数据，不允许改写标题、步骤、依据、风险和连接等生成内容。

旧备注没有 `source` 时保持现有行为，不需要批量迁移。删除灵感会话时不级联删除备注或子会话；引用方只失去来源回跳能力，并显示“来源已删除”。删除确认弹窗应显示受影响的子会话和备注数量。

以下两个契约属于本轮不实施的 Phase 2B，仅作为后续独立确认时的设计基线：

```ts
export interface OutlineDraft {
  schemaVersion: 1
  id: string
  title: string
  content: string
  targetType: 'root' | 'volume' | 'chapter'
  targetRef?: string
  source: BrainstormSourceRef
  status: 'draft'
  createdAt: string
  updatedAt: string
}

export type WritingContextSelection =
  | {
      sourceType: 'material'
      sourceId: string
      title: string
      excerpt: string
    }
  | {
      sourceType: 'brainstorm'
      sourceId: string
      sessionId: string
      ideaId: string
      title: string
      excerpt: string
    }
```

大纲草稿建议存储于 `outline/drafts/<draft-id>.json`。通用写作上下文在内存中替代 `MaterialContextSelection[]`，现有素材选择通过 `sourceType: 'material'` 适配，不能破坏素材的插入、预览和使用记录。

## 七、服务层改造

### 7.1 拆分职责

当前 `runBrainstorm` 同时负责上下文读取、提示词、网络请求和结果切分。改造后拆分为：

```text
buildBrainstormContext(request)
  -> buildBrainstormPrompt(request, context)
  -> requestBrainstormCompletion(prompt, signal)
  -> parseBrainstormResponse(raw, allowedEntityRefs)
  -> saveBrainstormSession(session)
  -> setActiveSession(session.id)
```

`buildBrainstormContext` 同时返回拼接后的上下文、`contextManifest`、允许引用的实体表和警告。纯函数和 IO 函数分开，便于测试解析、提示词和模式上下文策略。

解析成功后界面进入 `saving` 状态。会话文件保存成功后进入 `success`；保存失败时进入 `unsaved`，保留本次内存结果供用户复制并允许重试保存，但不得伪装成已经进入历史。

### 7.2 Provider 请求约束

生成前必须校验：

- 已配置活动 Provider。
- `base_url` 非空。
- `api_key` 非空。
- `models.analysis` 非空。

请求要求：

- 规范化 `base_url` 尾部斜杠。
- 使用 `AbortController`。
- 默认 60 秒超时。
- 新生成开始时取消上一次未完成请求。
- 离开面板时取消请求。
- 非 2xx 响应最多读取 500 字符错误正文，过滤请求头、API Key 和其他敏感字段后转换为用户可理解的错误。
- 空响应视为错误。
- 读取 Provider 返回的 `usage.prompt_tokens` 和 `usage.completion_tokens`；缺失时保留为空，不伪造精确 token。

### 7.3 并发与状态一致性

界面状态使用明确状态机：

```ts
type BrainstormStatus = 'idle' | 'generating' | 'saving' | 'success' | 'unsaved' | 'error'
```

同时保存本次请求 ID。只有最新请求可以更新当前结果，避免旧请求覆盖新模式结果。

模式、范围或项目发生变化时：

- 取消正在进行的请求。
- 不把旧结果展示为新模式结果。
- 已保存的结果仍保留在历史中。

深化、变体、组合和反馈重做使用各自的请求 ID。父会话或父建议在请求进行期间被删除时，已发出的请求可以完成，但子会话保存前必须验证父引用；无法验证时停止保存并提示用户。

### 7.4 上下文错误处理

- 单个章节读取失败只跳过该章节，不能丢弃全部章节上下文。
- 可选文件不存在属于正常情况，但要记录来源缺失状态。
- 必需范围没有章节时阻止生成并给出明确提示。
- 世界观、角色或伏笔读取失败时，在结果顶部展示上下文警告。

### 7.5 AI 使用统计

所有成功的生成、深化、变体、组合和反馈重做都必须进入项目 AI 使用统计，记录：

- `feature: 'brainstorm'`。
- `operation`。
- 耗时。
- Provider 返回的 input/output token；Provider 未返回时只记录耗时和次数。
- 有明确当前章节时记录章节序号，没有时章节字段为空。

现有 `StatEvent` 和 `logAIGenerated` 只面向章节生成，需要扩展可选的 `feature`、`operation`、`input_tokens` 和可空章节字段，同时保持旧事件兼容。取消、超时、网络失败和结构校验失败不计为成功生成次数。

现有统计页的“Token 消耗”口径继续保持 output tokens，避免升级后历史曲线含义改变；新增的 input tokens 写入事件供后续单独展示，不与现有 `total_ai_tokens` 混加。

- 结果深化后保留父子关系。
- 恢复默认设置只重置项目偏好，不删除当前输入或历史。
- 落地操作必须经过确认，不直接覆盖目标模块。
- 历史列表为空、多页、筛选后为空时显示正确状态。

### 11.3 工程门禁

修改任何 `src/**/*.ts` 或 `src/**/*.tsx` 后必须运行：

```bash
npm run lint:risk
```

涉及构建行为时运行：

```bash
npm run check
```

并补充：

```bash
npm test
```

UI 完成后使用 Playwright 检查桌面和窄屏视口，确认：

- 左右分栏和移动端上下布局正确。
- 结果区独立滚动。
- 工具栏不被列表挤压。
- 长文本和操作按钮不重叠。
- Modal、历史分页和错误状态可操作。

Playwright 验收使用固定的本地 Provider 响应或网络拦截，不调用真实付费模型，覆盖成功、超时、空响应、非法引用和部分上下文失败场景。

## 十三、风险与控制

| 风险 | 控制方式 |
|---|---|
| 上下文过多导致 token 超限 | 为不同来源设置预算和优先级，截断时向用户提示 |
| AI 建议与设定冲突 | 输入模式化上下文，输出连接依据和风险，不自动采纳 |
| 结构化输出仍可能异常 | 运行时严格校验，异常时允许重试，不伪造成功结果 |
| 多次推演产生大量历史 | 提供筛选、收藏和主动删除，不自动清理用户数据 |
| 单一历史文件损坏或写入冲突 | 每会话独立文件、按文件串行更新、损坏文件只读警告 |
| 跨模块落地造成数据污染 | 所有写入先进入预填或草稿状态，并要求用户确认 |
| AI 返回虚构或歧义实体 | 只允许已验证引用跳转和预填，未验证引用降级为文本 |
| 主面板组件继续膨胀 | 按职责拆分控件、结果、历史和弹窗，但避免无意义抽象 |
| 不同 Provider 行为不一致 | 统一超时、错误解析和 JSON 校验，使用分析模型配置 |
| 灵感调用未进入统计 | 扩展兼容旧数据的 StatEvent，成功调用统一记录 feature 和 operation |
| Phase 2B 扩大写作/大纲模块范围 | 已确认本轮不实施；后续重新启动时设置独立确认门 |

---

## 十四、推荐实施顺序

严格按以下顺序推进：

1. Phase 0：先消除状态错配、无限等待和空结果等可靠性问题。
2. Phase 1：完成问题输入、模式化上下文、结构化结果和最小会话持久化，这是核心价值。
3. Phase 2A：增加深化、变体、组合、备注和伏笔预填，使用已有或向后兼容的数据模型形成基础闭环。
4. Phase 2B：本轮跳过；后续若重新启动，必须单独确认，不得自行扩大范围。
5. Phase 3：在已有会话存储上增加历史筛选、收藏、删除和分页。
6. Phase 4：完成可访问性、响应式和边界状态验收。

不建议先增加更多灵感模式。现有四种模式足以覆盖第一阶段，优先把“问题输入、范围控制、结构化比较、继续推演、保存落地”做完整。

## 十五、确认结果

以下三项已于 2026-07-20 确认，实施时按本节执行，不得自行扩大范围。

### 15.1 大纲草稿

- **确认结果：本轮不实施。** 暂缓到独立的大纲改造阶段。
- 本轮灵感模块只支持复制、备注和伏笔预填，不引入大纲草稿生命周期。

### 15.2 通用写作上下文

- **确认结果：本轮不实施。** 暂缓到 Phase 2A 验收后再决定。
- 本轮不改造 `ProjectView -> ChapterManager -> Editor -> contextEngine` 链路，也不把灵感伪装成素材上下文。
- 后续如实施，必须一次性建立 `WritingContextSelection` 联合类型，并完成素材回归验证。

### 15.3 项目偏好的记忆范围

- **确认结果：只记忆非内容型偏好。**
- 保存模式、创意尺度、结果数量和上下文来源。
