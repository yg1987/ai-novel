## 一、目的与边界

写作、审查与大纲都会展示“卷 → 章节”，也都需要在单卷数百章、全书数千章时快速定位章节。三者必须共享章节位置、排序和导航规则，不能各自从文件名或仅章号推导身份。

三者的职责明确如下：

- **写作**维护已落地的正文，以及实际卷名、章节名。
- **大纲**维护总纲、分卷纲和章节细纲；它只显示并使用“卷 N / 第 M 章”的位置，不重复维护卷名或章节名。
- **审查**只消费已有正文的写作章节，并叠加审查摘要和详情。

本方案不引入章节移动、跨卷排序、旧数据迁移或空的实际写作卷。

## 二、统一章节位置与名称

### 2.1 ChapterRef 是卷章节位置

```ts
interface ChapterRef {
  volume: string    // 内部位置，如“卷1”
  chapterId: string // 内部位置，如“ch001”
}

type ChapterKey = string & { readonly __brand: 'ChapterKey' }
// chapterRefKey({ volume: '卷1', chapterId: 'ch001' }) === '卷1:ch001'
```

`ChapterRef` 表示小说中的明确位置，即使该位置目前只有细纲、尚未落地正文，也可以作为大纲细纲的目标。所有跨模块状态、React key、缓存键和持久化映射使用 `ChapterKey`；不得只使用 `chapterId`。

### 2.2 写作维护实际名称

写作端的名称和位置分离：

```text
卷1 · 风起
  第一章 · 晴空
```

- `卷1`、`第一章`来自内部位置与排序。
- `风起`、`晴空`分别是用户在写作端维护的实际卷名、章节名。
- 改名只改展示，不改变 `ChapterRef`、文件路径或大纲细纲位置。
- 审查复用写作端名称显示；大纲不复制或编辑这些名称。

`listChapters(projectId)` 只返回已有正文；它是正文是否落地、正文标题和审查目录的事实来源，但不是大纲创建规划位置的前置条件。

## 三、大纲按位置存储

大纲不维护独立章节名称、绑定关系或“未绑定节点”。它按确定的卷章节位置存储：

```text
outline/
  outline.md                    # 总纲
  volumes/
    卷1.md                       # 卷1的分卷纲
  chapters/
    卷1/
      ch001.md                   # 卷1第一章细纲
```

路径由位置辅助函数唯一生成：

```ts
outlineVolumeFile(volume: string): string
outlineChapterFile(ref: ChapterRef): string
```

大纲侧栏按以下并集生成：

1. 写作端已有正文的 `ChapterRef`；
2. `outline/chapters/<volume>/<chapterId>.md` 已存在的细纲位置。

因此有三种清晰状态：

| 状态 | 含义 | 可用操作 |
|---|---|---|
| 只有正文 | 该位置尚未写细纲 | 创建细纲 |
| 只有细纲 | 该位置仍在规划 | 开始写作 |
| 两者都有 | 正文与细纲都已存在 | 编辑正文或细纲 |

“开始写作”不是绑定操作：它直接在该 `ChapterRef` 创建正文。正文删除时保留同位置细纲；审查报告归档后不再关联新正文。删除细纲时不删除正文。

## 四、卷与章节创建规则

### 4.1 不存在空的实际写作卷

写作页的“新建卷”必须是一个原子操作：一次创建新的内部卷和该卷第一章正文。卷名和章节名是可选的自定义展示名，不是创建正文的前置条件。写作目录中不保留没有正文的实际卷。

```text
新建卷
  → 创建 卷N/ch001.md
  → 可选写入卷名（例如“风起”）
  → 可选写入章节名（例如“晴空”）
```

这替代当前只在组件内存中创建空卷的行为。

写作页计算新卷时只看已有正文卷的最大卷号加一；若同一位置已有大纲细纲，直接复用该细纲，不另建绑定或副本。

### 4.2 新建分卷纲、章节与从细纲开始写作

- 在已有写作卷中新建章节，只能追加到该卷已有正文的最后一章之后；同位置已有细纲时直接复用该细纲。
- 大纲中新建分卷纲取写作位置与大纲位置中的最大卷号加一；它只创建规划卷，不创建实际写作卷。
- 大纲中新建细纲取该卷正文与细纲位置中的最大章节号加一，不能根据文件数量补号或覆盖已有细纲。
- 从细纲“开始写作”仅允许创建该卷下一个应落地的章节；若该卷尚无正文，则只能从 `ch001` 开始。卷名和章节名均可留空。
- 所有创建操作先检查目标正文或细纲文件是否存在；存在即拒绝覆盖并提示用户打开已有内容。

本版不提供章节移动、插队或自动重排。删除后位置编号可以有空缺；新建不会悄悄补空缺。

### 4.3 唯一创建入口

写作页和大纲页不得各自拼接正文路径或生成编号。创建逻辑集中为以下领域操作，由两个 Tab 共同调用：

```ts
createNewWritingVolume(projectId, { volumeName, firstChapterName })
createNextWritingChapter(projectId, volume, { chapterName })
startWritingFromOutline(projectId, ref, { volumeName?, chapterName })
createOutlineVolume(projectId)
createOutlineChapter(projectId, ref)
```

- 只有 `createNewWritingVolume()` 能创建实际卷，并同时创建 `ch001`。
- `createOutlineVolume()` 只创建“卷 N”的分卷纲位置，不写入实际卷名或正文目录。
- `startWritingFromOutline()` 复用相同位置校验；卷名和章节名不参与能否创建的判断。
- 每个操作在写入前验证位置、正文文件、细纲文件和名称元数据，失败时不改变当前选择。

### 4.4 可选名称与默认显示

- 内部卷位置和章节位置始终自动生成，例如 `卷1`、`卷2`、`ch001`、`ch002`。
- 用户不填写自定义名称时，界面直接显示位置名称：`卷1`、`卷2`、`第一章`、`第二章`，不得显示“未命名卷”或“未命名章节”。
- 用户填写名称后显示组合名称，例如 `卷1 · 风起`、`第一章 · 晴空`。
- 空名称不写入名称元数据；用户后续清空自定义名称时删除对应元数据并恢复位置名称。
- 默认位置名称是派生显示，不写回名称文件，也不改变 `ChapterRef`。

## 五、共享长篇导航

### 5.1 显示规则

- 写作：未自定义时显示 `卷1` → `第一章`；自定义后显示 `卷1 · 风起` → `第一章 · 晴空`。
- 大纲：`卷1` → `第一章`。
- 审查：只显示有正文的位置，名称复用写作端显示。

定位永远使用完整位置。跳转控件先选卷，再选章节；文本搜索的每个结果也必须显示完整“卷 N / 第 M 章”地址，不能把裸章号当作唯一目标。

### 5.2 树行为与性能

共享行为包括自然排序、卷折叠、章节段、外部跳转和目标滚入可视区：

- 卷按内部序号排序，卷内按章节序号排序；展示名不参与排序。
- 默认每段 50 章；章节段是运行时导航层，不改变文件结构。
- 跳转、新建和搜索命中自动展开目标卷及章节段；清空搜索后恢复用户原有展开状态。
- 树不使用传统分页；审查深度历史等线性列表才使用通用 `<Pagination>`。
- 写作与大纲先以折叠和章节段懒展开控制 DOM；审查面对数千章摘要时使用 `@tanstack/react-virtual`。

共享树只复用位置、折叠、分段和定位行为。写作的编辑菜单、审查状态徽标和大纲的“创建细纲/开始写作”仍由各 Tab 注入。

## 六、实施顺序与验收

1. 清空现有测试项目数据；不实现迁移、兼容读取或迁移 UI。
2. 定义唯一 `ChapterRef` / `ChapterKey` 与位置、文件路径辅助函数，修复多卷同 `ch001` 的状态和跳转。
3. 改造写作创建流程：新建卷同时创建第一章；卷名和章节名可选，新建章节只追加。
4. 改造大纲为确定路径的按位置细纲，并接入“只有正文 / 只有细纲 / 两者都有”状态。
5. 审查改为消费完整写作章节目录和完整 `ChapterRef`，再完成摘要、虚拟树和报告归档。

验收至少覆盖：

- `卷1 · 风起 / 第一章 · 晴空` 与 `卷2 · 入局 / 第一章 · 暗流` 的选择、保存、审查完全隔离。
- 先建 `卷1 / 第一章` 细纲，再从细纲创建名称为“风起 / 晴空”的正文。
- 写到 156 章后才使用大纲时，不自动生成空细纲；可从任意已有位置创建细纲。
- 删除正文后保留细纲；删除细纲后保留正文；删除正文后新建同位置正文不继承旧审查报告。
- 单卷 300 章通过章节段与完整卷章地址定位；5 卷、3000 章的审查树不读取报告详情且 DOM 行数受控。
