# 伏笔逻辑改造计划

> 创建日期：2026-07-15  
> 状态：方案设计  
> 关联问题：角色 panel 显示 33 处关联伏笔（5 章），关联逻辑过度宽松 + 自动创建不可控

---

## 一、设计原则变更

| | 当前逻辑 | 改为 |
|---|---|---|
| 伏笔来源 | AI 每次保存章节时自动分析并创建 | **人为手动创建**，AI 只做审查和建议 |
| 触发时机 | 被动（保存 → 自动） | **主动**（手动建伏笔 / 手动触发 AI 分析） |
| 数据更新 | 创建后永不更新、不清理 | 手动管理，全生命周期可控 |
| AI 角色 | 自动发现伏笔 → 创建记录 | **审查**已有伏笔的一致性 + **灵感建议** |

---

## 二、改造内容

### Phase A：断开自动创建链路

**目标**：保存章节不再自动调 AI 创建伏笔。

**涉及文件**：

| 文件 | 改什么 |
|------|--------|
| `src/components/Editor.tsx` ~L167 | 从 `handleSaveNow` 中移除 `void runIngest(html)` 调用 |

`runIngest` 方法本身和 `analyzeChapter`、`saveChapterSnapshot`、`syncForeshadowing` 等底层函数是否删除/保留另议（见 Phase C），Phase A 只断开触发链路。

**影响**：
- 保存不再产生 API 开销
- 之前已创建的旧伏笔数据不受影响，用户可在伏笔面板手动清理
- 第 1 章保存不再自动弹出"新增伏笔"条目

---

### Phase B：伏笔审查增强（优化已有功能）

**目标**：让已有的一致性审查更好地服务于"手动管理伏笔"的模式。

当前 `consistencyCheck.ts` 已有 6 项确定性检查：
1. **沉寂伏笔** — planted/advanced 状态超过 N 章未推进
2. **超期未回收** — 超过计划回收章节仍未处理
3. **回收延迟** — 实际回收比计划晚了 N 章
4. **伏笔密度** — 未回收伏笔数 / 总章数
5. 角色缺席检查（与伏笔不直接相关）
6. 时间线顺序检查（与伏笔不直接相关）

**改进点**：
- 审查面板入口改名为「一致性审查」（不仅限于伏笔）
- 审查结果中为伏笔问题增加"跳转到伏笔面板"的操作入口
- 审查规则面板 (`ReviewRulesEditor`) 中伏笔相关阈值（`dormantForeshadowWarn`、`densityWarningThreshold` 等）保持用户可调

**涉及文件**：

| 文件 | 改什么 |
|------|--------|
| `src/components/ReviewPanel.tsx` | 审查结果中伏笔问题增加跳转链接 |
| `src/services/consistencyCheck.ts` | 不改逻辑，确认 4 项伏笔检查覆盖度 |
| `src/components/ForeshadowPanel.tsx` | 确认接收 jump-to / highlight 参数（已有 `highlightId` prop） |

### Phase C：新增"伏笔灵感分析"功能

**目标**：用户选中一卷（或全篇），AI 分析后给出伏笔建议，用户决定是否采纳。

#### 触发入口

在伏笔面板顶部的健康概览区，新增一个按钮：

```
┌─────────────────────────────────────────────┐
│ 伏笔债评分 85/100  未回收 12 条  密度 0.18  │
│                            [🔍 AI 分析建议] │
└─────────────────────────────────────────────┘
```

点击后弹出选择范围：

```
┌──────────────────────────┐
│ AI 伏笔灵感分析           │
│                          │
│ 分析范围：               │
│ ○ 第1卷 · 入道 (ch1-15)  │
│ ○ 第2卷 · ...            │
│ ○ 全篇                   │
│                          │
│     [开始分析]            │
└──────────────────────────┘
```

#### AI 分析逻辑

输入给 AI 的内容：
1. 所选范围的所有章节正文（截断到 token 限制）
2. 已有伏笔列表（名称 + 状态 + 简述）
3. 现有角色列表（名称）

AI 输出结构（JSON）：

```json
{
  "gaps": [
    {
      "location": "第3章",
      "reason": "主角在第5章突然会御剑，前4章未提他练剑",
      "suggestion": "在第2或第3章加一处他偷偷练剑的场景",
      "relatedCharacters": ["主角名"]
    }
  ],
  "callbacks": [
    {
      "target": "第1章有一把古剑的描述，后面再未出现",
      "suggestion": "可回收为某个已设伏笔的关键道具",
      "relatedForeshadow": "已有伏笔ID"
    }
  ],
  "density": {
    "chapters_with_too_many": ["第3章（7条）"],
    "chapters_with_none": ["第6章", "第7章"],
    "overall_assessment": "前密后疏，建议在第4-5章分散"
  },
  "summary": "总结性建议"
}
```

#### 展示方式

在伏笔面板中新增一个 tab 或展开区域，以卡片形式展示每条建议：

```
┌─────────────────────────────────────────────────┐
│ 💡 AI 灵感建议 · 第1卷                    [关闭] │
├─────────────────────────────────────────────────┤
│ 🔴 缺口                                         │
│ 📍 第3章                                        │
│ 主角在第5章突然会御剑，前4章未提练习              │
│ 💡 建议在第2或第3章加练剑场景                    │
│ 👤 角色: 林烬                        [📝 采纳]   │
├─────────────────────────────────────────────────┤
│ 🟡 呼应                                         │
│ 📍 第1章 → 未使用                               │
│ 古剑描述出现一次后从未再提                      │
│ 💡 可回收为「XXX」的伏笔道具                     │
│ 🔗 已有伏笔: 暂无                     [📝 采纳]  │
├─────────────────────────────────────────────────┤
│ 🟢 密度                                         │
│ 第3章 7 条伏笔过多，第6-7章 0 条                 │
│ 💡 建议将第3章部分伏笔延后到第5-6章              │
│                                  [📝 采纳全部]   │
└─────────────────────────────────────────────────┘
```

点击「采纳」后：
- 缺口建议 → 打开伏笔创建表单，预填名称/描述/关联角色
- 呼应建议 → 跳转到对应章节编辑器
- 密度建议 → 高亮提示

## 三、与前版计划的关系

`doc/伏笔模块完善计划.md` 中的功能（紧迫度分类、线索追踪、健康评分、审查规则）保留不动。本次改造聚焦于**数据来源策略**的变更，不重写面板 UI 和审查逻辑。

具体来说：
- 伏笔面板 CRUD 操作 → 不改
- 状态机（planted → advanced → resolved/abandoned）→ 不改
- 紧迫度分类 + 健康评分 → 不改
- 上下文注入（foreshadowContext.ts）→ 不改
- AI 自动创建（memorySync.ts 中的新增伏笔逻辑）→ Phase A 断开触发，函数保留备用

---

## 四、实施顺序

1. **Phase A** — 一锤子，最简单，切了就生效
2. **Phase B** — 轻量改动，验证现有审查功能覆盖度
3. **Phase C** — 新功能开发，工作量最大

建议先做 A → 验证效果 → 再做 B + C。
