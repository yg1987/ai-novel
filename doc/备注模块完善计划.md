> 创建日期：2026-07-14
> 状态：全部三层已完成

## 一、背景

备注 Tab 是 v0.2 引入的辅助功能，定位为「写作过程中的碎片化思绪收容所」。第一层改进已完成（拆文件存储、补全 question 类型、textarea 输入、inline 编辑、六按钮过滤器、客户端分页），当前状态如下：

| 维度 | 当前状态（第一层完成后） |
|---|---|
| 数据模型 | `NoteEntry { id, content, type, chapterRef, done, resolved, createdAt }` |
| 类型支持 | `'note' \| 'todo' \| 'question'`，三种类型均已实现 |
| 存储 | 独立文件 `notes/<id>.json`，旧 `notes.json` 自动迁移 |
| 编辑能力 | 添加、删除、待办/疑问状态切换、inline 编辑 |
| 搜索 | 拆文件后 Rust 后端搜索自然覆盖单条备注，结果可读 |
| AI 集成 | 零。上下文引擎 DataSourceRegistry 不含 notes，不参与向量检索 |
| 章节关联 | `chapterRef` 字段存在但始终为空字符串，UI 无关联入口 |
| 布局 | 单列全宽面板，`overflow: hidden` + `overflow-y: auto` 两层滚动控制 |

## 二、使用场景分析

### 场景 1：章节关联备忘（核心价值）

> "写到第 3 章发现主角性格偏软了，在第 1 章加一段童年闪回就能圆回来"
> "第 7 章埋了玉佩的伏笔，收尾时别忘了呼应"

**特征**：价值绑定在具体章节上，脱离章节上下文就没意义。最有价值的场景——写第 N 章时能看到关于第 N 章的所有历史备注。

### 场景 2：项目级 TODO

> "大纲第 3 卷的势力关系还没理顺"
> "角色'陈默'的动机需要重新设计"

**特征**：跨章节的管理项，不跟具体章节绑定但跟项目进度强相关。

### 场景 3：灵感碎片

> "要不要把反派也写成穿越者？"
> "想到一个很妙的章节标题"

**特征**：随手记的脑洞，轻量级记录，可能用也可能不用。

### 第一层：基础补全

**目标**：把 MVP 修到日常可用。改动小，效果立竿见影。

#### 1.1 输入与编辑体验

**现状**：两个问题——
1. 添加用的是 `<input>`（单行），备注和待办经常超过一行（如"第 3 章结尾情绪转折太快，需要在转身之前加一段内心独白"），单行滚动看不到上文
2. 已有内容不可编辑，写错只能删除重建

**方案**：

**1.1a 输入框改为 `<textarea>`**

```tsx
<textarea
  className="notes-input"
  value={newContent}
  rows={2}
  placeholder="添加备注、待办或疑问…"
  onKeyDown={(e) => {
    if (e.key === 'Enter' && !e.shiftKey) {
      e.preventDefault()
      handleAdd()
    }
  }}
/>
```

- 两行高度起步，自动撑高到最多 6 行
- Enter 提交，Shift+Enter 换行
- **不加字数限制**——这是给自己记东西，不是发布内容，限制会打断思路

**1.1b 支持编辑已有备注**

点击备注内容区域进入 inline 编辑模式（切换为 `<textarea>`），失焦或 Enter 保存。

**改动范围**：
- `src/components/NotesPanel.tsx`：`<input>` → `<textarea>`，新增 `editingId` 状态，渲染时判断编辑态
- `src/style.css`：`.notes-input` 改为 textarea 样式（最小高度、行高），`.note-edit-area` 和 `.note-edit-textarea` 编辑态样式

#### 1.2 补全「疑问」类型

**现状**：`doc/操作说明.md` 和 `doc/系统设计说明.md` 都描述了三种批注类型（备注/TODO/疑问），但代码只有两种。

**方案**：
- `NoteEntry.type` 扩展为 `'note' | 'todo' | 'question'`
- `NoteEntry` 新增 `resolved: boolean` 字段（用于疑问类型，语义与待办的 `done` 平行但独立）
- 下拉框增加「❓ 疑问」选项
- 疑问类型显示「✓ 解决 / ↩ 重开」按钮，切换 `resolved` 状态
- 过滤器增加「疑问❓」和「已解决✅」两个视图（见 1.4）

**改动范围**：
- `src/components/NotesPanel.tsx`：类型定义、下拉框选项、过滤按钮、状态切换按钮
- `src/style.css`：`.note-item.question` 样式

#### 1.3 修复搜索集成

**现状**：Rust 后端 `search_project_files` 遍历 `notes/` 目录并对 `notes.json` 做全文搜索。问题是它是一个大 JSON 数组，搜索结果是整块 JSON 字符串，用户看到的 snippet 是一堆 `[{...},{...}]`，完全不可读。

**方案**：在前端搜索链路上增加 notes 的处理——在 `hybridSearch` 拿到 notes 来源的结果后，解析 `notes.json` 并对每条 note 的 `content` 做匹配，替换掉原始的 JSON snippet。

或者更简单：把 notes 拆成单独文件（`notes/<id>.json`），每条一个文件，搜索时自然就是独立结果。

**推荐用拆文件方案**，因为：
- 和项目中角色、世界观、章节的文件组织方式一致
- Rust 搜索和后端不需要任何改动
- 未来做向量索引也更容易（每条 note 独立 embedding）

**改动范围**：
- `src/components/NotesPanel.tsx`：CRUD 操作改为读写 `notes/<id>.json`，用 `listProjectFiles` 列出所有备注
- 数据迁移：首次加载时自动把旧 `notes.json` 拆分为独立文件

#### 1.4 过滤器重构

**现状**：当前只有「全部 / 备注 / 待办」三个按钮，按类型过滤。问题是待办一旦标记完成就混在列表中（仅靠 `done` 样式区分），无法快速只看未完成项。同样，补全 question 类型后已解决/未解决也会混在一起。

**方案**：改用类型 + 状态合并的过滤器，一行按钮覆盖所有视图：

```
[全部] [备注] [待办⏳] [已办✓] [疑问❓] [已解决✅]
```

语义：
- **全部**：不区分类型和状态，所有条目
- **备注**：`type='note'` 的条目（无状态概念，不需要进一步区分）
- **待办⏳**：`type='todo'` 且 `done=false`，即未完成的待办
- **已办✓**：`type='todo'` 且 `done=true`，已归档的完成项
- **疑问❓**：`type='question'` 且 `resolved=false`，待解决的疑问
- **已解决✅**：`type='question'` 且 `resolved=true`，已解决的疑问

设计理由：
- 备注没有状态，所以只有一种视图
- 待办的「未完成」和「已完成」天然需要分开——找要做的事看「待办」，回顾做过的事看「已办」
- 疑问同理——还有悬而未决的看「疑问」，已确认的归入「已解决」
- 操作路径短，一次点击就切到目标视图，不需要两级过滤

**改动范围**：
- `src/components/NotesPanel.tsx`：`filter` 状态类型扩展，过滤逻辑更新，按钮组重写
- `src/style.css`：新增 `.notes-filter` 按钮样式微调（可能需要在 6 个按钮时换行或缩小间距）

#### 1.5 客户端分页

**现状**：备注列表无分页，直接 `notes.map(...)` 渲染所有条目。拆文件后每个项目可能有几十上百条备注，一页全渲染既不性能也不可用。

**方案**：客户端分页，无需改后端。

- 首次加载时一次性读取 `notes/` 目录下所有文件（每个文件几十字节，100 条也才几 KB）
- NotesPanel 内部维护 `page`（当前页码，从 1 开始）和 `PAGE_SIZE`（默认 20）
- 根据当前 filter 过滤出 `filtered` 列表，再按时间倒序排序
- 渲染时只取当前页：`filtered.slice((page - 1) * PAGE_SIZE, page * PAGE_SIZE)`
- 底部显示「加载更多」按钮或分页控件（页码 + 上一页/下一页）
- 切换 filter 时重置到第 1 页

**为什么不选无限滚动**：备注列表有搜索需求（全局搜索 Tab），如果列表项位置随滚动动态加载，定位到某条备注后回到面板时位置会丢失。分页的位置是确定的——"第 3 页第 5 条"。

**改动范围**：
- `src/components/NotesPanel.tsx`：新增 `page` state，分页计算逻辑，分页控件 UI
- `src/style.css`：`.notes-pagination` 样式

### 第二层：章节关联

**目标**：让备注从「独立碎片」变成「写作流程中随叫随到的记忆辅助」。这是这个功能产生真正价值的转折点。

**设计原则**：避免硬编码，遵循解耦——组件间通过 props 回调通信，共享逻辑抽取为独立模块，不引入隐式全局状态。

#### 2.1 `chapterRef` 字段可用化

**前置事实**（代码审查确认）：
- 章节文件结构为 `chapters/{卷名}/{chapterId}.md`，两层嵌套
- 获取章节列表的正确 API 是 `listChapters(projectId)` → `ChapterMeta[]`（含 id、title、volume、order），**不能用** `listProjectFiles`（只能列出一层目录）
- NotesPanel 当前 props 只有 `{ projectId }`，无导航能力
- ProjectView 是唯一 tab 状态持有者，已有一条导航先例 `handleNavigateToReview`

**方案**：

**(a) chapterRef 编码格式**：`"{volume}/{chapterId}"`，空字符串 = 项目级。这个格式唯一确定一个章节。

**(b) NotesPanel 侧**：
- 调用 `listChapters(projectId)` 加载章节列表
- 添加备注区域增加可选章节下拉（按 volume 分组 `<optgroup>`）
- 备注卡片 header 显示章节名称标签，点击触发 `onNavigateToChapter(chapterRef)`
- 章节名称从 `chapterList` 中按 chapterRef 查找 `ChapterMeta.title`，找不到时降级显示原始字符串

**(c) ProjectView 侧**：参照已有 `handleNavigateToReview` 模式，新增回调传递给 NotesPanel：

```tsx
const handleNavigateToChapter = (chapterRef: string) => {
  setNavigateChapterRef(chapterRef)
  setTab('writing')
}
```

**(d) ChapterManager 侧**：接收 `initialChapterRef?: string | null` prop，在 `useEffect` 中解析 chapterRef（`parseChapterRef`），查找匹配章节并自动选中，同时展开对应分卷。

**改动范围**：
- `src/components/NotesPanel.tsx`：新增 `onNavigateToChapter` prop、章节下拉、标签渲染、跳转逻辑
- `src/components/ProjectView.tsx`：新增 `handleNavigateToChapter` 回调，传给 NotesPanel
- `src/components/ChapterManager.tsx`：接收 `initialChapterRef`，解析 chapterRef 并自动选中对应章节
- `src/api/tauri.ts`：类型导入 `listChapters`（已存在）

#### 2.2 按章节分组视图

**方案**：在备注面板过滤栏增加视图切换——「📋 时间线」（当前默认）和「📂 按章节」。

「按章节」模式下：
- 未关联章节的备注归入「📌 项目级」
- 关联了章节的按 `(volume, chapterTitle)` 分组，每组可折叠/展开
- 折叠状态用 `useState<Set<string>>` 维持（会话内有效，不持久化——切换 tab 后重置是合理行为）
- 每个分组内仍按时间倒序排列
- 分页仍然生效（每页 20 条，跨分组连续计数）

**改动范围**：
- `src/components/NotesPanel.tsx`：新增 `viewMode` 状态、`groupedNotes` 计算逻辑、分组渲染、折叠状态

#### 2.3 待办状态在章节列表中可见

**数据共享方案**：ChapterManager 和 NotesPanel 各自调用 `loadAllNotes` 加载备注数据。备注文件少（<100 个），双重 I/O 开销可忽略。不引入全局状态。

**实现**：
1. 把 `loadAllNotes` 从 NotesPanel.tsx 抽取到 `src/services/notesStorage.ts`，同时导出 `getNotesForChapter`、`countPendingTodos` 等工具函数
2. NotesPanel 和 ChapterManager 各自引入
3. ChapterManager 章节列表中渲染待办徽标：`🟡 {count}`（仅 `type='todo'` + `done=false` + `chapterRef` 匹配当前章节）

**改动范围**：
- `src/services/notesStorage.ts`：**新建**，共享备注数据加载与查询逻辑
- `src/components/NotesPanel.tsx`：引入 notesStorage，删除内部 `loadAllNotes` 定义
- `src/components/ChapterManager.tsx`：侧栏底部「本章备注」区块 + 章节列表待办徽标
- `src/style.css`：`.chapter-todo-badge`、`.chapter-notes-section` 样式

## 六、设计原则与注意事项

### 设计原则

1. **避免硬编码**：所有数据查询逻辑（如 `loadAllNotes`、`getNotesForChapter`）抽取到独立服务模块，组件只做渲染和用户交互。

2. **组件解耦**：跨组件通信通过 props 回调（如 `onNavigateToChapter`），不引入隐式全局状态或事件总线。两个组件需要同一份数据时各自加载，不互相依赖。

3. **chapterRef 编码格式**：统一 `"{volume}/{chapterId}"`，空字符串 = 项目级。所有读写 chapterRef 的代码都遵循这一约定，不散布格式转换逻辑。

4. **先建共享模块再改组件**：每层执行的第一步是建立该层需要的共享基础设施（如 `notesStorage.ts`），再逐个改组件引用。

### 注意事项

1. **数据兼容性**：拆文件方案需要处理旧项目已有 `notes.json` 的迁移，迁移逻辑必须幂等（多次执行不产生重复数据）。

2. **性能**：拆文件后，备注数量多（50+）时 `listProjectFiles` + 逐个文件读取可能有性能问题。可以在 NotesPanel 中缓存已加载的备注列表，通过 `saveNotes` 时同步更新缓存。

3. **文档同步**：补全 question 类型后，`doc/操作说明.md` 和 `doc/系统设计说明.md` 中的「疑问」描述就不再是文档漂移了。同时需要更新 `AI改写扩写功能补充计划.md` 中关于 NotesPanel「纯展示 div」的过时描述。

4. **不做过度工程**：第三层的智能提醒（积压标记、红点、导出检查）每一项都需要评估是否真正必要。如果是个人使用，可能完全不需要——自己写的备注自己清楚。建议在实际使用第一、二层一段时间后，根据真实痛点决定第三层做哪些。

5. **不加强制约束**：待办和疑问的"完成/解决"状态始终保持可选。这是写作辅助工具，不是项目管理系统，强制完成机制会破坏写作流畅感。

6. **滚动条模式**：第一层实践中确认了正确的滚动实现——外层容器 `overflow: hidden` 限制高度，内层列表 `flex: 1; min-height: 0; overflow-y: auto` 触发滚动。第二层新增的「本章备注」区块如需要滚动，遵循同一模式。
