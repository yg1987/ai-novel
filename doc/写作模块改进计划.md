## 一、文档目的

本文档登记写作模块已确认的章节标识问题，并明确卷章节创建、长篇章节导航、编辑器保存一致性和本版功能边界。

本文档不是写作模块的完整重构方案，也不代表已完成对编辑、保存、生成、版本、备注、导出等链路的全面检查。正式实施前必须重新运行 CodeGraph，补全调用关系和回归测试方案。

## 二、统一章节标识前置约定

全系统中的章节不能只用 `chapterId` 标识。统一采用：

```ts
interface ChapterRef {
  volume: string
  chapterId: string
}
```

内存集合键统一通过已有公共函数生成：

```ts
chapterRefKey({ volume, chapterId })
// => `${volume}:${chapterId}`
```

约定：

- 组件和服务边界传递 `ChapterRef`，不只传 `chapterId`。
- React key、`Map`、`Set`、`Record` 和选中状态使用统一 `ChapterKey`。
- 禁止不同模块各自拼接章节键。
- 文件/API 路径仍保留结构化的 `volume` 与 `chapterId` 参数。

本版不提供章节排序、位置调整或跨卷移动，因此 `ChapterRef` 在本版生命周期内同时承担唯一引用与存储定位职责。本版不为尚不存在的移动能力引入额外的不可变章节 UID，也不预留移动交互。

## 三、当前已确认情况

### 3.1 正文章节文件本身按卷隔离

写作正文读写接口已经同时接收 `volume` 和 `chapterId`：

```ts
getChapterContent(projectId, volume, chapterId)
saveChapterContent(projectId, volume, chapterId, content)
```

磁盘路径位于对应卷目录，因此第一卷 `ch001` 和第二卷 `ch001` 的正文文件本身可以共存。

这部分设计方向正确，应保留。

### 3.2 当前选中状态只保存 `chapterId`

`ChapterManager` 当前使用：

```ts
const [activeChapterId, setActiveChapterId] = useState<string | null>(null)
```

随后通过：

```ts
chapters.find((c) => c.id === activeChapterId)
```

定位当前章节。

不同卷可以同时存在 `ch001`。选择后续卷的 `ch001` 时，以上查找可能返回前一卷的同 ID 章节，导致：

- 当前卷判断错误。
- 编辑器读取错误卷的正文。
- 保存时使用错误卷路径。
- 多个同 ID 章节同时显示为选中。

### 3.3 部分章节附属状态按 `chapterId` 建索引

当前已观察到以下状态或元数据只以 `chapterId` 为键：

- 自定义章节标题 `chapterTitles[ch.id]`。
- 当前选中章节。
- 章节行 active 判断。
- 部分菜单打开状态。
- 备注数量映射。
- 编辑器章节预计字数覆盖。
- 跳转审查页的参数。

这些数据在多卷同 ID 场景下可能共享、覆盖或同时生效。

## 四、初步改进方向

### 4.1 选中状态改为完整章节引用

建议状态语义调整为：

```ts
const [activeChapterRef, setActiveChapterRef] = useState<ChapterRef | null>(null)
```

或者同时保存：

```ts
const [activeChapterKey, setActiveChapterKey] = useState<ChapterKey | null>(null)
```

章节查找必须同时匹配：

```ts
c.volume === ref.volume && c.id === ref.chapterId
```

### 4.2 所有章节 Map 改用统一 ChapterKey

以下数据应逐项确认并改为 `volume:chapterId`：

- 自定义章节标题。
- 字数覆盖。
- 备注数量缓存。
- 行菜单状态。
- 版本历史目标。
- 导出目标。
- 当前选中和跳转目标。

不能只修改 `activeChapterId`，否则界面选中正确后，附属数据仍可能跨卷串用。

### 4.3 导航事件传递完整引用

写作页向审查、备注、素材和其他模块跳转时，应统一传 `ChapterRef` 或已有的 `CurrentChapterRef`，避免下游重新用不完整 ID 猜测卷。

### 4.4 新建卷即创建第一章

当前项目数据是可清空的测试数据，不实现旧元数据迁移或冲突处理。新建卷改为单一原子操作：

1. 计算下一个内部卷位置，例如 `卷3`。
2. 创建 `chapters/卷3/ch001.md`。
3. 卷名和第一章名称由用户按需填写；均可留空。
4. 只保存非空名称元数据，正文与必要元数据写入成功后将新章节设为当前章节。

因此写作目录不存在空卷；左侧始终至少显示“卷 N / 第 M 章”位置。未填写名称时直接显示 `卷N`、`第M章`，不得显示“未命名”；填写后显示“卷 N · 实际卷名 / 第 M 章 · 实际名称”。创建前必须检查正文目标，存在时拒绝覆盖。

名称可选规则同时适用于“新建分卷与第一章”“新建章节”和“从细纲开始写作”。空值不写入名称元数据，后续清空自定义名称时恢复位置名称。

### 4.5 长篇章节导航

本节的树模型、搜索和分段规则遵循 `doc/卷章节目录与大纲关联公共方案.md`。写作页是章节目录的事实源；共享导航只复用树行为，不承载写作页的编辑、删除和菜单业务。

左侧继续保留“卷 → 章节”的主要结构，并根据章节数量自动切换展示方式：

- 单卷不超过 50 章时，直接显示章节列表，保持现有使用习惯。
- 单卷超过 50 章时，自动生成“第 1–50 章”“第 51–100 章”等章节段。
- 章节段是运行时导航层，不创建新目录，不修改章节文件结构。
- 默认只展开当前章节所在的章节段，外部跳转和新建章节后自动展开目标卷与目标章节段。
- 左侧顶部提供固定搜索框；搜索结果按卷分组并显示完整“卷 N / 第 M 章”位置与实际名称。
- 点击搜索结果后选中完整 `ChapterRef`，并保证目标章节行进入可视区域。
- 清空搜索后恢复章节树，同时保留当前章节所在卷和章节段的展开状态。
- 默认每段 50 章；项目级可选 25、50、100 章，也可选择始终直接显示或始终分段。跳转控件先选择卷，再选择章节，不以裸章号作为唯一目标。
- 当前阶段不默认引入虚拟列表。分段后每次通常只渲染一个章节段，只有性能实测仍不达标时才评估虚拟化。

如写作页后续需要虚拟化，必须复用公共方案定义的扁平 `TreeRow` 模型和导航语义，不得重新实现一套搜索、折叠或定位规则。

章节行在狭窄侧栏中不再并列展示多个操作按钮。备注、发布复制、版本历史和删除统一收纳到单个“更多”菜单，避免标题被操作区挤压，并减少长列表中的组件数量。

### 4.5.1 2026-07-21 实施审计

P0：取消卷名、章节名必填；补齐搜索、章节段定位与目标滚入可视区。

P1：提供 25/50/100 章分段选择和卷章直达；章节操作收纳到单个“更多”菜单。

### 4.6 章节编号语义

章节身份、存储顺序和展示编号必须分离：

- `chapterId` 是卷目录内的存储标识，不作为全局身份或展示编号。
- `order` 表示卷内顺序；本版不提供调整顺序的操作。
- 全书连续编号根据卷顺序和卷内顺序动态计算，不持久化为章节身份。
- 默认沿用卷内编号，项目级允许切换为全书连续编号。
- 导出时可以独立选择卷内编号或全书连续编号。
- 无论采用哪种显示方式，React key、缓存键、文件路径和跳转参数都不得使用展示编号代替 `ChapterKey`。

### 4.7 编辑器保存一致性

编辑器实例必须在完整 `ChapterRef` 下加载和保存正文，并保证保存时序正确：

- 自动保存、手动保存和卸载保存必须绑定同一完整章节引用。
- 只有写入成功后才能更新“已保存版本”，失败后必须保留待保存状态。
- 同一章节的保存任务需要串行化或使用等价的顺序保护，防止旧请求晚完成后覆盖新内容。
- 切换章节、返回书架和删除当前章节前，需要刷新待保存内容。
- 保存失败必须提供明确反馈和重试入口，不得只写入控制台日志。

## 五、后续详细审查范围

下一轮写作模块专项审查按以下三层执行。

### 5.1 本轮必须修复

1. `ChapterManager` 的选中、创建、删除、卷显示名称、卷折叠、章节段展开、搜索和章号直达状态。
2. `Editor` 的正文加载、自动保存、手动保存、卸载保存、保存失败重试和并发保存顺序。
3. 自定义章节标题、章节预计字数、生成字数决议、重置逻辑、备注数量和行菜单状态使用统一 `ChapterKey`。
4. 写作页向审查、备注、伏笔、关系图、搜索结果和其他章节入口传递完整章节引用。
5. 删除章节或卷时，正文、自定义标题、字数覆盖、备注、审查结果、快照、素材来源和当前导航状态的级联清理。
6. 新建多卷同序号章节后的 React key、列表高亮、当前编辑器、章节段定位和菜单状态。
7. `chapterId` 被用于日志、缓存、索引、持久化键或异步任务标识的其他位置。
8. 新建卷同时创建第一章、写入卷名与章节名，且不存在空的实际写作卷。

### 5.2 需要回归验证

1. 正文 API 按 `projectId + volume + chapterId` 读取和保存正确文件。
2. 版本历史按完整章节引用列出、读取、恢复、重命名和删除版本。
3. 单章发布复制始终读取目标卷下的章节；整书导出按选定编号方式输出正确顺序。
4. 项目存档完整包含卷名、章节名与各章节附属元数据，不把存档误当作单章目标操作。
5. 素材引用、AI 上下文和章节来源记录在多卷同 ID 场景下互不混淆。
6. 卷显示名称修改只改变展示，不改变卷目录和 `ChapterRef`。

### 5.3 本版明确排除

- 章节拖拽排序、按钮排序、指定位置调整和跨卷移动。
- 卷目录的物理重命名。
- 与章节正确性无关的正文编辑器工具栏、排版和专注模式重构。

以上排除项不得在实施过程中顺带加入；如未来需要，另立方案重新审查身份模型和全部下游引用。

## 六、初步验收场景

后续完整方案至少必须覆盖：

- 第一卷和第二卷同时存在 `ch001`。
- 分别选择两章时，只有目标行高亮。
- 两章分别读取和保存自己的正文。
- 两章可以设置不同的自定义标题。
- 两章可以设置不同的预计字数覆盖。
- 两章的版本历史、备注数量和导出目标互不混淆。
- 从第二卷 `ch001` 跳转审查页时仍保持第二卷引用。
- 新建卷时立即得到“卷 N · 卷名 / 第一章 · 章节名”，刷新页面后该卷和章节仍正常显示。
- 单卷 300 章时显示 6 个默认章节段，不一次渲染全部章节行。
- 输入章号或标题可以跨卷定位，点击结果后目标卷、章节段和章节行自动进入可见状态。
- 新建第 301 章后自动生成并展开“第 301–350 章”章节段，且新章节立即成为当前章节。
- 自动保存、手动保存和切章连续发生时，较旧保存结果不会覆盖较新正文。
- 保存失败后仍保持未保存状态，并可重试成功。
- 删除章节或卷后不存在孤立的标题、字数、备注、审查、快照或素材来源记录。
- 卷内编号和全书连续编号只改变展示及导出结果，不改变章节身份和文件路径。

## 七、本版已确定的产品边界

### 7.1 不提供章节移动能力

本版没有章节移动这一动作，不提供拖拽排序、按钮排序、指定位置调整或跨卷移动。新建章节始终追加到目标卷末尾。删除后允许卷内编号出现空缺，本版不自动重排现有章节 ID、文件名或引用。

### 7.2 长篇导航采用自动分段

长卷采用自动章节段、标题搜索和章号直达，具体规则见 4.5。当前不把虚拟列表作为前置依赖，也不使用传统分页切割章节树。

### 7.3 编号与身份分离

默认显示卷内编号，允许项目级切换全书连续编号，并允许导出时单独选择。展示编号不得参与章节身份和持久化键，具体规则见 4.6。

### 7.4 编辑器只处理正确性改进

本版审查正文加载和保存正确性，但不扩展为编辑器整体交互重构。工具栏、排版、专注模式等体验改造需要另立专项计划。

### 7.5 新卷不以空目录存在

写作页不提供仅在内存中存在的空卷。用户新建卷时必须同时创建第一章正文，并填写卷名与章节名；只有细纲、尚未有正文的卷只在大纲 Tab 中按“卷 N”位置显示。
