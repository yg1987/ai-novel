## 一、改造背景

当前素材功能以 `resources/<分类>/<文件>` 为业务模型。素材 Tab 已经可以在应用内新建、编辑和删除文件，也提供了基础 AI 分类建议；写作页素材侧栏可以浏览文件、预览前 500 个字符并插入编辑器；搜索模块还会读取素材文件并尝试建立项目级向量索引。

问题不在于“完全不能在应用内编辑”，而在于整个功能仍然是一套文件管理器：

1. **文件不等于素材**：标题、正文、来源、标签、收藏和项目关联都没有独立字段。
2. **收集流程不自然**：新建素材仍要求用户先选择目录并考虑文件名和扩展名，不符合随手收集的习惯。
3. **分类能力受目录限制**：一条素材只能放在一个文件夹，无法用标签和项目关联交叉组织。
4. **写作调用能力弱**：侧栏只能插入被截断的预览文本，不能选择摘录，也不能加入本章 AI 上下文。
5. **搜索和索引不完整**：关键词搜索、项目级向量索引和素材文件 CRUD 各自维护路径语义，删除素材后也没有完整的旧向量清理闭环。
6. **现有 AI 分类不可直接复用**：建议分类写死在提示词中，无法基于用户实际配置的类型和分类工作。

改造后的素材库应回答三个问题：

- 写作前：看到有用内容时，能否快速收进来？
- 写作中：需要人物、场景或知识参考时，能否快速找到并使用？
- 写作后：能否知道素材是否用过、用在什么项目或章节？

### 1.1 本次重做决策

本次不在旧文件模型上继续叠加字段，直接建立新的结构化素材库：

- 删除旧素材业务代码，不维护新旧两套接口。
- 不扫描、不转换 `resources/<分类>/<文件>`，不实现迁移脚本或兼容读取。
- 新实现使用独立的 `materials/` 存储根目录，与旧 `resources/` 彻底隔离。
- 应用不再展示或搜索旧素材文件。
- 不自动删除用户磁盘上的旧 `resources/` 文件，避免升级时发生静默数据丢失；确认无保留价值后由开发者或用户自行清理。
- 清理项目向量库中的旧 `resources/` 页面记录，避免全局搜索继续返回已经不可访问的旧素材。

这里的“删除旧代码”不包括通用基础设施。项目现有的素材 Tab 路由、写作侧栏位置、TipTap 编辑器、`Button`、`Modal`、`Pagination` 和通用搜索结果 UI 可以继续复用。

## 二、产品定位与边界

### 2.1 素材与其他模块的边界

| 模块 | 主要用途 | 典型内容 |
|---|---|---|
| 素材 | 外部参考、灵感原料、待消化内容 | 历史资料、人物原型、场景参考、对白片段、网页摘录 |
| 备注 | 写作过程中的临时想法和待办 | “第 6 章节奏太快”“补一段主角反应” |
| 世界观 | 当前作品中已经确定的事实 | 地理、势力、规则、力量体系 |
| 大纲 | 故事将如何展开 | 总纲、分卷规划、章节细纲 |

素材可以被吸收到设定、大纲或正文中，但不直接替代这些模块。

### 2.2 核心原则

1. **文字优先**：小说创作的核心素材是可阅读、可检索、可摘录的文字。
2. **先收集，后整理**：默认提供“收件箱”，用户不分类也能立即保存。
3. **预设可用，结构可改**：提供小说常用类型和分类，类型名称、分类层级和标签都可以调整，并可一键补回缺失的初始预设。
4. **全局复用，项目可控**：素材可以全局共享，也可以只关联指定项目，两种状态必须显式区分。
5. **保留来源**：素材正文、用户摘要和外部来源信息必须区分，避免后续无法追溯。
6. **AI 只辅助整理和调用**：AI 可以摘要、推荐类型和标签、查找相关素材，但不能擅自覆盖原始素材。
7. **单一数据源**：素材 Tab、写作侧栏、全局搜索和 AI 上下文必须使用同一套素材 ID 和存储接口。

### 2.3 明确不做

- 不做类似 Notion 的通用文档系统。
- 不做图片编辑、图片生成和图库管理。
- 不做音频、视频素材管理。
- 不做复杂的多人协作、云同步和版权管理系统。
- 不在素材库内维护角色、世界观和大纲的第二套数据。
- 不迁移、不兼容读取旧 `resources/` 素材。
- 不在第三阶段加入图片多模态 AI；图片先只做人工参考。
- 不为少量素材过早引入语义检索和自动推荐。

## 三、目标数据模型与存储

### 3.1 核心数据结构

素材不能继续由“目录名 + 文件名”隐式表达业务信息。所有业务对象使用稳定 ID，显示名称和文件路径都不能充当关联键。

```ts
type MaterialContentFormat = 'plain_text' | 'markdown'
type MaterialScope = 'global' | 'projects'

type MaterialSourceType =
  | 'original'      // 用户原创或手动整理
  | 'book'          // 书籍或纸质资料
  | 'web'           // 网页
  | 'file'          // 导入文件
  | 'image'         // 图片参考

interface MaterialItem {
  schemaVersion: 1
  id: string
  title: string
  kindId: string
  content: string
  contentFormat: MaterialContentFormat
  summary: string
  sourceType: MaterialSourceType
  sourceName: string
  sourceUrl: string
  categoryId: string
  tags: string[]
  scope: MaterialScope
  projectIds: string[]
  favorite: boolean
  attachmentIds: string[]
  sourceDocumentId?: string
  sourceSectionId?: string
  sourceLocator?: string
  createdAt: string
  updatedAt: string
}

interface MaterialKindDefinition {
  id: string
  name: string
  order: number
  presetKey: string | null
  archived: boolean
}

interface MaterialCategory {
  id: string
  name: string
  parentId: string | null
  order: number
  systemKey: 'inbox' | null
}

interface MaterialAttachment {
  id: string
  ownerType: 'material' | 'document'
  ownerId: string
  originalName: string
  storedName: string
  mimeType: string
  size: number
  relativePath: string
  createdAt: string
}

type MaterialDocumentFormat = 'txt' | 'epub'

interface MaterialDocument {
  schemaVersion: 1
  id: string
  title: string
  author: string
  format: MaterialDocumentFormat
  attachmentId: string
  scope: MaterialScope
  projectIds: string[]
  sectionIds: string[]
  createdAt: string
  updatedAt: string
}

interface MaterialDocumentSection {
  id: string
  documentId: string
  order: number
  title: string
  relativePath: string
  characterCount: number
}

type MaterialUsageAction = 'insert' | 'ai_context'

interface MaterialUsage {
  id: string
  materialId: string
  action: MaterialUsageAction
  projectId: string
  volume: string
  chapterId: string
  chapterTitle: string
  excerpt: string
  createdAt: string
}
```

### 3.2 字段约定

- 所有 ID 使用 UUID，不接受标题、分类名或原始文件名作为路径参数。
- `content` 是素材主体；新建文字素材默认使用 `plain_text`。
- 导入 Markdown 时保留原文并标记为 `markdown`。预览可以渲染格式，但插入正文、关键词搜索和 AI 上下文统一使用转换后的纯文本。
- `summary` 是用户或 AI 整理后的说明，允许为空，不能覆盖 `content`。
- `kindId` 引用可编辑的类型配置，不使用固定 TypeScript 联合类型。
- `categoryId` 引用一个实际分类；“全部素材”和“收藏”是查询视图，不是分类记录。
- “收件箱”是唯一不可删除的系统分类。分类支持两级结构，`parentId` 为空表示一级分类；第一版不支持更深层级。
- 删除普通分类时，分类下素材移动到收件箱；删除父分类前必须先处理或一并移动其子分类。
- `scope='global'` 时 `projectIds` 必须为空，并在所有项目中可见。
- `scope='projects'` 时 `projectIds` 至少包含一个项目 ID，只在关联项目中可见。
- 在项目内新建素材时默认关联当前项目；用户可显式改为全局素材。
- 删除项目时，从素材的 `projectIds` 中移除该项目；如果项目范围素材失去全部关联，则转为全局素材，避免静默删除。
- 完整小说和长文本使用独立的 `MaterialDocument` 保存书籍元数据与章节结构，不把数 MB 全文写入单条 `MaterialItem.content`。
- `MaterialDocumentSection` 保存章节/分段元数据，正文按章节独立存储；资料源阅读和搜索按章节定位。
- 从资料源选文创建素材时复制实际摘录到 `MaterialItem.content`，同时用 `sourceDocumentId`、`sourceSectionId` 和 `sourceLocator` 保留可跳转来源。删除资料源不会删除已经创建的素材摘录，但来源跳转会失效并在界面中明确标示。
- 附件通过 `ownerType + ownerId` 明确归属于普通素材或资料源；一个附件不能同时归属多个业务对象。
- 附件元数据和使用记录独立保存，不能把不断增长的使用记录写回素材主文件。
- 删除素材时同步删除其使用记录；删除项目时保留使用记录中的章节标题快照，用于历史回溯。
- 时间统一保存为 UTC ISO 8601，界面按本地时区展示。

### 3.3 预设与自定义规则

首次使用创建以下素材类型预设：

- 灵感片段
- 情节桥段
- 人物参考
- 场景参考
- 对话表达
- 知识资料
- 其他

用户可以重命名、排序、新增或归档类型。归档类型不会删除已有素材，只是不再出现在新建素材的默认选择中。

“恢复初始预设”只补回被归档或缺失的预设，不覆盖用户已经修改的名称、排序和自定义类型。

### 3.4 存储结构

保持项目本地优先、文件可备份的特点，第一阶段使用独立 JSON 文件，不引入新的业务数据库：

```text
materials/
  items/
    <material-id>.json
  attachments/
    materials/
      <material-id>/
        <attachment-id>.<ext>
    documents/
      <document-id>/
        <attachment-id>.<ext>
  documents/
    items/
      <document-id>.json
    sections/
      <document-id>/
        <section-id>.txt
  usage/
    <usage-id>.json
  config/
    categories.json
    kinds.json
    cleanup-state.json
```

说明：

- `materials/` 与旧 `resources/` 完全隔离。
- `cleanup-state.json` 只记录旧代码和旧向量索引清理状态，不承担素材迁移。
- 每条素材和使用记录独立写入，避免单个大 JSON 文件损坏整个素材库。
- Rust 后端维护列表摘要和标准化搜索文本的内存索引，首次访问时从 `items/` 构建，CRUD 后同步更新；前端不能逐条调用读取接口拼装列表。
- 列表接口在后端完成筛选、排序和分页，只返回当前页摘要；详情接口按 ID 单独读取完整正文。
- 全文搜索查询内存索引中的标准化纯文本，不在每次搜索时重新读取所有 JSON，也不直接搜索原始 JSON 字符串。
- 长文本资料源不进入现有素材正文内存索引；第三阶段建立独立的章节级关键词索引，搜索结果必须定位到具体资料源和章节，不能只返回整本书。

### 3.5 后端接口边界

第一阶段至少提供以下结构化命令：

```text
list_materials(filter, page, page_size) -> MaterialPage
get_material(material_id) -> MaterialItem
create_material(input) -> MaterialItem
update_material(material_id, patch) -> MaterialItem
delete_material(material_id) -> void
list_material_categories() -> MaterialCategory[]
save_material_categories(categories) -> void
list_material_kinds() -> MaterialKindDefinition[]
save_material_kinds(kinds) -> void
search_materials(query, filter, limit) -> MaterialSearchResult[]
```

第三阶段再增加资料源预览、导入、章节读取、书内搜索、选文转素材、附件管理和资料源删除命令，第二阶段再增加使用记录命令。所有命令必须验证 UUID、规范化附件文件名，并确认最终路径位于 `materials/` 根目录内，禁止使用用户输入直接拼接路径。

## 四、分阶段实施计划

四个阶段按顺序实施。每一阶段完成后都能独立使用，并通过验收后再进入下一阶段。

```text
第一阶段：全新结构化文字素材库
    ↓
第二阶段：融入小说写作流程
    ↓
第三阶段：小说与长文本资料导入
    ↓
第四阶段：受控的智能整理与检索
```

---

### 第一阶段：全新结构化文字素材库

**目标**：删除旧业务实现，建立一套不依赖文件名和 Markdown 目录的素材管理闭环。

#### 4.1.1 删除旧实现

- 替换现有 `ResourcePanel` 的文件列表和 Markdown 文件编辑逻辑。
- 删除旧的 `list/write/read/delete_resource_file` 前后端业务接口。
- 删除旧的素材目录关键词搜索逻辑。
- 删除旧 `resourceIndexer` 及保存文件时自动写入项目 LanceDB 的逻辑。
- 删除写死分类的旧 AI 推荐实现；第四阶段基于用户真实类型和分类重新实现。
- 全局搜索不再接受旧 `resources/<分类>/<文件>` 结果。
- 对所有项目向量库执行 `page_id` 前缀为 `resources/` 的清理，并用 `cleanup-state.json` 保证重复执行安全。
- 旧 `resources/` 文件保留在磁盘，但新应用不读取、不展示、不搜索。

第一阶段结束时，代码中不能再存在依赖旧素材文件 API 的调用方。不能为了临时兼容保留双写或 fallback。

#### 4.1.2 素材 Tab 布局

采用三栏工作区，不使用装饰性卡片堆叠：

```text
左侧 220px            中间素材列表              右侧详情区
收件箱                搜索与筛选                标题、类型、来源
全部素材              素材行                    正文与摘要
收藏                  标题、摘要、标签          作用域、项目关联
自定义分类树          更新时间                  编辑、删除、收藏
```

- 根容器遵循项目完整 flex 高度传递链，不使用 `height: calc(...)`。
- 左侧分类栏固定为 220px；中间列表使用响应式约束宽度，右侧详情区自适应，避免窄窗口被固定三栏挤爆。
- 列表使用通用 `<Pagination>`，默认每页 20 条。
- 新建、类型与分类管理、删除确认等弹窗统一使用 `<Modal>`。
- 空状态提供明确的“新建素材”入口，不显示大段功能说明。

#### 4.1.3 基础操作

- 新建、编辑和删除文字素材。
- 直接粘贴文字并保存，不要求输入文件名或扩展名。
- 编辑标题、正文、摘要和来源信息。
- 收藏或取消收藏。
- 移动分类，增加或移除标签。
- 设置为全局素材，或关联一个或多个项目。
- 管理素材类型和两级分类，并可恢复初始类型预设。
- 删除素材、分类和类型前展示明确影响范围并二次确认。

#### 4.1.4 搜索、分页与跳转

- 按标题、正文纯文本、摘要、来源名称和标签进行本地关键词搜索。
- 支持按类型、分类、标签、收藏状态和当前项目可见性筛选。
- 多个筛选条件同时生效，切换条件时重置到第 1 页。
- 搜索和筛选在 Rust 后端执行，前端只接收分页结果。
- 全局搜索结果使用素材 ID 作为路径标识。
- `ProjectView` 保存待打开的素材 ID，切换到素材 Tab 后由 `ResourcePanel` 自动选中并定位，不能只切换 Tab 不打开结果。
- 第一阶段不写入新的素材向量，不返回旧素材向量结果。

#### 4.1.5 写作侧栏最低可用适配

旧素材 API 删除后，写作侧栏不能处于损坏状态。第一阶段同步完成最低适配：

- 使用新列表和详情接口读取当前项目可见素材。
- 支持选择素材并插入完整纯文本。
- 不再截断为 500 个字符。
- 暂不加入高级筛选、选区摘录、选文收集和 AI 上下文，这些由第二阶段完成。

#### 4.1.6 第一阶段验收标准

- 用户可以完全在应用内新建、编辑、分类、搜索和删除素材。
- 新建素材不要求理解 Markdown 文件、目录和扩展名。
- 类型、分类、标签和项目作用域均使用稳定 ID，不依赖显示名称。
- 素材 Tab、最低可用写作侧栏和全局搜索使用同一份新素材数据。
- 搜索结果可以准确打开对应素材，删除素材后搜索结果立即消失。
- 旧 `resources/` 文件不会出现在新素材 Tab、写作侧栏或搜索结果中。
- 旧 `resources/` 向量记录清理可重复执行，不影响角色、章节等其他向量数据。
- 使用 1000 条素材、每条正文平均 5KB 的测试数据：冷启动首次列表加载不超过 2 秒，预热后的筛选或关键词搜索不超过 500ms，分页交互不超过 100ms。性能数据在同一参考机器上连续测量 10 次并记录。
- 未选择素材、列表为空、读取失败和保存失败都有明确状态。
- 素材 Tab 在常用桌面尺寸下无整体页面滚动、文字遮挡和布局溢出。

### 第二阶段：融入小说写作流程

**目标**：素材不只是在独立 Tab 中保存，而是能在写作时快速找到、引用和沉淀。

#### 4.2.1 跨组件状态契约

在实现 UI 前先确定共享状态：

```ts
interface CurrentChapterRef {
  projectId: string
  volume: string
  chapterId: string
  chapterTitle: string
}

interface MaterialContextSelection {
  materialId: string
  title: string
  excerpt: string
}
```

- `ProjectView` 持有 `CurrentChapterRef` 和当前章的 `MaterialContextSelection[]`，保证素材 Tab 与写作 Tab 使用同一状态。

#### 4.3.1 依赖验证门槛

第三阶段涉及新增 Rust 依赖，开始编码前必须先形成并确认具体依赖子方案，然后实际添加依赖、编译和运行最小样例。不得仅凭估计跳过主方案，也不得在依赖失败后自行切换 fallback。

当前计划的主实现方向：

- EPUB：使用 ZIP + XML/XHTML 解析方案读取 OPF 元数据、阅读顺序和目录，并对压缩包路径与解压大小做安全限制。
- 网页请求：`reqwest` + `url`，使用 Rustls，避免额外系统 OpenSSL 依赖。

依赖子方案需要确认具体 crate 版本、许可证、维护状态、Windows 构建结果和打包体积。任何主依赖实际编译或解析失败时，先展示失败证据，由用户决定修复还是另选方案。

#### 4.3.2 MOBI 技术调研结论与范围决策

2026-07-17 在 Windows x86_64 MSVC、Rust 1.96 环境中使用公开测试样本进行了最小探针，覆盖中文/Unicode、CP1252、无压缩、PalmDOC、Huff/CDIC、NCX 目录和 DRM 标记。结果如下：

- `mobi 0.8.0`（MIT）可以读取常见元数据、压缩类型和加密类型，但没有直接可用的目录/章节 API。中文和 Unicode 无压缩样本因 UTF-8 多字节字符跨 4096 字节记录边界而严格解码失败；宽松解码分别产生 136 和 107 个替换字符。Huff/CDIC 公开样本触发整数下溢 panic。
- `iepub 1.3.6`（MIT）在一个中文样本上成功解析出 24 个目录和章节，但 CP1252 PalmDOC 样本因源 HTML 标签不规范而整体失败；Huff/CDIC 没有被正确解压，随后发生 UTF-8 解码失败；DRM v2 样本触发整数下溢 panic。
- `mobi-sys 0.1.2` 只是 `libmobi` 的底层 FFI 绑定，要求系统预装 C 库与 Clang，并引入额外的 Windows 构建、签名、打包和分发责任。考虑产品主线和维护成本，在完成正式编译探针前终止该方向。

调研结论：当前没有满足“现成、低维护、中文可靠、目录可用、面对异常文件不 panic、Windows/Tauri 易分发”要求的 MOBI 方案。项目核心是小说创作，不承担 MOBI 解析器研发和广泛格式兼容，因此作出以下范围决策：

- 第三阶段暂不支持 `.mobi`，文件选择器不把 MOBI 列为可导入格式，后端收到 MOBI 文件时返回明确的“不支持该格式”错误。
- 不复制或修补第三方解析器源码，不自研 PalmDOC/Huff/CDIC/目录解析，不捆绑 Calibre 等外部转换工具，也不为此引入需要单独维护的 C 运行时链路。
- AZW、AZW3、KFX 和任何 DRM 内容继续明确不支持，不提供解密或格式转换。
- 后续只有出现维护活跃、许可证兼容、可直接集成并通过上述样本与 Windows 发布构建的成熟方案时，才重新评估 MOBI；重新支持前必须单独更新计划并获得确认。

#### 4.3.3 长文本资料源与素材条目分层

- 整本小说作为 `MaterialDocument` 资料源保存，原始文件作为附件保留；资料源不是一条可以“插入全文”或整体加入 AI 上下文的普通素材。
- 导入后保存书名、作者、格式、目录和章节顺序；TXT 无可靠目录时使用章节标题识别，并允许用户在导入确认界面选择“按识别章节导入”或“作为单篇长文本导入”。
- 资料源正文按章节/分段独立保存和读取，详情区提供目录、章节阅读和书内关键词搜索；搜索结果直接定位到具体章节和命中片段。
- 用户选中章节正文后可以“摘为素材”，自动复制选文，并记录资料源、章节和位置；用户确认标题、类型、分类、标签和作用域后再保存。
- 已摘出的素材继续使用第二阶段既有的插入、使用记录和 AI 上下文流程。整本书不能默认插入正文或加入 AI 上下文。
- 素材 Tab 增加“素材条目 / 资料源”视图切换，二者共用项目作用域，但列表、详情和搜索结果保持明确区分。

#### 4.3.4 电子书格式范围

按以下顺序独立交付和验收：

1. TXT：检测 UTF-8、GB18030/GBK、UTF-16 等常见编码，预览识别结果并支持章节标题识别。
2. EPUB：读取书名、作者、目录、spine 阅读顺序和 XHTML 正文，安全转换为纯文本并保留原始文件。

#### 4.3.5 网页与补充来源

- 用户可以保存网页地址、页面标题、来源站点和摘录正文。
- Tauri 后端请求网页并提取主要正文，避免前端跨域限制。
- 只允许 `http` 和 `https`；拒绝 `file:` 等本地协议。
- 请求超时 15 秒，最多跟随 5 次重定向，响应正文最大 5MB，只处理 HTML 和纯文本 Content-Type。
- 抓取和解析结果进入编辑确认界面，用户确认后才保存。
- 原始网址始终保留；抓取失败时展示明确错误，不创建空素材。
- 不做浏览器扩展、后台订阅和自动刷新网页内容。
- Markdown 仍可作为普通文字素材导入，保留原文并标记 `contentFormat='markdown'`，不进入长文本资料源主流程。
- DOCX 和 PDF 不属于第三阶段核心范围，也不作为本阶段验收条件；完成 TXT、EPUB 的真实使用验证后，再根据用户需求单独决定是否增加。

#### 4.3.6 图片参考

图片只作为小说创作的辅助参考，不作为素材库主轴：

- 支持 JPG、PNG、WebP 附件。
- 图片素材必须有标题，并允许填写文字说明、来源和标签。
- 详情区显示图片预览；列表以标题和文字说明为主。
- 普通搜索只检索标题、说明、来源和标签，不分析图片内容。
- 图片不能直接插入小说正文，也不加入 AI 多模态上下文。

图片参考与电子书导入独立交付，不能阻塞 TXT 和 EPUB 的核心验收。

#### 4.3.7 附件与资料源生命周期

- 文件选择复用项目已有 Tauri 对话框能力。
- 附件先复制到同目录临时文件，解析、章节写入和业务元数据全部成功后再提交最终文件名；任一步骤失败都回滚本次临时产物。
- 删除带附件的素材或资料源必须使用通用确认弹窗，并显示附件数量、章节数量和对已有摘录的影响。
- 删除资料源时同步删除原始附件、章节正文和独立关键词索引；已经摘为普通素材的内容保留，不随资料源删除。
- 删除素材或资料源后的文件清理失败时记录可重试状态，不能留下无提示的孤立文件。
- `relativePath` 由后端生成，前端不能提交任意磁盘路径作为存储位置。

#### 4.3.8 第三阶段验收标准

- TXT 和 EPUB 都能经过预览确认后导入，目录和章节阅读顺序正确。
- 书内搜索结果可以定位到具体章节和命中片段，不能只定位到整本书。
- 可以从资料源选文创建普通素材，并保留书名、章节和稳定来源引用；后续插入和 AI 上下文只使用明确摘出的内容。
- MOBI、AZW、AZW3、KFX 和 DRM 内容都被明确拒绝；损坏、空白和超限文件同样不能创建空资料源、章节残留或孤立附件。
- TXT 和 EPUB 分别具有成功、特殊编码/结构、损坏文件、超限文件和取消导入测试。
- 删除资料源时能同步处理原始附件、章节和关键词索引，并保留已经摘出的普通素材；删除弹窗明确告知影响。
- 图片有明确文字说明和来源，不把界面变成无组织的缩略图库。
- 所有外部素材都可以追溯到网址、原始附件或已经保留的来源快照。
- 新增依赖通过 Windows 开发构建和发布构建。

### 第四阶段：受控的智能整理与检索

**目标**：当素材数量增长后，降低整理和查找成本；所有 AI 结果都由用户确认后写入。

#### 4.4.1 重建 AI 辅助整理

旧的写死分类建议已在第一阶段删除，本阶段基于新数据模型重新实现：

- 根据正文生成可编辑摘要。
- 从当前已有标签中推荐标签，不直接创建大量新标签。
- 提示词动态传入用户当前可用的素材类型和分类，不能写死预设名称。
- 推荐结果包含目标 ID 和显示名称，保存前校验目标仍然存在。
- 对长文档提取与小说写作相关的事实、描写和可用片段。
- 所有结果都先预览；批量操作必须提供逐条变更清单，用户确认后保存。

#### 4.4.2 相关素材推荐

- 在当前章节侧栏中，根据章节正文推荐少量相关素材。
- 推荐结果与用户主动加入上下文严格分开，推荐不等于自动注入。
- 支持“不相关”反馈，只影响当前项目内的后续排序。
- 默认最多展示 5 条，避免干扰写作。

#### 4.4.3 语义搜索决策门槛

第一阶段已经移除旧素材向量链路。只有同时满足以下条件后，才重新接入项目现有 LanceDB：

- 真实素材数量和使用反馈证明关键词搜索不足。
- 素材正文、纯文本转换、摘要和项目作用域已经稳定。
- 可以使用 `materials/<material-id>` 作为稳定 `page_id`。
- 全局素材能够按关联项目建立或删除项目级向量记录。
- 素材删除、取消项目关联和重新生成 embedding 都有索引清理测试。

如果条件不满足，第四阶段只交付 AI 辅助整理，不实施语义搜索。该决策不是 fallback；进入第四阶段时需要根据实际数据单独确认。

#### 4.4.4 第四阶段验收标准

- AI 生成的摘要、类型和标签在保存前都可修改或拒绝。
- AI 使用用户当前实际类型和分类，不输出不存在的硬编码选项。
- AI 不覆盖原始正文和来源。
- 推荐素材不会自动加入提示词或修改章节。
- 关闭智能功能后，前三阶段全部功能仍然可用。
- 若启用语义搜索，必须有删除、取消关联、重建和关键词/向量混合排序测试。

## 五、实施顺序与工作量

| 阶段 | 前置依赖 | 主要交付 | 建议工作量 |
|---|---|---|---|
| 第一阶段 | 无 | 删除旧实现、新数据层、三栏素材 Tab、CRUD、自定义类型与分类、结构化搜索、最低可用侧栏 | 7-10 个开发日 |
| 第二阶段 | 第一阶段数据结构稳定 | 写作侧栏完善、选文收集、插入、AI 上下文、使用记录 | 5-7 个开发日 |

| 第四阶段 | 前三阶段形成真实素材数据 | AI 整理、相关推荐、按决策门槛接入语义搜索 | 5-8 个开发日 |

工作量为单人开发、自动化测试和基础手工验证的估算，不包含大规模格式兼容测试。每一阶段单独验收，不建议四个阶段连续一次性实现。

推荐执行顺序：

3. 使用基准数据验证列表和搜索性能，再验收第一阶段。
4. 第二阶段先建立跨 Tab 章节状态和素材上下文状态，再做选文收集和 AI 注入。
5. 第三阶段先确认长文本资料源模型，再实际编译 EPUB 依赖最小样例；按 TXT、EPUB 顺序完成核心交付和独立验收，之后再交付网页、Markdown 和图片补充来源。MOBI 不进入本阶段开发。
6. 第四阶段根据真实素材规模决定是否重新接入 LanceDB，不能仅因为技术上可做就默认加入。

## 七、测试、性能与数据安全

### 7.1 第一、二阶段必测场景

- 新建、编辑、删除、收藏、类型、分类、标签和项目作用域。
- 标题重复、空标题、超长正文、特殊字符和中文内容。
- 类型归档、恢复预设、分类删除和子分类处理。
- 项目删除后素材作用域转换。
- 多条件筛选、后端分页、搜索结果定位和删除后搜索失效。
- 旧素材 API 不再被调用，旧文件不再被读取，旧向量结果不再出现。
- 写作侧栏插入全文、插入选文、章节选文存素材和光标恢复。
- 素材上下文切换章节后的清理、单独预算和全局预算上限。
- 使用记录只在实际插入或实际 AI 生成后创建。

### 7.2 第三阶段必测场景

- 网页协议、超时、重定向、响应大小和 Content-Type 限制。
- TXT 的 UTF-8、GB18030/GBK、UTF-16 编码、无章节标题、误识别章节和单篇导入。
- EPUB 的 OPF 元数据、TOC/spine 顺序、嵌套路径、缺失目录、异常 XHTML、路径穿越和解压体积限制。
- MOBI、AZW、AZW3、KFX 和 DRM 内容的拒绝路径，错误信息必须明确且不能产生任何业务记录或附件残留。
- 长文本章节阅读、书内搜索定位、选文转素材、来源回跳，以及资料源删除后摘录保留和失效来源提示。
- 损坏、加密、空白和超限文件；附件导入取消、保存失败、资料源删除和孤立附件重试清理。
- 图片格式、尺寸限制、预览失败和来源信息。

### 7.3 数据安全

- 新素材命令只接受 UUID 和结构化字段，不能接受任意目录作为业务参数。
- 所有最终路径都由后端生成并验证仍位于 `materials/` 根目录。
- EPUB 等归档格式必须限制压缩包条目数量、单条目大小、总解压大小和路径穿越；电子书解析不能向 `materials/` 之外写入文件。
- JSON 和附件写入使用同目录临时文件、刷新落盘和平台安全替换；该写入助手必须在 Windows 上测试覆盖替换失败和中途退出场景。
- 批量配置修改和索引清理执行前保留可恢复状态，失败时报告具体步骤，不静默走 fallback。
- 素材 ID 一经生成，不随标题、分类或附件变化。
- 删除素材和附件必须使用通用确认弹窗，不能静默删除用户文件。
- 网页和文档解析内容视为不可信输入，只以纯文本或经过安全处理的 Markdown 展示，不执行原始 HTML。
- API 错误对用户显示可理解信息，同时保留足够的开发日志；不能只写 `console.error` 后静默失败。

## 八、已确定决策与阶段决策点

### 8.1 已确定决策

| 决策 | 结论 |
|---|---|
| 旧素材代码 | 删除，不保留兼容接口 |
| 旧素材文件 | 不迁移、不读取、不自动删除 |
| 新存储根目录 | `materials/`，与旧 `resources/` 隔离 |
| 存储方式 | 独立 JSON + 后端内存索引，不新增业务数据库 |
| 素材作用域 | 显式区分全局和指定项目 |
| 素材类型 | 稳定 ID、预设可恢复、名称和顺序可编辑 |
| 内容格式 | 新建默认纯文本，Markdown 导入保留格式并统一转纯文本调用 |
| 长文本模型 | 整本小说作为独立资料源按章节存储；只有用户明确选取的摘录才成为普通素材并进入写作流程 |
| 核心导入格式 | TXT、EPUB；两种格式独立交付和验收 |
| MOBI | 已完成最小技术探针；当前 Rust 方案在中文解码、PalmDOC/Huff、目录或异常文件安全性上不满足要求，暂缓支持且不自研解析器 |
| 明确不支持 | MOBI、AZW、AZW3、KFX 和任何 DRM 内容；不提供解密或格式转换 |
| DOCX/PDF | 不属于第三阶段核心范围，完成 TXT/EPUB 验证后再根据真实需求单独决策 |
| 图片 | 第三阶段只做人眼参考，不进入多模态 AI |
| 旧素材向量 | 第一阶段清理并停止写入，不留旧结果 |

### 8.2 阶段决策点

| 阶段结束 | 需要确认的问题 |
|---|---|
| 第一阶段 | 全新结构化文字素材是否已经适合日常收集、管理和查找？ |
| 第二阶段 | 素材是否真正进入章节写作和 AI 辅助流程，而不是孤立资料页？ |
| 第三阶段 | TXT 和 EPUB 是否已经覆盖主要小说资料来源，资料源分章查阅与摘录流程是否适合真实写作？ |
| 第四阶段 | 素材规模和真实反馈是否证明需要语义检索？ |

如果前一阶段的核心问题没有得到肯定答案，应先修正该阶段，不进入下一阶段。执行中若发现已确认方案不可行，必须暂停、展示失败证据并提出修正方案，得到用户确认后再继续。
