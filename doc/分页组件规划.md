# 分页组件规划

> 创建日期：2026-07-14
> 状态：已完成

---

## 一、背景

当前项目中仅 NotesPanel 有分页（ad-hoc 内联实现），其余所有面板的列表都是全量渲染。实际使用中数据量很容易突破几十上百条，全量渲染导致：

- 列表过长，滚动体验差
- 首次渲染慢（大量 DOM 节点）
- 每个面板各自写一套分页逻辑，重复且不一致

### 1.1 当前分页实现（NotesPanel）

```typescript
// NotesPanel.tsx — 唯一的分页，ad-hoc 实现

const PAGE_SIZE = 20                        // 硬编码常量
const [page, setPage] = useState(1)          // 页码状态

const totalPages = Math.max(1, Math.ceil(filtered.length / PAGE_SIZE))
const safePage = Math.min(page, totalPages)
const paged = filtered.slice((safePage - 1) * PAGE_SIZE, safePage * PAGE_SIZE)

// 过滤/视图切换时手动重置
const handleFilterChange = (f) => { setFilter(f); setPage(1) }

// 🐛 已知 bug：只有 totalPages > 1 时才渲染分页栏
{totalPages > 1 && (
  <div className="notes-pagination">
    <button disabled={safePage <= 1}>← 上一页</button>
    <span>{safePage} / {totalPages}</span>
    <button disabled={safePage >= totalPages}>下一页 →</button>
  </div>
)}
```

### 1.2 问题清单

| # | 问题 | 严重度 | 影响 |
|---|------|:---:|------|
| 1 | 不到 20 条时分页栏消失 | 🟡 S2 | 用户不知道数据已全部展示、也不确定是否有分页功能 |
| 2 | 分页逻辑散落在业务组件内 | 🟡 S2 | 每个需要分页的面板都要重复实现 PAGE_SIZE / safePage / paged slice |
| 3 | 过滤切换时手动 `setPage(1)` | 🟢 S3 | 容易遗漏，新面板忘了手动重置会导致越界 |
| 4 | 无复用组件 | 🟡 S2 | 18 个面板有列表，1 个有分页，其余裸奔 |

## 二、设计方案

### 2.1 组件拆分为两件套

```
src/hooks/usePagination.ts    ← 逻辑 hook：slice + 页码管理（新目录）
src/components/Pagination.tsx  ← 纯 UI 组件：按钮 + 页码显示
src/style.css                  ← 合入 .pagination 样式
```

> `src/hooks/` 为新建目录。和 `src/services/` 分层：hooks 管 React 状态逻辑，services 管数据层 + Tauri API。

#### `usePagination<T>(items, pageSize)` — 逻辑 Hook

```typescript
// src/hooks/usePagination.ts

import { useState, useMemo, useCallback } from 'react'

interface PaginationResult<T> {
  /** 当前页码（1-based） */
  page: number
  /** 设置页码 */
  setPage: (page: number) => void
  /** 总页数 */
  totalPages: number
  /** 当前页数据切片 */
  paged: T[]
  /** 手动重置到第一页（在过滤/搜索条件变化或 pageSize 变化时调用） */
  reset: () => void
}

export function usePagination<T>(
  items: T[],
  pageSize: number = 15,
): PaginationResult<T> {
  const [page, setPage] = useState(1)

  const totalPages = useMemo(
    () => Math.max(1, Math.ceil(items.length / pageSize)),
    [items.length, pageSize],
  )

  const safePage = Math.min(page, totalPages)

  const paged = useMemo(
    () => items.slice((safePage - 1) * pageSize, safePage * pageSize),
    [items, safePage, pageSize],
  )

  const reset = useCallback(() => setPage(1), [])

  return { page, setPage, totalPages, paged, reset }
}
```

**设计要点：**

- **不自动重置** — `reset()` 由调用方在过滤/搜索条件变化时显式调用。数据刷新（新增/编辑/删除）不跳页，维持当前页码
- **pageSize 默认 15** — 兜底值，面板可通过 hook 参数覆盖
- 如果过滤后当前页码超出范围，`safePage = Math.min(page, totalPages)` 自动 clamp 到有效页
- `useMemo` 避免不必要的 slice 计算

#### `<Pagination>` — UI 组件

```typescript
// src/components/Pagination.tsx

interface PaginationProps {
  /** 当前页码（1-based） */
  currentPage: number
  /** 总页数 */
  totalPages: number
  /** 总条数（未分页前，用于显示"共 N 条"） */
  totalItems: number
  /** 当前每页条数 */
  pageSize?: number
  /** 可选条数列表（不传则不显示下拉框） */
  pageSizeOptions?: number[]
  /** 页码变化回调 */
  onPageChange: (page: number) => void
  /** 每页条数变化回调（用户切换下拉框时触发） */
  onPageSizeChange?: (pageSize: number) => void
}

export default function Pagination({
  currentPage,
  totalPages,
  totalItems,
  pageSize,
  pageSizeOptions,
  onPageChange,
  onPageSizeChange,
}: PaginationProps) {
  if (totalItems === 0) return null  // 空数据不显示

  return (
    <div className="pagination">
      <button
        className="pagination-btn"
        disabled={currentPage <= 1}
        onClick={() => onPageChange(currentPage - 1)}
      >
        ← 上一页
      </button>
      <span className="pagination-info">
        第 {currentPage} / {totalPages} 页，共 {totalItems} 条
      </span>
      {onPageSizeChange && pageSize && pageSizeOptions && (
        <span className="pagination-size">
          每页
          <select
            className="pagination-size-select"
            value={pageSize}
            onChange={(e) => onPageSizeChange(Number(e.target.value))}
          >
            {pageSizeOptions.map((n) => (
              <option key={n} value={n}>{n}</option>
            ))}
          </select>
          条
        </span>
      )}
      <button
        className="pagination-btn"
        disabled={currentPage >= totalPages}
        onClick={() => onPageChange(currentPage + 1)}
      >
        下一页 →
      </button>
    </div>
  )
}
```

**设计决策：**

| 决策 | 理由 |
|------|------|
| `totalItems === 0` 返回 `null` | 空数据时不占空间 |
| 始终显示（即使 1 页） | 修掉 NotesPanel 的 "<20条不显示" bug；用户需要知道"这是全部数据" |
| 显示"共 N 条" | 让用户感知总量，判断是否需要搜索/过滤 |
| `onPageChange` 而非双向绑定 | 单向数据流，hook 持有页码状态 |
| 翻页时列表滚动到顶部 | `onPageChange` 触发后由调用方将列表容器 `scrollTo(0, 0)` |
| 默认每页 15 条 | 兼顾信息密度和滚动体验的中间值 |
| `pageSize` 下拉框可选 | 不传 `onPageSizeChange` / `pageSizeOptions` 就不渲染，简单场景零额外 UI |

### 2.2 使用示例

```typescript
// 改前
const filtered = entries.filter(...)
filtered.map(renderItem)  // 全量渲染

// 改后
import { usePagination } from '../hooks/usePagination'
import Pagination from './Pagination'

const DEFAULT_PAGE_SIZE = 15

const filtered = useMemo(() => entries.filter(...), [entries, statusFilter, categoryFilter])
const [pageSize, setPageSize] = useState(DEFAULT_PAGE_SIZE)
const { paged, page, setPage, totalPages, reset } = usePagination(filtered, pageSize)

// 过滤条件变化时重置到第一页
const handleFilterChange = (f: string) => { setStatusFilter(f); reset() }

// 渲染
paged.map(renderItem)
<Pagination
  currentPage={page}
  totalPages={totalPages}
  totalItems={filtered.length}
  pageSize={pageSize}
  pageSizeOptions={[15, 30, 50]}
  onPageChange={(p) => {
    setPage(p)
    document.querySelector('.foreshadow-list')?.scrollTo(0, 0)
  }}
  onPageSizeChange={(ps) => { setPageSize(ps); reset() }}
/>
```

## 三、改造范围与优先级

### 3.1 判断原则

分页只用于**主内容区**的列表——数据是"被浏览的主体"，不是导航工具。以下情况**不分页**：

| 情况 | 理由 | 例子 |
|------|------|------|
| 左边栏列表 | 侧栏是导航/选择器，不是内容主体；分页会阻断导航 | ChapterManager 章节列表、CharacterPanel 角色列表、WorldviewPanel 世界观列表、MaterialSidebar 素材侧栏 |
| 树形/层级结构 | slice 无法正确处理嵌套；用户需要看到完整层级 | OutlinePanel 卷-章大纲 |
| 拖拽排序列表 | 分页切断跨页拖拽 | CharacterPanel 角色排序 |
| 图表/可视化 | 不是列表视图 | RelationshipGraph、ChapterGraph、StatisticsPanel |

### 3.2 全量面板分析

| # | 面板 | 数据量 | 需要分页 | 优先级 | 建议 pageSize | 备注 |
|---|------|:---:|:---:|:---:|:---:|------|
| 1 | **NotesPanel** | 10-100+ | ✅ | 🔴 P0 | 15 | 已有 ad-hoc 分页，替换为复用组件 |
| 2 | **ForeshadowPanel** | 50-200+ | ✅ | 🔴 P0 | 15 | 条目可展开（推进轨迹），不宜太多 |
| 3 | **ResourcePanel** | 100+ | ✅ | 🟡 P1 | 12 | 网格卡片布局，每行 3-4 张 |
| 4 | **SearchPanel** | 50-500+ | ✅ | 🟡 P1 | 15 | 搜索结果含摘要，紧凑列表 |
| 5 | ChapterManager | 50-500+ | ❌ | — | — | 左边栏 + 树形结构 |
| 6 | CharacterPanel | 20-200+ | ❌ | — | — | 左边栏 + 拖拽排序 |
| 7 | WorldviewPanel | 5-10 类别 | ❌ | — | — | 左边栏 + 类别固定 |
| 8 | OutlinePanel | 50-300+ | ❌ | — | — | 左边栏 + 树形层级 |
| 9 | MaterialSidebar | 50-100+ | ❌ | — | — | 左边栏（编辑器侧栏） |
| 10 | ReviewPanel | 10-50/章 | ❌ | — | — | 多个子列表嵌套，分页增加复杂度 |
| 11 | ReviewRulesEditor | 10-50 | ❌ | — | — | 编辑视图，列表短 |
| 12 | VersionHistoryPanel | 10-30/章 | ❌ | — | — | 单章范围，通常不超一页 |
| 13 | BrainstormPanel | ≤20 | ❌ | — | — | AI 输出有限 |

| 15 | StatisticsPanel | ≤60 | ❌ | — | — | 图表数据，不适合分页 |

| 17 | RelationshipGraph | ≤200 | ❌ | — | — | 力导向图，非列表视图 |

**总结：18 个面板中，仅 4 个需要分页。**

## 六、边界情况

1. **空数据集**：`totalItems === 0` 时不渲染 `<Pagination>`（返回 null）
2. **数据恰好整除 pageSize**：`Math.ceil(N / 15)` 页数正确 → 最后一条数据不丢
3. **当前页超出范围**：`safePage = Math.min(page, totalPages)` 防止越界。过滤后数据减少导致当前页超出时，自动 clamp
4. **过滤时数据急剧减少**：过滤变化时调用方显式 `reset()`，回到第一页
5. **数据刷新（新增/编辑/删除）不跳页**：Hook 不自动重置，仅 `safePage` clamp 保证不越界
6. **pageSize > 数据总量**：只有 1 页，正常显示分页栏（上一页/下一页 disabled）
7. **切换 pageSize**：`onPageSizeChange` 回调中调 `reset()` 回到首页，避免当前页超出范围
8. **快速连续点击**：React 18 自动批处理，无竞态问题
9. **翻页时列表位置**：`onPageChange` 回调中由调用方将列表滚动容器 `scrollTo(0, 0)`

## 七、与伏笔模块的关系

分页组件独立于伏笔模块，但 ForeshadowPanel 是第一批接入的面板。两个改造互不阻塞：

- 伏笔 Phase 1 修 bug → 已完成
- 分页 Step 1-2 → 先做
- 分页 Step 3 → 伏笔加分页
- 伏笔 Phase 2 四级分桶 → 后续

分页改动极小（加一个 hook + 替换 map 为 paged.map + 底部放组件），不会影响伏笔的后续 Phase 2 改造。
