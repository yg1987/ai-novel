## 一、审查结论

角色 Tab 已具备角色卡的完整闭环：创建空白角色、随机起名、AI 创建、编辑保存、删除、排序、项目级提示词、选区 AI 改写，以及从搜索和关系图跳转到角色。Phase 1–5 已完成角色名安全、未保存保护、异步读取一致性、AI 创建/补全语义收敛、百万级命名与性别管理，以及结构化资料、稳定 ID、组织归属、关系历史、图谱编辑和跨模块联动。

## 二、现有模型与边界

### 2.1 存储模型

| 数据 | 当前存储 | 说明 |
|---|---|---|
| 角色卡 | `characters/<角色名>.md` | Markdown 正文即角色资料 |
| 排序 | `characters/order.json` | 保存侧栏角色展示顺序 |
| AI 提示词 | `memory/ai-prompts.json` | 项目级 `character_create` 提示词 |

新角色卡采用以下字段约定：

```markdown
角色：林烬
性别：男
身份/职业：
外貌特征：
性格特点：
背景经历：
动机目标：
说话风格：
标签：[标签1, 标签2, ...]
```

`性别` 取值为 `男`、`女`、`未知`。Markdown 仍是角色资料的主要内容载体：新手可按示例逐行填写，熟练用户可自由扩展其他字段。

**已确认的数据策略：不批量改写旧角色卡正文。** 性别字段仍不提供自动猜测或正文迁移；已有不含 `性别` 的角色卡按 `未知` 处理。Phase 4 为实现稳定角色 ID，允许在首次加载时根据现有 `characters/*.md` 非破坏性生成结构化目录和别名索引，但不得借此重写角色 Markdown、章节正文或删除用户数据。结构化目录初始化失败时必须保留原文件并显示错误，不得用空目录覆盖。

### 2.2 已验证的非问题

- 角色 Tab 使用 `.panel-layout`、`.panel-sidebar`、`.panel-editor`，高度传递、左右分栏和独立滚动满足项目布局约定。
- 角色排序可持久化到 `order.json`；新建、删除后会与实际 Markdown 文件合并。
- 关系图已能从角色卡文本中提取“身份、标签、阵营/势力/组织”等信息，具备基础联动，但目前仍依赖用户自行输入一致的自由文本。

## 三、已确认问题

### 3.1 P0：切换、导航和删除会静默丢失未保存编辑

**现状**

- 角色编辑只用 `editing` 区分阅读和编辑，没有“脏数据”状态或初始内容快照。
- 侧栏切换角色时直接切换当前文件并退出编辑。
- 从角色 Tab 切换到其他 Tab 时，`ProjectView` 只对世界观模块执行未保存变更拦截，角色组件会直接卸载。
- 删除按钮立即执行删除；删除当前角色时，内存中未保存的内容不会提示、也不会保留。

**影响**

- 用户写完角色设定后误点另一个角色、其他 Tab 或删除按钮，会丢失本次编辑。
- 数据丢失没有可撤销入口，风险高于一般体验问题。

**改进方向**

1. 在角色面板维护 `savedContent` 或等价快照，派生 `hasUnsavedChanges`。
2. 角色切换、Tab 切换、返回书架、删除当前角色前统一请求确认：保存、放弃、取消。
3. 提取通用的面板未保存变更协议，使 `ProjectView` 不再只特判世界观模块。
4. 删除确认对话框必须复用 `<Modal>`；文案明确说明角色卡会被删除且当前版本不可恢复。
5. 保存成功后更新快照；读取失败、保存失败时保留编辑内容且显示可见错误。

### 3.2 P0：角色名可能造成覆盖或越界文件路径

**现状**

- 前端重名判断仅使用精确字符串匹配。
- Windows 文件系统通常大小写不敏感，`Alice` 与 `alice` 可能指向同一文件；当前创建流程可能允许后者覆盖前者。
- 角色名直接拼接为 `<角色名>.md`。通用 `write_project_file` 与 `delete_project_file` 直接将文件名 `join` 到目录中，没有把普通写入/删除限制为单个相对文件名。

**影响**

- 英文名大小写差异可能导致角色资料被静默覆盖。
- 输入 `..`、路径分隔符或 Windows 非法文件名字符时，可能写入非预期路径或产生难以恢复的异常文件。

**改进方向**

1. 建立单一 `normalizeCharacterName()`：trim、Unicode NFC 归一化，并使用大小写不敏感的比较键检测重复。
2. 建立 `validateCharacterName()`：禁止空名、`.` / `..`、路径分隔符、控制字符及 Windows 非法字符；限制合理长度并给出中文错误提示。
3. 创建和 AI 输出的角色名都必须使用相同的规范化、校验和重名逻辑；随机起名也用同一比较键避重。
4. 后端将普通读/写/删的文件名限制为单个安全路径组件，复用或对齐 `safe_relative_path()` 的安全规则；不得只依赖前端校验。
5. 写入时若目标已存在，除明确的“保存当前角色”外一律拒绝覆盖；创建操作必须返回冲突提示。

### 3.3 P1：异步读取可能把旧角色内容显示在新角色上

**现状**

当前文件变化后异步读取 Markdown，完成时直接写入编辑器状态。用户快速从 A 点击到 B 时，A 的请求若后返回，可能覆盖 B 已加载或正在编辑的内容。

**影响**

- 界面标题为 B，编辑框却显示 A 的内容。
- 用户继续编辑并保存时，可能把 A 的内容写入 B，形成交叉覆盖。

**改进方向**

1. 每次加载创建递增请求序号，只有最后一次请求可以提交内容和错误状态。
2. 或在 `then/catch` 中核对当前活动角色和请求目标仍一致；组件卸载后不再写入状态。
3. 加入“加载中”状态，加载期间禁用保存和 AI 辅助，避免用户在未完成读取时操作。

### 3.4 P1：编辑器“AI 辅助”与实际行为不一致

**现状**

编辑器工具栏的“✨ AI 辅助”和侧栏“✨ AI 创建”复用同一创建函数。该函数读取侧栏的新角色名；在当前角色编辑器中通常为空，因此会让 AI 新起名字、直接写入一张新角色卡并切换过去，而不是补全当前角色。

**影响**

- 用户在当前角色中点击“AI 辅助”，预期是完善当前内容，实际却新建角色。
- 原角色的未保存内容可能随切换丢失。
- 自定义提示词的职责混合“创建新角色”和“补全当前角色”，难以写出稳定提示。

**改进方向**

1. 保留侧栏“AI 创建角色”，其输入为可选角色名，生成成功后先进入预览再创建文件。
2. 新增编辑器“AI 补全当前角色”：以 `activeFile`、现有内容和项目上下文为输入，不创建文件。
3. AI 返回内容先进入可对比的草稿/预览，用户选择覆盖、插入或取消后才写入编辑状态；不要直接落盘。
4. 分离 `character_create` 与 `character_complete` 提示词键，各自支持恢复默认和项目级自定义。

### 3.5 P1：文件操作错误和进行中状态对用户不可见

**现状**

新建、保存、删除、排序等失败大多仅输出到控制台；保存和删除也没有独立 loading 状态，用户可以重复点击。

**改进方向**

1. 为加载、新建、保存、删除、排序维护独立的进行中状态，并在相关按钮上禁用重复操作。
2. 将失败原因展示在角色面板的可见错误区域；成功操作提供简短反馈。
3. 排序保存失败时恢复上一次成功顺序，或明确标记“排序尚未保存”并允许重试，不能只在内存中悄悄变化。

### 3.6 P2：角色资料管理缺少规模化和引导式能力

**现状**

- 角色只有一列长列表，缺少搜索、标签筛选、阵营/身份分组和重命名。
- 关系图虽然会解析角色卡的部分字段，但角色页没有可见的关联关系、所属组织或相关伏笔入口；关系图本身也没有手动编辑能力。
- 角色专用样式中还残留关联伏笔、多选角色的规则，但组件没有对应 UI，说明该能力尚未形成完整功能。

**改进方向**

1. 在不改变 Markdown 主体自由度的前提下，提供可折叠的“结构化资料”区域：立场、标签、所属组织/势力、角色状态等。首次创建时给出可编辑预设，而不是强制必填。
2. 将“立场”和“所属组织”分开建模：立场用于主角/反派/中立等分组；所属组织允许多选，并预留职位、当前/曾经所属等扩展字段。
3. 为角色之间的关系提供独立的结构化记录：目标角色、关系类型、说明、状态、章节范围和来源。作者手动确认的关系优先于章节分析和共同出场推测，快照重建不得静默覆盖。
4. 在角色详情和关系图节点详情中提供关系查看、添加、编辑和删除入口；关系图继续负责发现和查看，复杂编辑以角色详情为主。
5. 在角色详情中展示并跳转到关系图节点、所属组织、关联伏笔和章节出现记录；没有数据时给出下一步引导。
6. 搜索、标签/立场/组织筛选作为资料管理的配套能力分阶段加入；长列表首阶段优先搜索和筛选，达到明确数量/性能门槛后再评估虚拟列表，不把分页组件硬套到角色侧栏。
7. 重命名独立设计安全契约：更新结构化引用和排序记录，但在未确认范围前不得静默批量替换正文自由文本。

### 3.7 P2：缺少角色模块的自动化测试

**现状**

当前已有 `characterNames.test.ts` 和 `characterProfiles.test.ts` 覆盖角色名与性别相关纯逻辑，但尚无结构化目录、组织、关系历史、关系图合并、重命名事务和角色面板交互测试。

**改进方向**

将可测试逻辑抽到服务或纯函数，优先覆盖高风险规则；组件交互采用定向测试验证，不以全项目验证替代本模块测试。

### 3.8 P2：随机姓名规模不足，群像剧难以持续复用

**现状**

当前随机起名依赖有限的姓氏和男女名字数组，组合数量约为一千余个。对于有上百名配角、路人和临时人物的群像剧，重名概率会快速升高，也无法按角色性别预先生成姓名。

**改进方向**

1. 不维护百万条静态全名；改用“姓氏 × 名字首段 × 名字尾段”的组合式生成，达到百万级以上理论组合量。
2. 姓氏库同时包含常见单姓和常用复姓；男女分别维护经过人工筛选的首段、尾段词库，避免纯随机汉字造成乱码。
3. 随机起名返回 `{ name, gender }`，创建新角色卡时直接写入 `性别：男` 或 `性别：女`。
4. 用户手动创建角色时默认填入 `性别：未知`，并可在编辑器中直接修改。

### 3.9 P2：角色性别缺少快速筛选和项目统计

**现状**

角色信息只有自由文本，没有稳定的性别字段。群像剧作者无法快速查看男女角色数量、筛选候选配角，或确认角色池是否失衡。

**改进方向**

1. 以角色卡首个 `性别：` 字段解析性别；只接受 `男`、`女`、`未知`，缺失或不合法值按新格式错误处理，不回退猜测。
2. 角色侧栏提供“全部 / 男 / 女 / 未知”快速筛选，并显示各分类数量；筛选不修改角色卡内容。
3. 统计模块增加“角色性别分布”卡片和图表，统计当前项目的男、女、未知数量。
4. 性别字段使用可编辑的选择控件写入角色卡，避免用户手写时产生词汇不一致。

## 四、实施顺序

### Phase 1：数据安全与未保存保护（P0）

1. 实现角色名归一化、校验、规范化重名检测。
2. 后端收紧普通项目文件读/写/删的路径安全校验，并确保现有合法角色卡兼容。
3. 实现角色编辑脏数据追踪与保存/放弃/取消确认。
4. 删除角色改为确认操作；删除当前角色时先处理未保存内容。
5. 补充文件名和未保存保护的定向测试。

**阶段验收**

- `Alice` 与 `alice`、非法文件名、路径字符都不能创建或覆盖资料。
- 切换角色、切换 Tab、返回书架、删除角色不会无提示丢失编辑。
- 所有新增弹窗都复用 `<Modal>`。

### Phase 2：加载一致性与操作反馈（P1）

1. 为角色读取加入最后请求胜出保护与加载状态。
2. 为新建、保存、删除、排序加入 loading、成功和失败反馈。
3. 排序落盘失败时可恢复或重试，避免仅内存排序与磁盘顺序不一致。

**阶段验收**

- 快速连续切换角色不会出现标题和内容不一致。
- 任一文件操作失败时，用户能看到原因并可继续处理。
- 操作进行中无法重复提交。

### Phase 3：拆分 AI 创建与 AI 补全（P1）

1. 将“AI 创建角色”限制在侧栏创建流程。
2. 新增“AI 补全当前角色”草稿预览流程，不直接创建文件或覆盖内容。
3. 分离创建、补全两类提示词和默认模板。
4. 覆盖 AI 无名称、重名、空返回、网络失败和用户取消的定向测试。

**阶段验收**

- 编辑器 AI 操作始终作用于当前角色，且未经确认不落盘。
- 创建与补全的提示词、按钮文案和结果去向一致。

### Phase 4：结构化资料与关系/阵营联动（P2，已实施）

**阶段目标**：在保留 Markdown 自由编辑的前提下，建立可验证的角色身份、组织归属和角色关系数据层；让关系图从“只读的运行时推导结果”变成“作者确认关系与章节推导证据并存的可编辑视图”；同时完成搜索、筛选、跨模块跳转、稳定 ID、重命名和删除闭环。本阶段列出的所有子项都属于本次实施范围，不再另设后续增强。

#### 4.1 先确定数据边界和文件契约

新增以下项目级文件，所有 JSON 均使用 `schemaVersion`、严格运行时校验和 `atomicWriteProjectFile` 写入；解析失败时保留原文件并显示错误，不得静默用默认空数据覆盖：

```text
characters/catalog.json          # 稳定角色 ID、名称、结构化资料和组织归属
characters/relationships.json    # 作者确认的角色关系及关系历史阶段
characters/config.json           # 立场、状态、组织类型和关系类型的项目级预设
characters/order.json             # 改为保存稳定 characterId，兼容读取旧名称数组
worldview/organizations.json     # 稳定组织 ID、别名、层级和组织描述
```

`characters/catalog.json` 的最小记录：

```ts
interface CharacterRecord {
  id: string
  name: string
  fileName: string
  aliases: string[]
  identity: string
  stanceId: string
  statusId: string
  gender: '男' | '女' | '未知'
  tags: string[]
  affiliations: CharacterAffiliation[]
  createdAt: string
  updatedAt: string
}

interface CharacterAffiliation {
  organizationId: string
  role: string
  status: 'active' | 'former' | 'hidden'
  startChapter?: ChapterRef
  endChapter?: ChapterRef
  notes: string
}
```

`characters/relationships.json` 只保存作者确认的数据；章节分析和共同出场仍然是每次构图时生成的证据，不复制进手工关系文件：

```ts
interface CharacterRelationship {
  id: string
  characterAId: string
  characterBId: string
  direction: 'undirected' | 'a-to-b' | 'b-to-a'
  periods: RelationshipPeriod[]
  notes: string
  createdAt: string
  updatedAt: string
}

interface RelationshipPeriod {
  id: string
  typeId: string
  status: 'active' | 'ended' | 'uncertain'
  startChapter?: ChapterRef
  endChapter?: ChapterRef
  description: string
}
```

`worldview/organizations.json` 的组织记录使用 `parentId` 表达层级；角色只保存 `organizationId`，不再把组织名称作为关系主键：

```ts
interface OrganizationRecord {
  id: string
  name: string
  aliases: string[]
  kindId: string
  parentId?: string
  description: string
  status: 'active' | 'dissolved'
  createdAt: string
  updatedAt: string
}
```

`characters/config.json` 使用稳定选项 ID，显示名称可以由用户修改：

```ts
interface CharacterModuleConfig {
  schemaVersion: 1
  stances: OptionDefinition[]
  statuses: OptionDefinition[]
  organizationKinds: OptionDefinition[]
  relationshipTypes: RelationshipTypeDefinition[]
  updatedAt: string
}

interface OptionDefinition {
  id: string
  label: string
  order: number
}

interface RelationshipTypeDefinition extends OptionDefinition {
  tier: 1 | 2 | 3
  weight: number
  color: string
  defaultDirection: 'undirected' | 'a-to-b'
}
```

首次创建项目使用默认预设；用户可以增删改名称、调整顺序并一键恢复默认。删除仍被使用的选项必须要求替换为其他选项，不得留下无法显示的 ID。现有 `RELATION_META` 只作为默认预设和兼容映射，不再是唯一的硬编码类型来源。

#### 4.2 稳定 ID、目录初始化和名称解析

1. 新增 `characterCatalog.ts`，负责加载、校验、创建和更新目录；新增角色时生成不可变 `characterId`，角色显示名和 Markdown 文件名可以变化。
2. 首次发现没有目录时扫描现有 `characters/*.md`，为每个文件生成目录记录、别名和默认结构化字段；只写入 `catalog.json`，不批量改写角色正文。现有 `order.json` 的名称数组在内存中解析为 ID，成功保存后升级为 `{ schemaVersion: 2, characterIds: string[] }`。
3. `characterNodeId` 和 `organizationNodeId` 改为接收稳定 ID；节点的 `label` 仍显示当前名称。`parseGraphNodeId`、`ProjectView`、关系图和所有跳转回调统一传递 ID，不再把节点 ID 的尾部当作角色名。
4. 新增名称解析器：按规范化名称和 `aliases` 将章节快照、伏笔关联和旧关系文本解析为稳定 ID。解析不到的名称保留为“未解析引用”诊断，不自动创建幽灵角色节点。
5. 角色重命名保留旧名称到 `aliases`，结构化关系、组织归属、伏笔关联和章节解析因此不需要批量改写；正文中的自由文本引用只提示影响范围，不静默替换。

#### 4.3 Markdown 投影与结构化字段编辑

1. 新增纯函数 `characterMarkdown.ts`，只按完整字段名解析和更新首个标准字段，不再使用当前 `key.includes(...)` 的宽松匹配。标准字段固定为：`角色`、`性别`、`身份/职业`、`立场`、`角色状态`、`标签`、`所属组织`。
2. 结构化资料编辑器修改 `CharacterRecord` 后，同时更新编辑器草稿中的 Markdown 投影；保存时使用一个 `saveCharacterBundle` 流程写入 Markdown 和目录记录，任一写入失败都保留草稿并显示错误。
3. Markdown 正文仍是用户可见的自由内容。用户直接编辑标准字段后，保存流程重新解析这些字段并同步目录：立场、标签、性别和组织名称以 Markdown 的标准字段为准；未改变的组织保留职位、历史状态等详细归属信息；新增组织名称必须选择已有组织或确认创建最小组织记录，移除名称只结束该角色的当前归属，不删除组织实体和历史记录。
4. 遇到重复标准字段、非法立场/状态或不存在的组织时，不自动删除正文；在结构化区域显示冲突提示，用户确认后才规范化。
5. 结构化资料 UI 提供“立场”“角色状态”“性别”“标签”和“所属组织”控件；组织归属使用可搜索多选和“添加组织”入口，职位、当前/曾经所属、起止章节和备注在归属详情中编辑。

#### 4.4 组织目录、阵营层级和归属历史

1. 新增 `organizationStore.ts` 和 `OrganizationManager`，负责组织目录 CRUD、别名、类型、父组织、描述和解散状态；保存前校验父级不能形成环，不能把组织设置为自己的祖先。
2. 世界观“势力/组织”区域增加“组织目录 / 叙述正文”切换。组织目录是结构化成员和层级的来源，`forces.md` 继续保存世界观叙述；从角色面板创建的组织可以回到该目录编辑。
3. 首次初始化只从角色卡明确的 `所属组织/阵营/势力/组织` 字段生成待确认候选；不再把世界观文件名或一级标题直接当作组织，避免当前 `extractWorldviewOrganizations` 产生伪组织。`forces.md` 中的候选组织通过一次性导入预览确认。
4. 组织详情显示成员、子组织、角色职位和当前/历史归属；角色归属变更写入 `CharacterAffiliation` 的起止章节和状态，不删除历史记录。
5. 组织重命名保留别名并更新 Markdown 投影；删除组织前列出成员和子组织，必须先迁移或解除引用；解散组织保留历史图谱节点。

#### 4.5 关系历史、证据合并和关系图编辑

1. 新增 `characterRelations.ts`，实现关系记录 CRUD、角色引用校验、同一对角色的重复记录合并和历史阶段校验；同一记录最多有一个未结束的当前阶段。
2. 关系来源固定为三层优先级：作者手动关系 > 章节快照中的明确关系 > 共同出场推测。手动关系的当前阶段决定图上主标签，快照和共同出场只追加 `evidence`、章节范围和出现次数，不得改变手动类型。
3. 扩展 `RelationshipLink`，增加 `kind`（`relationship` / `affiliation` / `appearance` / `participation` / `foreshadowing`）、`source`、`recordId`、`direction` 和证据摘要；结构性边不得继续伪装成普通 `ambiguous` 关系。
4. `relationshipStore.ts` 的构图顺序固定为：加载目录和组织 → 加载作者关系 → 加载角色卡归属 → 加载章节快照证据 → 添加共同出场证据 → 加载伏笔和章节关联 → 统一计算权重与洞察。边的合并键必须包含 `kind` 和稳定端点，不能再只按名称排序后覆盖不同来源。
5. 保留现有权重算法的信号思想，但把作者确认、明确章节关系和共同出场的权重输入拆开；重新生成快照只更新分析证据，不修改 `characters/relationships.json`。
6. 角色详情增加关系列表和关系历史编辑器：目标角色、关系类型、方向、当前/结束状态、起止 `ChapterRef`、说明和删除操作。关系图 `NodeDetail` 增加“添加关系”“编辑手动关系”“确认推测关系”“查看证据”入口；复杂表单统一用 `<Modal>`。
7. 图谱保持当前可视化布局，但对有方向的关系在详情和文档视图显示方向；无方向关系显示双向标签。未解析引用、已结束关系和推测边使用明确来源徽标，不与当前关系混为一谈。

#### 4.6 搜索、筛选和跨模块详情

1. `CharacterSidebar` 增加搜索框、立场筛选、性别筛选、标签多选和组织筛选；筛选条件组合执行，列表为空显示清除入口。首阶段不分页，保持侧栏独立滚动。
2. 新增 `characterConnections.ts`，提供按 `characterId` 聚合关系、组织归属、关联伏笔和章节出现记录的查询；所有模块使用该服务，不各自解析 Markdown 或 JSON。
3. 角色详情分为“角色卡 / 结构化资料 / 关系 / 跨模块关联”视图；关联视图显示关系图节点、组织、伏笔和章节，并提供明确跳转与空态引导。
4. 伏笔关联补充稳定 `characterId`，保留旧名称作为显示和解析兼容；章节快照继续保存 AI 使用的名称文本，由目录别名解析为 ID。章节跳转必须使用现有 `ChapterRef`（卷和章节），不新增只含全局序号的地址协议。
5. `ProjectView` 统一角色、组织、伏笔和章节的导航参数；跨 Tab 跳转前复用现有未保存协议，目标加载失败时保留当前页面和错误信息。

#### 4.7 重命名、删除和目录事务

1. 新增 Tauri `rename_character` 命令和前端 API：后端一次性校验新名称、改写角色文件名及 `角色` 标准字段、更新 `catalog.json` 和别名；失败时回滚临时文件和目录，不允许前端用“写新文件再删旧文件”模拟重命名。
2. 重命名开始前检查角色编辑脏状态、目标名称冲突、组织/关系/伏笔引用数量，并在 `<Modal>` 中展示“结构化引用会更新，正文自由文本不会批量替换”。
3. 新增 Tauri `delete_character` 命令：事务内删除 Markdown、目录记录、顺序记录、手动关系和 ID 关联；章节快照不改写，旧名称只作为不可解析历史文本保留。删除前显示关系、组织和伏笔影响，并要求明确确认。
4. 关系或组织删除只允许删除作者维护的结构化记录；章节推导证据由下一次构图自然消失，不直接修改快照文件。
5. 目录文件增加 `updatedAt` 和操作版本；服务在写入前重新加载当前版本，发现并发变化时拒绝覆盖并提示刷新，避免两个面板互相覆盖。

#### 4.8 UI 收口、实现顺序和定向交付物

按以下顺序实施，前一步的服务契约未通过定向测试时不得进入下一步：

1. `src/types/character.ts`、`src/types/novel.ts`：目录、组织、归属、关系历史、证据和可配置选项类型。
2. `characterConfig.ts`、`characterCatalog.ts`、`characterMarkdown.ts`：默认预设、校验、目录初始化、Markdown 投影和冲突处理。
3. `organizationStore.ts`、`characterRelations.ts`、`characterConnections.ts`：组织/关系 CRUD、合并优先级和跨模块查询。
4. `graph-id.ts`、`relationshipStore.ts`、`graph-readable.ts`：稳定 ID、边类型、证据合并和导航参数。
5. `src-tauri/src/lib.rs`、`src/api/tauri.ts`：`rename_character`、`delete_character`、必要的事务和错误码。
6. `CharacterPanel`、`CharacterSidebar`、`CharacterEditor` 及新增结构化资料/关系/重命名对话框；所有弹窗复用 `<Modal>`，不为角色列表引入分页。
7. `RelationshipGraph`、`NodeDetail`、`WorldviewPanel` 的组织目录和跨模块导航；最后删除无对应 UI 的死样式并补齐空态、loading、错误和禁用状态。
8. 新增服务测试和组件定向测试；修改 `src/**/*.ts(x)` 后运行 `npm run lint:risk`，只运行与本阶段相关的定向测试，不自动运行全项目检查。

#### 4.9 十项审查补充建议（本阶段必须落实）

以下十项是 Phase 4 的实施约束，不是“以后再考虑”的候选项；任一项未定义清楚前，不进入对应功能的 UI 开发。

1. **多文件事务与崩溃恢复**：新增 `save_character_bundle`、`rename_character`、`delete_character` 三个后端命令，统一处理 Markdown、目录、关系、顺序和伏笔引用。每次操作先写入项目级事务 journal，再以临时文件 + 原子替换提交；提交后删除 journal。应用启动或项目打开时检测未完成 journal，按 `prepare/commit` 状态幂等恢复或回滚。命令必须注册到 Tauri `invoke_handler`，不得用“写新文件再删旧文件”模拟重命名。
2. **归属历史改为时间段列表**：将单条 `CharacterAffiliation` 改为一个组织对应多个 `AffiliationPeriod`，每段保存职位、状态、起止 `ChapterRef` 和说明。保存前按章节目录比较起止位置，禁止同一组织的时间段重叠，最多一个未结束的当前段；职位变更只结束旧段并新增新段，不覆盖历史。
3. **伏笔稳定 ID 使用 schema v2**：在 `ForeshadowEntry` 增加 `relatedCharacterIds: string[]`，保留 `relatedCharacters: string[]` 作为显示和旧名称兼容；`ForeshadowStore` 升级到 schema v2。首次读取旧版只做别名解析和迁移预览，不静默改写；用户保存伏笔时先将原文件备份到 `memory/legacy-backups`，再写入 v2，迁移失败保持旧版可读。
4. **旧字段语义映射固定化**：导入角色卡时先解析精确的 `立场`；旧 `身份/职业` 中的“主角/核心/配角/反派/中立”映射到 `stanceId`；`阵营` 中命中立场预设的值仍映射立场，其余值才进入组织候选；`势力/组织/所属组织` 进入组织候选。未知值必须进入确认列表，禁止把“反派”等立场自动创建成组织。
5. **名称和别名全局唯一**：建立规范化名称索引，正式名和别名使用同一个大小写不敏感、Unicode NFC 的唯一性校验。新增、重命名和添加别名前检查全项目冲突；解析命中多个角色时返回“歧义引用”诊断并要求用户选择，不自动选第一个；未解析和歧义引用都要在关联视图中可查看和修复。
6. **可配置关系类型的旧数据迁移**：将 `RelationType` 从固定联合类型调整为配置 ID，默认 ID 保持 `ally`、`enemy`、`rival`、`family`、`mentor`、`friend`、`love`、`ambiguous` 不变。增加旧 ID 到配置记录的迁移映射；关系标签、颜色、层级、权重和方向全部从配置读取；遇到未知 ID 显示“未知关系”并保留原 ID，不能因配置删除而丢失关系。
7. **稳定 ID 的导航契约一次改完**：统一定义 `onNavigateToCharacter(characterId)`、`onNavigateToOrganization(organizationId)`、`onNavigateToChapter(chapterRef)` 和 `onNavigateToForeshadow(id)`。同步修改 `ProjectView`、`CharacterPanel.initialCharacter`、搜索结果打开逻辑、伏笔面板、关系图和世界观面板；导航状态在目标面板消费后清空，组织导航必须支持初始组织定位，不能继续从文件路径截取角色名。
8. **组织层级明确进入图谱**：为 `RelationshipLink.kind` 增加 `organizationHierarchy`，父组织到子组织生成结构边；默认关系模式隐藏该边，组织模式显示该边和成员归属边。组织目录树仍是编辑入口，关系图只读展示层级，避免 `parentId` 只存储但永远不可见。
9. **内容哈希和乐观并发控制**：`CharacterRecord` 保存 Markdown `contentHash` 和结构化投影哈希；`save_character_bundle` 接收期望版本/哈希，后端提交前重新读取并比较，不一致就拒绝覆盖并返回冲突。`updatedAt` 只用于显示，不能替代内容哈希；目录、关系和组织文件也必须检查 schema 版本和操作版本。
10. **关系历史的时间与显示规则固定化**：所有起止位置使用 `ChapterRef`，通过现有章节目录比较函数校验顺序和跨卷位置；关系阶段不得重叠，同一记录最多一个当前阶段。关系图默认显示当前关系和未结束证据，已结束关系、历史阶段和推测证据通过“历史/证据”筛选查看；当前阶段不存在时不得把最后一段已结束关系伪装成当前关系。

**阶段验收**

- 现有角色卡首次打开可生成稳定目录，不改写正文；新建、重命名、删除和排序都以稳定 ID 工作。
- 新手可以通过预设和结构化控件维护立场、性别、标签、组织、职位和角色状态；熟练用户直接编辑 Markdown 后，标准字段能被解析并同步。
- 组织支持别名、层级、成员、职位和历史归属，父级环、错误引用和删除冲突都有可见错误和恢复路径。
- 作者可以维护有方向或无方向的多阶段关系；关系图同时显示作者确认、章节分析和共同出场证据，快照重建不会覆盖作者关系。
- 角色可以按搜索、性别、立场、标签和组织快速定位，并能跳转到关系图、组织、伏笔和明确的卷/章节位置。
- 重命名和删除在后端事务内完成，结构化引用不会悬空，正文自由文本不被静默批量替换；失败可恢复，不留下半成品文件。
- 角色专用样式全部对应现有 UI，布局遵守 `.app-main` → `.project-view` → `.project-tab-content` → panel 的 flex 高度链路，长内容只在规定的内层容器滚动。

### Phase 5：百万级随机命名与性别管理（本轮实施）

1. 建立独立的组合式中文姓名生成服务：姓氏、男名首段/尾段、女名首段/尾段均使用可维护的去重词库。
2. 保证男、女姓名各自的理论组合量均不少于 100 万；随机起名需避开当前项目中已有角色名。
3. 新建角色卡写入标准 `性别：` 字段；编辑器提供男、女、未知的选择控件。
4. 侧栏添加性别筛选及数量徽标；统计面板添加性别分布。
5. 不实施重新规划后的 Phase 4 结构化资料、关系/阵营编辑、跨模块联动或重命名。

**阶段验收**

- 男、女姓名池各自的理论组合量达到百万级，且词库无重复词条。
- 一键随机起名会同时得到姓名和性别；新角色卡可直接保存为新格式。
- 100 个以上角色时，可按性别快速筛选并在统计页核对分布。
- 不产生重新规划后的 Phase 4 结构化资料、关系/阵营编辑或跨模块联动改动。

- 删除非当前角色与删除当前角色的提示和结果。
- AI 创建、AI 补全、草稿接受/取消、AI 失败反馈。
- loading 期间的按钮禁用与错误展示。
- 结构化资料的新增、编辑、删除和取消；自由 Markdown 正文不被破坏。
- 组织选择、新建、层级编辑、归属历史、冲突确认和跳转世界观组织目录。
- 关系的添加、方向、历史阶段、编辑、删除、确认推测、证据查看、空态和来源标识。
- 搜索与性别/立场/标签/组织组合筛选、清除筛选，以及关系图跳转角色详情后的状态一致性。
- 重命名和删除弹窗正确展示影响范围；取消、失败和回滚后当前角色与未保存草稿保持不变。
- 角色/组织稳定 ID 导航被目标面板消费后清空，搜索、伏笔、关系图和世界观入口不会重复跳转。
- 内容哈希冲突、歧义引用和未解析引用均显示可恢复的错误，不会覆盖磁盘内容。
- 关系图组织模式显示组织层级和成员归属；默认关系模式不混入已结束关系和组织层级边。

### 5.3 手工验收

- 在 Windows 上验证英文大小写重名不会覆盖角色卡。
- 快速连续选择至少三个角色，确认内容不会串位。
- 从搜索和关系图跳转角色后，侧栏高亮、内容加载与未保存保护均正常。
- 小屏和长角色列表下检查侧栏、编辑器分别滚动，页面不整体溢出。
- 连续生成并创建至少 100 名角色，确认性别字段、重复避让、筛选数量与统计分布一致。
- 为角色添加立场、多个标签和多个组织后，重新打开角色卡确认结构化字段与正文一致。
- 为同一对角色分别产生手动关系、章节明确关系和共同出场推测，确认展示优先级和来源标识正确。
- 建立至少三级组织层级并尝试制造父级环，确认非法保存被阻止；解散组织后历史成员和图谱节点仍可查看。
- 将同一对角色从朋友改为敌对并结束关系，确认关系历史、方向、起止卷/章节和当前图标签一致。
- 重命名有关系、组织、伏笔和章节历史的角色，确认结构化引用仍能解析，正文自由文本保持原样并有明确提示。
- 模拟目录、关系文件和后端重命名写入失败，确认原文件可继续打开且不存在半完成文件。

## 六、实施约束

- 不把角色卡 Markdown 改造成强制表单；Markdown 继续承载自由正文和标准字段投影，稳定 ID、详细组织归属和关系历史以结构化文件为准。
- 所有新增弹窗必须使用 `src/components/Modal.tsx`；列表若未来确需分页，必须使用 `src/components/Pagination.tsx`。
- 保持 `.panel-layout` → `.panel-sidebar` / `.panel-editor` 的 flex 高度链路，不使用 `height: calc(100vh - Npx)`。
- 修改 `src/**/*.ts` 或 `src/**/*.tsx` 后，交付前运行 `npm run lint:risk`；整体验证仅在用户明确要求时执行。
- Phase 4 的关系记录与角色 Markdown 字段采用混合存储：角色自身资料写回角色卡，跨角色关系独立保存，禁止在两张角色卡中重复维护同一条关系。
- 关系来源必须可区分，作者手动确认内容优先于章节分析和共同出场推测；快照重建不得静默覆盖手动关系。
- Phase 4 重命名只更新文件名、标准字段、稳定目录和结构化引用；正文自由文本明确不做批量替换。
- Phase 4 必须完成角色和组织稳定 ID 改造；初始化只新增目录索引，不批量改写既有角色卡、章节正文或快照文件。
- 关系与归属的起止位置统一使用现有 `ChapterRef`，不得引入只含全局章节序号的第二套地址协议。
- `catalog.json`、`relationships.json`、`config.json` 和 `organizations.json` 解析失败时禁止保存空默认值覆盖原文件。
